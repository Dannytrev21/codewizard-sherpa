### Deviation from story draft (important)

The story's draft AC text used `slice_key` and `probe_name` as
interchangeable terms for three probes — but they diverge:

- `scip_index` probe emits `schema_slice={"semantic_index": ...}` (slice
  key `semantic_index`, distinct from probe name)
- `tree_sitter_import_graph` probe emits `{"import_graph": ...}`
- `node_reflection` probe emits `{"reflection": ...}`

The envelope routes by **probe name** (per `_seam_shallow_merge` in
`cli.py:319`, which maps `probes[<probe_name>] = output.schema_slice`).
The schema's `$id` slug, filename, and envelope `$ref` key must all use
**probe name** (AC-10b binds these three). The schema's interior then
wraps the inner slice body under `properties.<inner_key>`, matching the
Layer A precedent (`node_build_system.schema.json`'s
`properties.build_system`).

The story's hardening pass added the AC-10b agreement check explicitly
to catch the copy-paste shape where filename, `$id`, and envelope `$ref`
fall out of sync. This implementation follows the production convention
the Layer A sub-schemas already established. No probe code was modified.

### Refactor decisions (design-patterns lens walked)

- **Open/Closed at the script body** (tuple-registry of builders) — held
  off on a `@register_probe_schema_builder` decorator until the third
  consumer (Phase 3 Layer C); 7 entries cross the rule-of-three threshold
  on count, but the script's surface is still small enough that a tuple
  list is read-at-a-glance and a decorator would be premature.
  Documented as a Phase-3 follow-up.
- **Smart-constructor at the serialization boundary** — single
  `write_schema_file(path, schema)` chokepoint for all seven schemas;
  byte-identical reruns (T-06) are enforced by one function.
- **Functional core / imperative shell** — pure inner builders; the
  recursive `additionalProperties` walker, the `$defs` hoist, the
  `_wrap_inner` composer, and `_set_id_and_schema` are all pure
  functions; `main()` is the imperative shell that maps over `_BUILDERS`
  and writes files.
- **`_walk_object_nodes` deliberately test-only** — not promoted to
  production. The validator kernel (`codegenie.schema.validator`)
  stays `jsonschema` + `referencing` and nothing more.
- **`$defs` hoist at the wrapper** — inner builders may emit `$defs`
  (Pydantic models do); the wrapper hoists them to the schema root so
  `#/$defs/<Name>` refs (Pydantic's convention) resolve against the
  file root, not the inner slice node.

### Anti-pattern smells checked

- No `dict[str, Any]` shortcut in the kernel — `_SchemaDict = dict[str, object]`
  is the kernel type; Pydantic-emitted schemas come through `model_json_schema()`
  which Pydantic types as `dict[str, Any]` (a library boundary).
- No silent fallthrough — every minimal-valid fixture is itself validated
  by `test_minimal_slice_fixtures_are_themselves_valid` so a fixture
  drift breaks T-07's rejection control loudly.
- Pointer-format brittleness avoided — the rejection test uses
  `_format_pointer(error.absolute_path)` rather than `error.json_path`
  (which differs across `jsonschema` minor versions).

### Carry forward for future stories

- The S4-07 draft's slice-key/probe-name conflation should be flagged
  to the `phase-story-writer` skill — when decomposing future
  sub-schema stories, the writer should resolve names against
  `schema_slice` keys in production code, not against the probe module
  name.
- `lint-imports` PATH dependency in test invocation (`shutil.which`) is
  brittle for local dev; CI runs `lint-imports` directly so the test is
  green there. A follow-up could fall back to `sys.executable + -m
  importlinter` when the script is not on PATH.
- The story prescribed `index_health.schema.json` as the only Pydantic-
  generated sub-schema; `dep_graph` and `scip_index` are ALSO generated
  from their Pydantic models in this run. AC-5's byte-identical
  re-run test pins `index_health` specifically (the deepest case); a
  future story could parametrize it across all generated sub-schemas.
