### Refactor decisions (design-patterns lens)

- **Extracted `_spawn_with_invariants`** from `run_allowlisted`'s body. Justification: rule-of-two — both `run_allowlisted` and the new bwrap path needed identical SIGTERM-escalation + `_RUNNING_PROCS` registration. Duplication was the alternative; the seam was honest. The allowlist check intentionally stayed at the caller (one trust boundary per public function — `run_allowlisted` checks `argv[0]`; `run_external_cli` checks the *inner* `argv[0]` before bwrap wrapping).
- **Did NOT introduce a sandbox-wrapper registry.** Story §Out-of-scope and Notes-for-implementer explicitly say wait for Phase 5 microVM (rule-of-three threshold). The wrapper-pattern `_maybe_wrap_with_bwrap` is a Command-pattern shape — pure argv build + side-effectful tmpdir creation in one place. The pure/impure split is a documented future refactor when the registry threshold is hit.
- **Did NOT add `bwrap` to `ALLOWED_BINARIES`.** Story §AC-3a forbids it; the merged 02-ADR-0001 §Consequences last bullet enshrines the wrapper-pattern exception; S1-06 AC-15 ships the closed-set regression. The bwrap spawn lives inside `exec.py` (same trust tier, same file) — this is the documented architectural pattern, not a workaround.
- **Newtype validator at the boundary, not at the constructor.** `_validate_probe_name` is called from both `run_external_cli` and `_maybe_wrap_with_bwrap` (defense in depth on the macOS path too where `_maybe_wrap_with_bwrap` is the first thing called). A `ProbeId` constructor-level validator is explicit out-of-scope (separate Phase 2 follow-up under S1-05's family).
- **`_platform_is_linux()` indirection** — added solely to defeat mypy's `Literal`-narrowing of `sys.platform`. Single-purpose, single-line, and arguably future-proof (one seam for all platform checks). Documented why in the docstring.

### Lessons to fold into `_lessons.md`

- **L19** — mypy `--strict` narrows `sys.platform` via `Literal` types, marking the post-`sys.platform.startswith("linux")` branch as unreachable on darwin. Indirect through a small `def _platform_is_linux() -> bool: return sys.platform.startswith("linux")` helper; mypy cannot narrow through function boundaries. Phase 5 (microVM, namespaces) and future Layer-C probes that check the platform will hit the same trap.
- **L20** — `codegenie.exec` cannot top-level import from `codegenie.types.identifiers` because the latter re-exports `PackageManager` from `codegenie.probes.node_build_system`, which itself imports `codegenie.exec`. Resolve by putting the `ProbeId` (or any other identifier name) import under `TYPE_CHECKING` when only used in annotations — `from __future__ import annotations` makes the deferred import safe. Any future `exec.py` extension that wants to type-check on a kernel-tier newtype needs the same pattern.
