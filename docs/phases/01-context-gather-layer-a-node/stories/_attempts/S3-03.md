### Deviations from the story TDD plan

#### Deviation #1 — `structlog.testing.capture_logs` instead of in-test processor reconfiguration

**Story prescribed (TDD §Red, `test_parser_kind_yarn_lockfile_surfaces_on_size_cap_event`):**
```python
events: list[dict[str, Any]] = []
def capture(logger, method_name, event_dict):
    events.append(dict(event_dict)); return event_dict
structlog.configure(processors=[capture])
```

**Why it failed:** the default `structlog` `PrintLogger` rejects
`event_dict` keyword arguments; calling `_logger.info("...", cap_kind=...,
parser_kind=...)` after the test's `structlog.configure(processors=[capture])`
raised `TypeError: PrintLogger.msg() got an unexpected keyword argument
'cap_kind'`. The example processor returns the event dict but does not
short-circuit the rest of the chain.

**Resolution:** Adopted the codebase precedent (`tests/smoke/conftest.py`,
`tests/unit/test_audit_anchors.py`, `tests/unit/parsers/test_safe_yaml.py`)
— `structlog.testing.capture_logs()` as a context manager. Idiomatic,
no global state, no per-test logger reconfiguration. Same contract
asserted (cap event surfaces with `parser_kind="yarn_lockfile"`).

#### Deviation #2 — `tools/fuzz_yarn_lock.py` `print(...) # noqa: T201`

**Story prescribed (Implementation outline §7):** "write
`tools/fuzz_yarn_lock.py`; commit it. Run it; capture the output line
and paste into PR body."

**Why it tripped:** ruff's `T201` ("no `print()`") is enabled
project-wide for "all logging goes through structlog" (pyproject.toml
comment is explicit: "T201 (print) is load-bearing"). The fuzz harness
is a CLI tool, not application code, and is commanded by the story to
emit a line for the PR body.

**Resolution:** per-`print` `# noqa: T201` with a one-line comment
("CLI fuzz harness; structlog is overkill for tools/"). Rule 3
(surgical changes) — alternative would be widening `pyproject.toml`'s
`per-file-ignores` to `tools/**`, which the story does not authorize and
which would weaken T201 for any future tool. Per-line `noqa` keeps the
rule load-bearing for `src/` while admitting one CLI exception.

### Refactor decisions

- **DP-1 (rule of three — defer extraction).** Validation-note
  block-correction #4 + Notes §7 prescribed deferring `_translate(path,
  *, cause)` extraction. Followed verbatim. The trio is two-of-a-kind
  (pnpm/npm wrap `safe_X.load` and catch `MalformedYAMLError` /
  `MalformedJSONError` narrowly) plus one-of-a-kind (yarn wraps
  `open_capped`, dispatches between `pyarn` and hand-rolled, catches
  `Exception` broadly). Extracting `_translate` would either widen the
  helper to swallow `Exception` (loosening pnpm/npm's narrow catches) or
  require two helpers (defeating the abstraction). Phase 2 can revisit
  if a fourth lockfile (`bun.lockb`) reveals the right shape. **No
  helper extracted.**
- **DP-2 (registry pattern, already realized).** `parsers/_io.open_capped`
  is the kernel; `_yarn.py` is its third caller with
  `parser_kind="yarn_lockfile"`. Validated by
  `test_parser_kind_yarn_lockfile_surfaces_on_size_cap_event`.
- **DP-3 (strategy pattern, light form).** `_HAS_PYARN: bool` + `if/else`
  inside `parse()` is the simplest possible expression of a two-strategy
  dispatch. **Not** refactored to a `_PARSER_FN: Callable[[bytes],
  YarnLock]` module-level binding; CLAUDE.md Rule 2 — the indirection
  wouldn't pay for itself, and `_pyarn_parse` / `_parse_handrolled`
  share the `body: bytes -> YarnLock` signature without needing a
  Callable abstraction.
- **DP-4 (functional core / imperative shell).** `_parse_handrolled(body:
  bytes) -> YarnLock` and `_pyarn_parse(body: bytes) -> YarnLock` are
  pure (no I/O). Only `parse(path)` touches the filesystem (via
  `open_capped`). This split is what makes the local fuzz harness
  testable: it targets `_parse_handrolled` directly with byte-mutated
  inputs, no filesystem involvement.
- **DP-5 (Open/Closed at the family level).** `_yarn.py` does not
  reference any sibling parser (architectural test); `_lockfiles/__init__.py`
  remains inert (architectural test). Adding `_bun.py` later requires
  zero edits to either file.
- **Marker discipline (preserved).** `MalformedLockfileError` remains a
  positional-message-only marker. Path is recoverable from `args[0]`;
  cause is recoverable from `__cause__`. No `__init__` override; no
  instance state. Exact pattern S1-02 / S1-03 / S3-01 / S3-02 settled.

No anti-patterns flagged: no primitive-obsession-on-ID violations (no
IDs here), no anaemic types where sum types would do (typed exceptions
are tagged-union-like; no further refinement needed for a thin
adapter), no hidden state, no pure/impure tangle.

### Cross-story lesson — one new

**L-31 (S3-03 → future structlog event-capture tests):** for tests
that need to capture structured-log events emitted by code under test,
**use `structlog.testing.capture_logs()`** as a context manager — it
swaps in a capture chain for the duration of the `with` block and
restores cleanly. Do **not** call `structlog.configure(processors=[...])`
inside a test; the codebase's default `PrintLogger` rejects `event_dict`
kwargs, and ad-hoc reconfiguration trips on the resulting `TypeError`
the moment the structured event surfaces. The existing precedents are
`tests/smoke/conftest.py`, `tests/unit/test_audit_anchors.py`, and
`tests/unit/parsers/test_safe_yaml.py`.

**Carry-forward note (S2-02a → S3-03 update):** The S2-02a lessons
note speculated that S3-03 would dispatch on yarn variant (yarn-classic
→ pyarn, yarn-berry → safe_yaml.load) via a `Mapping[str, Parser]`.
The actual S3-03 design supersedes this — the dispatch is on
`_HAS_PYARN` (binary install vs. not), not on the resolved variant.
The hand-rolled scanner targets yarn classic v1 (the dominant legacy
fixture-portfolio format); berry-emitted `yarn.lock` files are out of
Phase 1 scope per the S3-03 §Out-of-scope row 4. The variant-dispatch
shape may still apply to a future S3-05 reconciliation step where
`NodeManifestProbe` decides whether to treat the parsed entries as
classic-style or berry-style — but that is a different decision from
the parser-strategy dispatch.
