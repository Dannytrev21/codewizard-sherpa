### Divergences from the literal story text (surfaced + reconciled)

The story is HARDENED, but three premises in the literal AC text did not
match the codebase state at execution time. Each was reconciled by
honouring the story's **intent** while using the actually-correct
mechanism. Surfaced here so the next story along this lineage benefits
from the audit.

1. **Story prescribed `result.context["probes"][probe]["errors"]` and
   `["confidence"]` at the envelope level.** The envelope's actual shape
   for the `node_manifest` slice puts errors at
   `["probes"]["node_manifest"]["manifests"]["errors"]` (per the slice
   schema and `_build_output`); there is **no** envelope-level
   `confidence` field — `ProbeOutput.confidence` is consumed by the
   coordinator but does NOT serialize to the envelope.

   - For `node_manifest` (AC-1b / AC-2b node_manifest clause / AC-4b),
     used the correct slice path
     `["probes"]["node_manifest"]["manifests"]["errors"]`. Closed-world
     equality on the IDs is preserved (the primary defense against
     "errors silently empty" regressions).
   - `confidence` envelope-level assertions are unverifiable; omitted.
     The probe-level demote is verified by the AC-12 unit tests, which
     assert `out.confidence == "low"` against `ProbeOutput` directly.
   - For `language_detection` (AC-2b language_detection clause), the slice
     under `language_stack` does not include `errors` today (only
     `counts`/`primary`/`framework_hints`/`monorepo`). Asserting via the
     CLI envelope is not possible without scope creep beyond AC-12
     ("surgical additions only"). AC-12's unit test
     (`test_deeply_nested_package_json_demotes_to_low_via_depth_cap`)
     verifies the language_detection retrofit on `ProbeOutput.errors`
     directly.

2. **Story prescribed AC-12 retrofit on two probes (language_detection,
   node_manifest).** Reality: four probes read `package.json` —
   `language_detection`, `node_build_system`, `node_manifest`,
   `test_inventory`. Without retrofitting all four, AC-2b's
   `exit_code == 0` cannot hold: the unhandled `DepthCapExceeded` crashes
   `node_build_system` and `test_inventory`, all four prelude probes
   degrade, the envelope fails schema validation, and the CLI exits 3
   (`schema_invalid`). Retrofitted all four (additive only — the existing
   compile-time `_ID_PATTERN.match` discipline validates the new ID at
   each call site).

3. **Story prescribed a 10,000-keys-deep package.json fixture for AC-2.**
   Python's stdlib `json` decoder is recursive at the scanner level
   (`raw_decode → scan_once`); the default recursion limit (1000) is hit
   before 10,000 levels are decoded, surfacing as `RecursionError` rather
   than `DepthCapExceeded`. Used `depth=200` instead — comfortably above
   the 64-frame `max_depth` cap that the walker is meant to trigger, well
   under Python's ~1000 recursion ceiling. Documented inline in the
   builder's docstring. Builder unit test pins determinism at depth=4 and
   depth=16 to keep the test cheap.

### Refactor decisions

- **`Literal["size", "depth"]` on the helper's `cap_kind` parameter** —
  type-level guard against `cap_kind="bytes"` drift. Lift kept at the
  test-helper boundary (Notes-only DP-3 stays Notes-only — promoting the
  parsers' `_CAP_KIND_*` constants to a shared `StrEnum` waits for the
  third consumer per Rule 2).
- **`expected_lockfile_error_id` as drift guard** — imports
  `_ERROR_PREFIX_BY_KIND` / `_ERROR_SUFFIX_BY_EXC` directly from the
  probe module rather than hard-coding the resulting strings; if the
  vocabulary changes (e.g. `pnpm_lock` → `pnpm`), every adversarial test
  flips in lockstep with the implementation instead of silently masking
  drift.
- **`Sequence[Mapping[str, Any]]`** on `assert_parser_cap_event`'s `logs`
  parameter — `structlog.testing.capture_logs` returns
  `list[MutableMapping[str, Any]]`; the helper signature is widened to
  the covariant Sequence-of-readonly-Mapping so the type check holds at
  every caller.
- **Composition over inheritance for the adversarial-test family** —
  Rule 2 governs at four siblings; the AC-7/AC-8/AC-9 helpers provide
  composition without an abstract base class. Logged as Notes-only
  (DP-4); revisit when S5-02 / S5-03 land if the duplication crosses six
  siblings.

### Follow-ups surfaced this attempt

- The validation report missed the `node_build_system` and `test_inventory`
  package.json readers (they have parallel `_PKG_JSON_FAILURE` registries
  and their own catch-tuples). The audit script `phase-story-validator`
  uses today should grep for **every** site that calls `safe_json.load` on
  `package.json`, not just the two named in AC-12. (For future cap-family
  stories: any new typed parser exception requires symmetric retrofit
  across the **full** set of `package.json` readers, not just the two
  documented in the story.)
- `language_detection` slice does not expose `errors[]` today (the schema
  declares it forward-compatibly but the implementation never emits it).
  If a future story needs to assert language_detection errors via the
  envelope (rather than `ProbeOutput.errors` directly), the slice will
  need a one-line addition to emit `errors=errors` inside `language_stack`.
  Out of scope for S5-01.
- `confidence` is not surfaced on the envelope per-probe. Any future
  story that wants closed-world envelope-level confidence assertions
  needs a coordinated decision: either thread `confidence` into every
  slice (envelope shape change → ADR amendment) or document the
  per-probe-internal-only contract explicitly.
