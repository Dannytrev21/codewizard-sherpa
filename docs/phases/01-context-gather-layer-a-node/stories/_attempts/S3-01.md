#### Deviation #1 — alias-chain depth-cap fixture replaced with plain deep nesting

**Story prescribed (TDD §Red, `test_parse_yaml_alias_amplification_reraises_depth_cap`):**

```python
lines = ["a0: &a0 {x: 1}"]
for i in range(1, 70):
    lines.append(f"a{i}: &a{i} {{x: *a{i - 1}, y: *a{i - 1}}}")
# expected: pytest.raises(DepthCapExceeded)
```

**Empirically:** `parsers.safe_yaml.load(p, max_bytes=1MB, max_depth=64)`
returns the parsed mapping successfully (70 top-level keys). No
`DepthCapExceeded`. The `id()`-memoized walker in
`parsers/_depth.py::_walk` visits each anchor dict at depth 1 with
shared subtrees already in `seen` — max walked depth = 2.

**Resolution:** Replaced the alias fixture with the proven
flow-style deep-nesting pattern from
`tests/unit/parsers/test_safe_yaml.py::_nest_dict(70)` —
`{k: {k: ... v}}` × 70. AC-4 (re-raise unchanged) is the contract
under test; any YAML that triggers `DepthCapExceeded` in
`safe_yaml.load` satisfies it. The replacement test is now
`test_parse_passes_through_depth_cap_from_real_safe_yaml`, and its
docstring cites the deviation. Cross-story lesson **L-29** captures
the root cause so S3-02 and S3-03 don't repeat the same pattern.

The validator's hardening report (`_validation/S3-01-pnpm-lockfile-parser.md`)
asserted "the `id()`-memoized walker catches alias amplification as
a logical-depth violation" — that was an architectural read of arch
§"Edge cases" row 1 that the actual walker implementation
contradicts. The walker correctly defeats alias amplification by
*skipping* shared subtrees, not by surfacing them as depth
violations. Arch §"Edge cases" row 1's "billion-laughs ->
DepthCapExceeded" must refer to a different shape (or the arch text
predates the id()-memoization choice in S1-03's `_depth.py`); for
S3-01's purposes, plain deep nesting suffices.

#### Deviation #2 — `os.fstat` monkey-patch replaced with `safe_yaml.load` monkey-patch

**Story prescribed (TDD §Red, `test_parse_oversized_file_reraises_size_cap`):**
a custom `FakeStat` class injected via `monkeypatch.setattr(os, "fstat", ...)`
to spoof a 60 MB file size pre-`safe_yaml.load`.

**Resolution:** Monkey-patched `_pnpm.safe_yaml.load` directly to
raise `SizeCapExceeded("synthetic: size>cap")`, asserting object
identity (`exc.value is raised`) on the re-raise. This proves the
load-bearing AC-4 contract — *unchanged* pass-through — more
directly than the os.fstat indirection (Rule 2: minimum code that
proves the contract). The same parametrized test covers
`SymlinkRefusedError`; the real-symlink integration test (also
required by the story) is kept verbatim. `DepthCapExceeded` keeps a
real-fixture integration test (deviation #1).

Object-identity assertion (`exc.value is raised`) is strictly
stronger than `pytest.raises(type(raised))` — it catches any
re-wrap that constructs a new instance of the same class.

### Refactor decisions

- **Module constants kept per-file** (`PNPM_LOCKFILE_MAX_BYTES`,
  `PNPM_LOCKFILE_MAX_DEPTH`). They will be identical across pnpm /
  npm / yarn (all 50 MB / depth 64). Lifting them to
  `_lockfiles/__init__.py` is **S3-03's** call after the third sibling
  exists (rule of three; Notes-for-implementer #7).
- **No shared `_translate(path, *, cause)` helper extracted.** The
  one-line translation pattern is the same across the three
  sibling parsers, but S3-01 alone doesn't cross the rule-of-three
  threshold. Extracting now creates a kernel that S3-02 inherits
  silently. Decision deferred to S3-03.
- **`cast(PnpmLock, raw)` is a runtime no-op.** Intentional — TypedDict
  is static-only; runtime validation belongs to `NodeManifestProbe`
  (consumer-side concern per Note-for-implementer #1 and #9). The
  functional-core / imperative-shell split: parse is I/O +
  translation, validation lives with the consumer.

### Design patterns reinforced

- **Plugin family with kernel-only edges.** `_pnpm` imports only
  from `codegenie.parsers` and `codegenie.errors`. AC-12's
  source-introspection test fires on the first attempt to add a
  sibling-import edge (CLAUDE.md "Extension by addition").
- **Markers-only exceptions.** `MalformedLockfileError` constructed
  with a single positional message string; path in `args[0]`, cause
  on `__cause__` via `raise ... from cause`. Same pattern S1-02 and
  S1-03 settled.
- **Smart constructor seam at the catch site.** The catch site (the
  probe, S3-05) builds the `WarningId` per ADR-0007 from
  `exc.args[0]`. The marker class is dumb; the construction logic
  lives where it has context. This is the inverse of an
  attribute-bag exception design.
- **Functional core / imperative shell.** `parse` is pure I/O +
  exception translation. No business logic, no field validation, no
  shape enforcement. Validation lives in `NodeManifestProbe` where
  consumer requirements (recipe planner vs distroless build vs
  SBOM) are visible.

### Anti-patterns avoided

- No defensive `try / except Exception` (would absorb typed
  pass-through classes — Rule 12).
- No re-validation of `safe_yaml.load`'s caps (no defense-in-depth at
  the parser layer; the caps live in the parser, the wrapper is one
  exception translation away from the caller).
- No `MalformedLockfileError.__init__(self, *, path, cause)` extension
  (would break Phase 0's markers-only invariant; would require
  edits to `tests/unit/test_errors.py::test_subclasses_are_markers_only`).
- No premature extraction of a `_translate` helper (rule of three
  not yet met — Rule 2).
