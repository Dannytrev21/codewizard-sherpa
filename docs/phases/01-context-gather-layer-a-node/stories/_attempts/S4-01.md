### What had to be reasoned about during execution

1. **Architectural conflict (Rule 7) — `references_secrets` field name vs Phase-0 secret-shaped-key defense.** The CISlice contract (`phase-arch-design.md §"Data model"`) mandates `references_secrets: list[str]`. Phase-0 ADR-0008 + ADR-0010 install a defense-in-depth `SECRET_FIELD_PATTERN = (?i)^.*(secret|token|...).*$` that fires on any dict key containing `secret`. Without an exemption the slice could never ship: every gather on a real GHA repo raised `SecretLikelyFieldNameError` and exited 3. Resolution per Rule 7: surface, don't average. Added a narrow `SECRET_FIELD_ALLOWLIST` (one entry, exact-match only), wired into both walkers via the same identity-shared frozenset (mirrors how `SECRET_FIELD_PATTERN` is itself shared), and ratified in ADR-0014. Surfaced for the PR body so reviewers see the allowlist mechanism is now a permanent surface.
2. **Test-fixture shape vs jobs-strict extraction.** Two TDD-plan tests (`test_secrets_sorted_deduped_literal_only`, `test_secrets_regex_does_not_match_env_inputs_or_capitalized`) write `steps:` without a `jobs:` wrapper — valid YAML but not a valid GHA shape. The original outline said extract secrets from `runs` (which my walker reads from `jobs.*.steps[].run`). Fix: added `_collect_string_values(node)` — depth-first walk yielding every string leaf — and route secrets-regex over that, not over `_extract_run_strings`'s narrower walk. `_extract_run_strings` and `_extract_uses_strings` stay strict so the `run` vs `uses` discriminator on `_IMAGE_BUILD_MARKERS` (AC-25) keeps its precision.
3. **Jenkinsfile `unit_test_command` fallback.** The TDD plan parametrizes `sh 'npm test'` (matches `_UNIT_TEST_MARKERS`) and `sh "npm run build"` (does NOT match any marker) and expects both to land in `unit_test_command`. The story's intent is that under Phase-1's regex-only-low-confidence Jenkinsfile mode, every captured `sh '…'` IS a candidate test command. Resolution: when `provider == "jenkins"` and no marker matched but `runs` is non-empty, `unit_test_command` falls back to `runs[0]`. Documented inline.
4. **`ctx.output_dir` doesn't exist on the runtime context.** The probe ABC's `ProbeContext.output_dir` field is a Phase-0 spec; the per-dispatch object the coordinator constructs is `BudgetingContext` (carries `workspace`, `raw_artifact_mb`, optional `parsed_manifest`/`input_snapshot`). My first cut wrote raw artifacts to `ctx.output_dir / "raw" / "ci.json"` and crashed at runtime (`AttributeError: 'BudgetingContext' object has no attribute 'output_dir'`). Fix: write under `ctx.workspace / ".codegenie" / "_probe_raw" / "ci.json"`. The `.codegenie/` namespace is already excluded from input fingerprinting (S3-06 lesson L-39); the writer's raw-artifacts loop reads it from there and re-emits under `.codegenie/context/raw/`.

### Lessons (carry forward to `_lessons.md`)

- **L-42 — `BudgetingContext` is the runtime context, not `ProbeContext`.** A probe that touches `ctx.output_dir`, `ctx.cache_dir`, or `ctx.logger` will pass unit tests (which use the full ABC dataclass) but `AttributeError` at runtime. The runtime exposes `workspace`, `raw_artifact_mb`, `parsed_manifest`, `input_snapshot`, `report_bytes`. Anything else is unit-test-only. Until the ABC is reconciled, write raw artifacts under `ctx.workspace / ".codegenie" / "_probe_raw" / <name>.json` (the `.codegenie/` namespace is already excluded from fingerprinting per L-39).
- **L-43 — A probe field name that contains "secret" / "token" / "password" / etc. trips Phase-0's defense.** When the data-model contract mandates such a name (here, `references_secrets`), use the `SECRET_FIELD_ALLOWLIST` mechanism with a fresh ADR justifying the construction-time guarantee that makes it safe. Do NOT widen the regex; do NOT rename behind the contract's back.
- **L-44 — Strict `jobs.*.steps[].run` walkers are too narrow for cross-cutting regex coverage.** Image-build / run-vs-uses detection wants strict; secrets / general-text-substring detection wants a depth-walker over all string leaves (`_collect_string_values`). Keep both; route each to the right consumer.
