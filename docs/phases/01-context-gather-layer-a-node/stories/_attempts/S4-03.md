**What worked:**
- Reading the S4-01 + S4-02 hardening reports + their attempt logs first;
  the per-probe shape translated line-for-line. Five module-level
  Open/Closed seams + `_WARNING_IDS` / `_ERROR_IDS` frozensets + import-
  time `_ID_PATTERN` loop is now a copy-paste discipline.
- The lcov scanner stayed under 50 LOC (currently 30 lines of executable
  code per coverage report). Reusing `open_capped` was the right call —
  no `O_NOFOLLOW` re-implementation, no fd-lifecycle bug surface, AC-25
  AST walk passes structurally. Pure dispatch over `_LCOV_PREFIX_MAP`
  with `str.startswith` keeps the regex-DoS attack surface at zero.
- Strict-by-enumeration for `commands` (six closed `properties`, no
  `additionalProperties` opt-out) lets the AC-45 walk-every-object
  invariant hold without a `security_context`-style exception. Adding
  a future canonical script name is one tuple entry + one schema
  property + one fixture row — pure Open/Closed.
- The `_minimal_slice()` factory routes `package.json` parse failures
  to `ProbeOutput.errors` while still emitting a complete slice
  (zero counts, no framework). Mirrors the S2-02 / S4-01 / S4-02
  errors-vs-warnings discipline (ADR-0007 line 50): structured ID on
  `errors`, never on `slice["warnings"]`. Tested across 3 exception
  types (size, malformed, symlink-refused) via a parametrized matrix.
- `structlog.testing.capture_logs` for AC-49 (probe.start + probe.success
  observable). `caplog` doesn't capture structlog; the testing helper
  is the codebase-stable seam.
- Byte-budget fuzz (≥ 5 MB/s on 1 MB pathological input) replaces
  the brittle wall-clock-only check from the original story. CI-stable
  + structurally meaningful: anything slower than 5 MB/s indicates
  ReDoS-grade pathology.

**What didn't (caught + fixed in this attempt):**
- First scanner draft used `max_bytes: int = _LCOV_MAX_BYTES` as the
  default-arg value — Python binds defaults at function-def time so
  monkeypatching `_lcov_scanner._LCOV_MAX_BYTES` to 16 in
  `test_size_cap_raises` had no effect (the function still saw
  `50 * 1024 * 1024`). Fix: switched to `max_bytes: int | None = None`
  + module-level lookup at call time. Documented inline.
- First sub-schema used `additionalProperties: {"type": "string"}` for
  the `commands` block — that means "additional string-typed properties
  allowed", not a strict object. AC-45's recursive walk treats the field
  as an open object and fails. Rewrote `commands` to closed `properties`
  enumerating the six canonical script keys + `additionalProperties: false`.
  Side benefit: schema-level enforcement that non-canonical scripts can
  never sneak into the slice (defense-in-depth on top of probe-level
  filtering in `_extract_canonical_scripts`).
- Probe's first `probe.success` event was missing `confidence="high"`.
  The integration warm-path test (`test_warm_path_memo_hits_once_across_two_probes`)
  asserts every consumer's probe-emitted success carries `confidence`.
  Fix: one keyword on the `_log.info` call. Same shape as `language_detection.py:480`.
- The warm-path integration test had a hardcoded
  `package_json_consumers = ("language_detection", "node_build_system", "node_manifest")`
  tuple; the new probe is the fourth memo consumer. Surgical add of
  `"test_inventory"` to the tuple. The test is robust afterwards: it
  derives `expected_hits = len(consumers) - 1`, so adding a fifth
  consumer in S5+ is one more append.
