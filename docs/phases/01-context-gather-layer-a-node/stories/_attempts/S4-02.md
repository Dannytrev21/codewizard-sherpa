# Attempt log: S4-02 — `DeploymentProbe` + sub-schema with zip-slip mitigation

## Attempt 1 — 2026-05-15 — SUCCESS

**Approach:** Mirror the S4-01 hardened shape verbatim — module-level
Open/Closed seams (`_DEPLOYMENT_TYPE` Literal, `_DEPLOYMENT_DETECTORS`
precedence tuple, `_DEPLOYMENT_PARSERS` dispatch registry, `_WARNING_IDS`
+ `_ERROR_IDS` frozensets with import-time `_ID_PATTERN` loop, `_demote`
+ `_CONFIDENCE_RANK` copied verbatim from `node_build_system.py:276`),
pure helpers (`_is_under`, `_extract_image_ref`, `_select_deployment_type`,
`_env_name_from_filename`, `_filter_k8s_kinds`, `_extract_container_specs`,
`_aggregate_exposed_ports`, `_aggregate_env_var_names`, `_walk_overlays`),
`_ParseResult` NamedTuple as the discriminator-free dispatch contract,
and `ImageRefBlock`/`EnvironmentEntry` TypedDicts. Schema first, then test
file (RED), then probe (GREEN). One refactor pass to fix four issues:
(1) `_env_name_from_filename` accepts both full filename and bare stem;
(2) Helm parser uses wider `values*.yaml` glob then partitions baselines
from env overlays so `values.prod.yaml` (dot variant) is discovered and
flagged via `helm.values_filename_unrecognized`; (3) `_walk_overlays`
counts every non-zip-slip resource entry against `max_files` (not just
existing files) so the file-cap warning fires on a 60-entry kustomization
even when none of the resources exist on disk; (4) test envelope wraps
slice as `{"deployment": <slice>}` to match the existing `probes.<name>`
merge convention.

**ReAct cycles:** ~12 (schema → register → tests → probe → 4 RED→GREEN
fixups → format → full suite + lint + mypy + pre-commit).

**What worked:**
- Reading the S4-01 hardening report + S4-01 attempt log first; the
  pattern translated almost line-for-line to S4-02's 5-way dispatch.
- `validate.environment_entry` adapter as a function attribute on the
  module-level `validate` callable — keeps the per-entry strict
  validation at AC-14 reachable without adding a second top-level
  callable that pollutes `__all__`.
- Wider Helm overlay glob (`values*.yaml`) + post-filter partition into
  baseline + envs — single source of truth, no duplicate file walks.
- Sentinel-exfiltration zip-slip test (AC-24): the smoking-gun
  observable (`31337 not in s["exposed_ports"]`) is structurally
  impossible to fake. Hand-checked the broken impl mentally:
  `str(root) + str(resource)` + `.startswith` would accept
  `/tmp/x/../sentinel.yaml` (because the string DOES start with
  `/tmp/x`), reach the loader, and 31337 would land in `exposed_ports`.

**What didn't (caught + fixed in this attempt):**
- Initial `_env_name_from_filename` only stripped the `values-` prefix;
  the unit test passes the full filename (with `.yaml`), so the
  function returned `("prod.yaml", True)`. Fix: strip `.yaml`/`.yml`
  suffix first, then check the prefix.
- Initial Helm parser globbed `values-*.yaml` only (not `values*.yaml`);
  `values.prod.yaml` (AC-15 dot-variant) was silently skipped. Fix:
  widen to `values*.yaml` then partition.
- Initial `_walk_overlays` counted only files that resolved to an
  existing on-disk file against `max_files`; a 60-entry kustomization
  with no actual resource files on disk left `safe_paths` empty and
  the warning never fired. Fix: count every non-zip-slip resource
  entry (regardless of existence) against the cap.
- Initial schema-rejection test envelope put the slice directly under
  `probes.deployment` (without the `{"deployment": <slice>}` wrapper).
  The merge convention (cli.py:284 — `probes_block[name] = dict(output.schema_slice)`
  where `schema_slice == {"deployment": ...}`) requires the wrap, and
  `_minimal_envelope_with_ci` in `test_ci.py` already establishes the
  precedent. Fix: wrap.
- Schema-walker test originally checked `pointer.endswith("/security_context")`;
  the `oneOf/1` object (the open-shape branch) fails that check and
  trips on `additionalProperties: true`. Fix: change the predicate
  to `"/security_context" in pointer` (anything in the security_context
  subtree is the documented exception).

**Root cause(s) of the four fixups:** all four are story↔implementation-detail
disagreements where the hardened story prescribed a behavioral contract
(the test) and the implementer's first-cut implementation served only
the canonical case. Each was caught by the test on first run and
resolved in one edit. Pattern matches L-7 / L-9 — the contract is the
behavior the test pins, not the spelling of the helpers.

**Lesson for next attempt (S4-03 / S4-04):** the S4-01 + S4-02 hardened
shape is now a load-bearing template — reach for it on every probe in
this phase. The four fixups all stemmed from incomplete scope
reasoning at the helper level (overlay vs. baseline glob, full-vs-stem
filename, count-against-cap policy, envelope wrapping convention).

**Deviations from the story spec (if any):**
- AC-14 references `validate.environment_entry(entry, schema)` — the
  story implies but does not explicitly mandate this adapter. Added
  as a function-attribute on `codegenie.schema.validator.validate`
  (the existing top-level callable). Minimal surface; doesn't pollute
  `__all__`. Documented in `_environment_entry` docstring.
- Helm overlay glob widened from `values-*.yaml` (story Implementation
  outline §5) to `values*.yaml` then post-filter partition into
  baseline / env files. Necessary to discover AC-15's
  `values.prod.yaml` dot-variant (the canonical dash-glob excludes
  it). Behavior contract from AC-15 unchanged; baseline detection
  uses the exact-name `_VALUES_BASELINES` frozenset.
- `_walk_overlays` counts every non-zip-slip resource entry against
  `max_files` (not just files that exist on disk). AC-26's TDD
  fixture creates 60 resource entries but doesn't materialise the
  underlying YAMLs; the cap behavior must fire on entries declared
  in `kustomization.yaml`, not on workload payloads (which are a
  separate layer of validation). Documented in `_walk_overlays`
  docstring.
