**Deviations from the story spec (surfaced for human review):**

1. **`test_well_balanced_5000_nested_…` payload reshape (AC-28).** The
   story-prescribed payload was
   `"{ " + "/* " * 5000 + '"k": 1 ' + " */" * 5000 + " }"`. With nested
   block-comment semantics (AC-13), the `"k": 1` token lives at depth 5000
   *inside* the comment block and gets stripped — the post-strip output is
   `{   }`, which decodes to `{}`. No correct implementation can satisfy
   the test's `assert out["k"] == 1` from that input shape. The fix
   preserves AC-28's intent ("5,000 well-balanced nested block comments
   parse in < 1 s") by moving `"k": 1` outside the comment block:
   `"{ " + "/* " * 5000 + " */" * 5000 + ' "k": 1 }'`. The wall-clock
   budget (< 1 s) and the well-balanced-nesting depth (5,000) are both
   preserved. **The story file's `tests/unit/parsers/test_jsonc.py`
   snippet under "Red — failing test first" should be updated to match;
   this is a one-line edit that does not change AC-28 itself.**
2. **Internal stripper exceptions vs. path-in-message.** The
   implementation outline offered two shapes for the strip stage's
   typed-error contract (AC-26 — path must appear in `args[0]`): either
   pass `path` into `_strip_comments` so it constructs
   `MalformedJSONError` directly, or keep `_strip_comments` pure and
   raise parser-local exceptions that `load` translates. **Picked the
   latter** — `_UnterminatedString` and `_UnterminatedBlockComment` are
   module-private signal exceptions, `load` catches and re-raises as
   `MalformedJSONError(f"{path}: unterminated …")`. This keeps
   `_strip_comments` a pure `bytes -> bytes` function (AC-10) callable
   from unit tests without a `Path`. Both shapes satisfy the ACs;
   documented in the module docstring.
3. **No use of `parsers/_io.open_capped`.** Per the validator's CN6/CV5
   clarification and the implementation outline, `jsonc.load` opens its
   own fd inline. The reason recorded in the module docstring: AC-9
   (`MalformedJSONError(f"{path}: short read")`) requires short-read
   detection beside `os.fstat`/`os.read`, and the shared `open_capped`
   silently returns truncated bytes on short read (S1-03 attempt log
   L-5 documents that `safe_json`'s short-read test was relaxed for
   exactly this reason). Duplication: ~10 lines that mirror
   `_io.open_capped` exactly — accepted (Rule 2: three similar lines
   beats premature abstraction; Rule 3: surgical changes — don't
   refactor `_io.py` to detect short reads when only one parser
   surfaces them as a typed error).
