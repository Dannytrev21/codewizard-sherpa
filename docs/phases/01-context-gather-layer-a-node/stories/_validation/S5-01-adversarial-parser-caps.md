# Validation report — S5-01 Parser-cap adversarial corpus

**Story:** [S5-01-adversarial-parser-caps.md](../S5-01-adversarial-parser-caps.md)
**Validated:** 2026-05-14
**Validator:** phase-story-validator (scheduled task: story-validation-corrector)
**Verdict:** **HARDENED**

## Summary

S5-01 owns the load-bearing parser-cap quadrant of the Phase 1 adversarial corpus — the four tests that prove ADR-0008's "in-process caps replace per-probe sandbox; ~95% threat closure at ~0 ms overhead" is honest. The story was directionally correct (the right four tests, the right `pytest.raises(SpecificError)` shape, the right "generate at setup, never check in" disposition for the 600 MB fixture, the right warning-ID prefixes) but had **six block-tier and four harden-tier weaknesses** that would have let a wrong implementation land silently.

The two most dangerous were:

1. **`warnings` vs. `errors` field confusion (CN-1, CN-2).** AC-1, AC-2, AC-4 asserted the WarningId surfaces on the slice's `warnings: list[str]` field. ADR-0007 §Decision and the existing implementations (`node_manifest.py:413`, `language_detection.py:454-458`) place typed-exception-derived IDs on **`errors: list[str]`**; `warnings` is the soft-degrade signal vocabulary (`lockfile.multi_present`, `kustomization.resource_outside_repo`, …). `phase-arch-design.md §Edge cases` row 1 also says `errors`. A test asserting `"pnpm_lock.depth_cap_exceeded" in warnings` would fail against a correct implementation — and an executor "fixing" the test by appending the ID to `warnings` would land an ADR-0007 violation. **Resolution:** replaced every `warnings` assertion that targets a typed-exception ID with `errors`; left the lone "Step 5 retrofit" `warnings` mentions untouched (none survived).

2. **AC-2 assumes a probe behavior that does not exist (CN-3).** The deep-nesting `package.json` test asserts both the `language_stack` and `node_manifest` slices record `package_json.depth_cap_exceeded`. But neither `LanguageDetectionProbe._PKG_JSON_FAILURE` (lang_det.py:188-194) nor `NodeManifestProbe._read_package_json` (node_manifest.py:348) catches `DepthCapExceeded` — only `SizeCapExceeded`, `SymlinkRefusedError`, `MalformedJSONError`. A `DepthCapExceeded` from `safe_json.load(package.json)` would bubble out, the coordinator would catch it generically, and the resulting slice would NOT contain a structured `package_json.depth_cap_exceeded` ID. The story's "If any test fails because the parser does not enforce the cap, that is a Step-1 regression" framing is wrong here — the Step-1 parser DOES enforce the cap; the Step-2 / Step-3 probes don't catch the typed exception. **Resolution:** new AC-12 names this as an explicit Phase-1 retrofit obligation: extend `_PKG_JSON_FAILURE` in both probes with a `DepthCapExceeded → ("package_json.depth_cap_exceeded", "low")` mapping, and update each probe's `_ERRORS` frozenset. AC-2 now asserts the **post-retrofit** behavior, with TDD step #2 explicitly listing the two-line probe edit.

The remaining block-tier weaknesses were a fictional pytest fixture (`run_gather` doesn't exist; `_invoke_gather` does), a thin structlog payload assertion (only `parser_kind` was pinned; the production code emits four structured fields per `_io.py:91-100` and `_depth.py:112-121`), the `cap_kind` literal vocabulary not pinned (story silently invited "bytes" but S1-03 settled on "size" — see `_validation/S1-03-safe-yaml-parser.md` §AC-14 footnote), and a missing pre-parse defense assertion for AC-4 (oversized lockfile) symmetric to AC-3's `json.loads` patch (`yaml.load` should also be guarded).

The four harden-tier findings:
- **No assertion that errors list contains ONLY the cap-exceeded ID.** A pre-existing `low` from `lockfile.multi_present` would pass the original assertion vacuously; mutation-resistance demands tighter assertions.
- **No disk-space guard for the 600 MB fixture.** Constrained CI runners and contributor laptops should skip the test cleanly via `shutil.disk_usage`, not crash with `ENOSPC`.
- **No drift guard between test-asserted IDs and the registry.** Hardcoded `"pnpm_lock.depth_cap_exceeded"` strings drift if the `_ERROR_PREFIX_BY_KIND` × `_ERROR_SUFFIX_BY_EXC` registry vocabulary in `node_manifest.py` changes; tests should construct the expected ID via the registry.
- **The billion-laughs example uses YAML anchor *literals*, not `*alias` references** — exercising the depth walker but not alias-amplification (which is S1-03 / `test_safe_yaml.py::test_depth_walker_dedupes_alias_targets_no_amplification`). The story's example was unclear about this; clarified in Notes.

The synthesizer rewrote the original **8 ACs** into **18 individually-verifiable ACs** organised into five named groups (Mechanism / Probe-retrofit / Test-marker + ergonomics / CI walltime / Hygiene). The TDD plan was hardened in three places: (1) the gather-level assertions in tests #1, #2, #4 now check the **complete** errors payload (registry-derived IDs, `confidence == "low"`, no spurious other IDs), (2) test #4 now mirrors test #3's pre-parse-canary structure (`yaml.load` patched to raise; pytest.raises(SizeCapExceeded) catches before the canary fires), (3) the structlog-payload assertion is split into a dedicated test that pins all four emitted fields against the cap-exceeded event. The design-pattern lifts (test-helper extraction at the rule-of-three threshold; ID construction from the registry; functional-core fixture builders) were surfaced both as ACs (where they are mechanically enforceable) and Notes (where they are taste).

**No `NEEDS RESEARCH` findings.** Every weakness was resolvable from authority docs (ADR-0007 + ADR-0008 + phase-arch-design + the four Phase-0 adversarial tests in `tests/adv/` + the existing parser implementations). Stage 3 skipped per skill's token-economy guidance.

## Critic findings

### Coverage (CV-)

- **CV-1 [block] — Cap-event payload underspecified.** Original AC-7 only asserted "the event was logged with `parser_kind` set correctly". The production code in `parsers/_io.py:91-100` and `parsers/_depth.py:112-121` emits four structured fields: `parser_kind`, `cap_kind`, `cap`, `path` (plus a legacy `parser` alias for back-compat). A regression that drops `cap_kind`, drops `cap`, or substitutes `path` with the relative form would silently pass the original assertion. **Resolution:** new AC-13 (size-cap event) + AC-14 (depth-cap event) pin all four fields, with literal `cap_kind ∈ {"size", "depth"}` per S1-02/S1-03 precedent.
- **CV-2 [block] — `cap_kind` literal vocabulary not pinned.** The story did not specify whether `cap_kind` is `"size"` / `"depth"` (settled in S1-02 / S1-03) or `"bytes"` / `"depth"` (the wording the story's parent specs sometimes used). `_validation/S1-03-safe-yaml-parser.md` documents the cross-parser convention as `"size"` / `"depth"`. **Resolution:** AC-13/AC-14 pin the literals.
- **CV-3 [harden] — `errors` list could contain ID + noise.** Original AC-1 said `"pnpm_lock.depth_cap_exceeded" in <errors>` — true even if the probe also appended a spurious `lockfile.multi_present` or `pnpm_lock.malformed`. **Resolution:** AC-1, AC-2, AC-4 now assert `errors == [<exact-single-ID>]` (closed-world assertion) AND `confidence == "low"`, ruling out vacuous greens.
- **CV-4 [harden] — No disk-space guard for the 600 MB fixture.** Contributors with full disks or constrained CI runners would see `OSError(ENOSPC)`; the story would fail from the wrong cause. **Resolution:** new AC-15 requires `shutil.disk_usage(tmp_path)` check + `pytest.skip(...)` when free space < 1 GiB.
- **CV-5 [harden] — No skip-on-Windows guard.** `O_NOFOLLOW` is POSIX-only. The four tests don't directly exercise symlinks, but the underlying `parsers/_io.py::open_capped` uses `O_NOFOLLOW`, which `os.open` accepts on Windows but silently ignores. The story is implicitly Linux/macOS. **Resolution:** Notes-only — the existing `tests/adv/conftest.py` autouse fixture is platform-neutral; per-test `pytest.mark.skipif(sys.platform == "win32", ...)` matches the Phase 0 precedent (`test_symlink_escape.py:34`).

### Test-Quality (TQ-)

- **TQ-1 [block] — `run_gather` fixture is fictional.** The story's TDD-plan tests reference `run_gather` as a "Phase-0 helper fixture". Phase 0 ships `_invoke_gather` (a module-local helper) at `tests/smoke/test_cli_end_to_end.py:87`, NOT a pytest fixture and NOT importable from there. An executor following the TDD plan literally would write a test that fails at collection with `fixture 'run_gather' not found`. **Resolution:** new AC-7 names a shared `tests/adv/_helpers.py::invoke_gather(repo: Path) -> Result` helper (extracted from the smoke test pattern), used by tests #1, #2, #4 (test #3 is parser-level only). The TDD-plan code samples updated to import this helper, not invoke a fictional fixture.
- **TQ-2 [block] — Pre-parse canary missing for AC-4.** AC-3 (huge_string) correctly patches `json.loads` to raise a sentinel and asserts `SizeCapExceeded` fires first. AC-4 (oversized_lockfile) had no symmetric guard — a regression that loaded the 60 MB YAML before the size check would silently pass because `yaml.CSafeLoader.load` happens to also reject some shapes. **Resolution:** AC-4 now mirrors AC-3's structure: patch `yaml.load` (the function `safe_yaml._parse_one` calls) to raise a sentinel `RuntimeError`; assert `SizeCapExceeded` raises pre-parse.
- **TQ-3 [harden] — IDs hardcoded; drift class.** Tests assert `"pnpm_lock.depth_cap_exceeded"` etc. as raw strings. The single source of truth for this vocabulary is `node_manifest._ERROR_PREFIX_BY_KIND` × `_ERROR_SUFFIX_BY_EXC`. A future PR that renames a prefix (e.g., `"pnpm_lock"` → `"pnpm"`) would silently pass the tests until the audit catches the drift. **Resolution:** AC-9 requires the test helpers construct expected IDs via the registry (or via a `tests/adv/_helpers.py::expected_lockfile_error_id(parser_kind, exc_type)` thin wrapper), with one explicit hardcoded-string assertion per ID kind retained as the human-readable anchor.
- **TQ-4 [harden] — Empty errors-set assertion missing on success path.** The four tests don't verify that the cap_exceeded event was the ONLY parser cap event emitted (a regression that also emitted a stray `cap_exceeded` for an unrelated path would silently pass). **Resolution:** AC-13/AC-14 require `len(cap_events) == 1`.

### Consistency (CN-)

- **CN-1 [block] — `warnings` vs. `errors`.** ADR-0007 §Decision: typed-exception-raised IDs land in `errors[]`; `warnings[]` is the soft-degrade vocabulary. Existing impls confirm: `node_manifest.py:413` appends `_error_id(parser_kind, exc)` to `errors`; `language_detection.py:456` appends `error_id` to `errors`. `phase-arch-design.md §Edge cases` row 1 column 5: `errors: ["pnpm_lock.depth_cap_exceeded"]`. **Resolution:** every AC that names a cap-exceeded WarningId now asserts it on `errors`, not `warnings`. Affected: AC-1 (was `warnings`), AC-2 (was `warnings`), AC-4 (was `warnings`).
- **CN-2 [harden] — Slice-path naming uses `node_manifest.warnings`.** node_manifest.py:475-481 nests warnings/errors under `manifests` (`schema_slice = {"manifests": {... "warnings": [...], "errors": [...] ...}}`), but `ProbeOutput.errors` is also surfaced at the envelope level. The story's `manifests["confidence"]` and `manifests["warnings"]` access paths in the TDD plan do not match the actual schema (confidence lives on `ProbeOutput`, not under `manifests`). **Resolution:** TDD plan rewritten to use `result.context["probes"]["node_manifest"]["confidence"]` and `result.context["probes"]["node_manifest"]["errors"]` access paths, which match the envelope contract.
- **CN-3 [block] — `DepthCapExceeded` not handled by package.json readers.** Both `LanguageDetectionProbe` (lang_det.py:454) and `NodeManifestProbe._read_package_json` (node_manifest.py:348) catch only `SizeCapExceeded`, `SymlinkRefusedError`, `MalformedJSONError`. A `DepthCapExceeded` from `safe_json.load` bubbles out; the slice does NOT contain `package_json.depth_cap_exceeded`. **Resolution:** new AC-12 (Probe-retrofit group) names the two-file edit explicitly: extend each probe's `_PKG_JSON_FAILURE` map with `DepthCapExceeded → ("package_json.depth_cap_exceeded", "low")` and add the ID to each probe's `_ERRORS` frozenset (the compile-time ADR-0007 assertion will pass; the pattern fits). The TDD-plan test #2's "green" step now lists this edit explicitly. Story scope amended in *Notes for the implementer*: this is acceptable because S2-01 / S3-05 are co-prerequisites of S5-01 (depends on S2-05, which depends on S2-01; depends on S3-06, which depends on S3-05).
- **CN-4 [harden] — Pyproject `adv` marker registration.** Original AC-5 said "if Phase 0 already registered it, skip". Phase 0 ships only the `bench` marker (pyproject.toml:183-185). **Resolution:** AC-5 now unconditionally requires the `adv` marker registration (the "idempotent" hedge was over-cautious — Phase 0 didn't register it, period).
- **CN-5 [nit] — Story Context says "≥ 20 hostile inputs"; arch lists 10.** Phase 1 ships 10 adversarial fixtures (phase-arch-design.md §"Adversarial tests"). The "≥ 20" wording in the original story context appears to be a transcription artifact from the Phase-1 Goal #5 in `phase-arch-design.md` line 22, which itself reads "≥ 20" — but the test-list in §"Adversarial tests" enumerates 10. The Goal-line "≥ 20" is the doc-bug, not the story. **Resolution:** Notes-only; this story is not the right place to fix the `phase-arch-design.md` Goal wording.
- **CN-6 [nit] — `pkg = None` in node_manifest causes early return.** node_manifest.py:390-393: when `_read_package_json` returns `None` (typed parser failure), the probe early-returns with `primary=None` and the lockfile is never read. So for AC-2 (deep-nesting `package.json`), the lockfile in the same fixture is irrelevant — only the `package_json.depth_cap_exceeded` ID surfaces. The story's TDD-plan code didn't include any lockfile in test #2's fixture; aligning. **Resolution:** AC-2 now explicitly says "no lockfile in fixture" to anchor the early-return semantics.

### Design-Patterns (DP-)

- **DP-1 [AC, harden] — Test helpers cross the rule-of-three threshold.** `tests/adv/` ships 4 Phase-0 tests + S5-01's 4 + S5-02's expected 3 + S5-03's expected 3 = ~14 adversarial tests by end of Step 5, all sharing a "build hostile fixture → assert specific typed exception → invoke gather → assert structured slice degradation" shape. The story already names `tests/adv/_helpers.py` for the "minimal Node repo" pattern; this validation extends the helper's scope to two more reusable seams. The kernel doesn't grow Open/Closed cliffs because it remains a thin function library — no class hierarchy, no registry of test-types. **Resolution (AC-form):** AC-7 names `invoke_gather(repo) -> Result` (was `run_gather` fixture); AC-8 names `assert_parser_cap_event(logs, parser_kind, cap_kind, cap_value, path)` for the structlog assertion (used by tests #1/#2 today, #5+ in S5-02/S5-03 onward). AC-9 names `expected_lockfile_error_id(parser_kind, exc_type)` for registry-driven ID construction. Each helper is one pure function with explicit-keyword signature; adding a test for `safe_toml` is one new caller of `expected_lockfile_error_id` with a new `parser_kind` literal — zero edits to the helper. Plugin-architecture-in-its-smallest-form (`parsers/_io.py:open_capped` is the same pattern: one kernel function, parameterized by a `parser_kind` literal).
- **DP-2 [AC, harden] — Functional core / imperative shell for fixture builders.** The story's `_billion_laughs_yaml(depth: int) -> bytes` is already pure (bytes-in / bytes-out, no I/O). Mirror this discipline for the other three: `_deeply_nested_package_json(depth: int) -> bytes`, `_huge_string_package_json_writer(fd, mb: int) -> None` (the 600 MB case is necessarily streaming so the "pure" version is the size argument; the writer is the imperative shell), `_padded_yaml_lockfile_writer(fd, mb: int) -> None`. Tests then use `tmp_path / name` for the imperative-shell write. **Resolution:** AC-16 requires the three small-fixture builders to be pure functions. The huge-string builder is an explicit exception (cannot fit in memory by design); test #3 calls a `_write_huge_string_package_json(path: Path, mb: int)` writer that streams in 1 MB chunks. The functional-core split makes each fixture builder deterministic, hashable, and individually unit-testable; future stories can adversarially fuzz the builders themselves.
- **DP-3 [Notes-only] — Tagged-union for `cap_kind` instead of free-string.** `parsers/_io.py:47` declares `_CAP_KIND_SIZE: Final[str] = "size"` and `_depth.py:38` declares `_CAP_KIND_DEPTH: Final[str] = "depth"` — already a constant, but the type is `str`. A `Literal["size", "depth"]` (or `StrEnum CapKind`) at the parser-event boundary would make `cap_kind="bytes"` a type-check failure rather than a runtime drift. **Not elevated to an AC** because the existing constants are sufficient at two consumers (Rule 2 — Simplicity First) and the test pins the literal values, which is the structural defense. Surfaced in Notes for a future Phase-2 promotion when more parsers join.
- **DP-4 [Notes-only] — Composition over inheritance for adversarial-test base.** The four tests share enough structure that a `class AdversarialParserCapTest` with hooks would be tempting. Rule 2 + the strong CI-debugging value of explicit per-test files (each with its own `pytest --tb=short` failure context) wins. The helpers in DP-1 give us composition without inheritance.
- **DP-5 [Notes-only] — Newtype for `WarningId` already exists.** `parsers/.../sub_schemas.json` carries the ADR-0007 pattern; `phase-arch-design.md "Data model"` declares `WarningId = Annotated[str, Pattern(...)]`. Tests can import this for type-level assertions on the IDs they construct. Notes-only because it's a polish item; the existing `_ID_PATTERN.match` runtime check in `node_manifest.py:284` is the structural defense.

## Most load-bearing fixes (block-tier)

1. **CN-1 — `warnings` → `errors` in AC-1, AC-2, AC-4.** Single most likely cause of an executor landing a wrong implementation. ADR-0007 + 3 implementation precedents fully constrain the resolution.
2. **CN-3 / new AC-12 — `DepthCapExceeded` probe-handler retrofit.** Without this, AC-2 fails against a "correct" parser. Two-file edit (`language_detection.py:188-194`, `node_manifest.py:275-279` + `:348`).
3. **TQ-1 / new AC-7 — `invoke_gather` helper, no fictional fixture.** Without this, the executor writes a test that fails at collection.
4. **CV-1 / CV-2 / new AC-13, AC-14 — Full structlog payload assertion with literal `cap_kind`.** Without this, structured-event regressions (drop `cap` field, drop `cap_kind`, substitute `path`) silently pass.
5. **TQ-2 / amended AC-3 + AC-4 — Pre-parse canary symmetry.** AC-4 must mirror AC-3's `patch + pytest.raises` shape; without it, a regression that reads-then-checks for the YAML cap silently passes.
6. **CV-3 / amended AC-1, AC-2, AC-4 — Closed-world `errors == [<id>]` assertion.** Without this, vacuous greens from pre-existing degradations slip through.

## Design-pattern lifts elevated to ACs

- **AC-7: `invoke_gather(repo)` helper** — replaces fictional fixture, single seam for CLI invocation, future-extensible to other adversarial tests.
- **AC-8: `assert_parser_cap_event` helper** — single seam for structlog event assertions.
- **AC-9: `expected_lockfile_error_id(parser_kind, exc_type)` helper** — drift guard between tests and the `node_manifest._ERROR_PREFIX_BY_KIND × _ERROR_SUFFIX_BY_EXC` registry; the helper consumes the production registry rather than re-stating it.
- **AC-16: Pure fixture-builder discipline** — pure `bytes`-returning functions for the three small fixtures; explicit streaming `Path`-writing function for the 600 MB case (necessarily imperative).

## Lifts surfaced in Notes only

- **DP-3 — `Literal["size", "depth"]`/StrEnum for `cap_kind`.** Friction-bounded today; surface for Phase 2 when the parser family grows.
- **DP-4 — Composition-over-inheritance for adversarial bases.** Rule 2 + per-test triage value; the helpers in DP-1 are sufficient.
- **DP-5 — `WarningId` Annotated type adoption in test helpers.** Polish-tier; existing compile-time pattern check in `node_manifest.py:284` is the structural defense.

## Conflicts resolved

- **CN-3 (DepthCapExceeded retrofit) vs. story scope (S5-01 is "exercise existing parsers", not "fix probe handlers").** Resolution: Consistency wins (Rule 7) — the AC that asserts the slice carries the structured ID is the contract; the implementation gap must be closed in this PR. The amendment is two lines per probe, mechanically additive (Open/Closed at the `_PKG_JSON_FAILURE` mapping). Surgical-changes (Rule 3) preserved: no other probe behavior changes; no new error categories.
- **DP-1 (helper extraction) vs. Rule 2 (Simplicity First).** Resolution: rule-of-three is met (4 Phase-0 tests + 4 S5-01 tests + future S5-02/S5-03), so extraction is YAGNI-compatible. Each helper is one pure function — no class, no registry, no abstraction beyond the immediate need.
- **CV-2 (`cap_kind` literal) vs. story silence.** Resolution: pin to `"size"` / `"depth"` per the S1-02/S1-03 implementation precedent (already locked in by `parsers/_io.py:47` and `parsers/_depth.py:38` constants).
- **TQ-3 (registry-driven IDs) vs. assertion readability.** Resolution: tests construct IDs via the helper but ALSO retain one explicit `# expected: pnpm_lock.depth_cap_exceeded` comment per assertion as the human-readable anchor — best of both worlds, and a future code-review reading the test sees both the canonical-source-of-truth and the literal expected value.
