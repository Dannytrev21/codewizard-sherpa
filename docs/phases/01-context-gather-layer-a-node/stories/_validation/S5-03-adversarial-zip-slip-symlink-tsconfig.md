# Validation report — S5-03 Zip-slip + symlink escape + tsconfig pathological adversarial corpus

**Story:** [S5-03-adversarial-zip-slip-symlink-tsconfig.md](../S5-03-adversarial-zip-slip-symlink-tsconfig.md)
**Validated:** 2026-05-15
**Validator:** phase-story-validator (scheduled task: story-validation-corrector)
**Verdict:** **HARDENED**

## Summary

S5-03 owns the **path-traversal + hand-rolled-parser-pathology** quadrant of the Phase 1 adversarial corpus — three tests pinning that (a) a hostile `package.json` symlink is refused at `O_NOFOLLOW` time and sensitive content never leaks, (b) a hostile `kustomization.yaml` with `../../etc/passwd` resources is rejected by `Path.resolve()` containment while legitimate sibling resources still process, and (c) a pathological `tsconfig.json` (nested block comments + unterminated string + circular `extends`) never hangs the gather. The story was directionally correct (the right three tests, the right `O_NOFOLLOW`/`Path.resolve()`/`jsonc.py` defense surfaces, the right "subdir/.. canary proves Path.resolve() is the mechanism" framing) but had **eight block-tier failure modes** that would have made the executor's first attempt fail for the wrong reason, and **eleven harden-tier weaknesses** around mutation-resistance, mutation-canary strength, and consistency with the S4-02 zip-slip and S5-01 adversarial-helper conventions just hardened.

The eight block-tier failures shared a common root cause: **the story's TDD-plan code samples did not match the actual signatures, event names, slice-shape contracts, and ADR-0007 field placements established by S1-02 / S1-04 / S2-02 / S4-02 / S5-01 / S5-02 in the just-shipped codebase**:

1. **`safe_json.load(tmp_path / "package.json")` is missing the mandatory `max_bytes` kwarg.** `parsers/safe_json.py:60` declares `def load(path: Path, *, max_bytes: int, max_depth: int = 64)` — `max_bytes` is keyword-only and required. The original sample call would have raised `TypeError` before O_NOFOLLOW ever fired; `SymlinkRefusedError` would never have surfaced. Same defect rejected in `_validation/S5-02-adversarial-lockfile-exec.md` TQ-1 for `safe_yaml.load`. **Resolution:** every call site is now `safe_json.load(path, max_bytes=5_000_000)` (the 5 MiB `package.json` cap established by `_PKG_JSON_MAX_BYTES` at `language_detection.py:198` and `_PARSE_MAX_BYTES` at `node_build_system.py`). The `jsonc.load(tmp_path / "tsconfig.json")` direct call has the same defect; resolved with `jsonc.load(path, max_bytes=1_000_000)` (the 1 MiB `tsconfig` cap).

2. **Slice access `result.context["probes"]["language_stack"]` is wrong on both axes.** The probe name is `language_detection` (not `language_stack`); `language_stack` is the **nested key** inside the slice (per `language_detection.py:470` and the just-shipped `tests/smoke/test_cli_end_to_end.py:107`). An executor following the story literally would see `KeyError: 'language_stack'` on the top-level `probes` block. **Resolution:** every slice access in the TDD plan uses `result.context["probes"]["language_detection"]["language_stack"]` for slice fields. The typed-exception ID `package_json.symlink_refused` is **not** observable in the YAML envelope at all — see block-tier finding #3.

3. **AC-1 asserts `package_json.symlink_refused` is in the gather YAML under `language_stack.warnings`. It cannot be, on three grounds.** (i) Per ADR-0007 §Decision and `language_detection.py:188-194`, typed-exception-raised IDs land in **`ProbeOutput.errors`**, not `warnings`. The `_PKG_JSON_FAILURE` map writes `SymlinkRefusedError → ("package_json.symlink_refused", "low")` and appends to `errors` at `language_detection.py:456`. (ii) `_seam_shallow_merge` at `cli.py:284` copies **only** `output.schema_slice` into the envelope. `ProbeOutput.errors` (where the typed ID lives) is **not** merged into the YAML — the gather envelope has `probes.language_detection.language_stack.{counts, primary, framework_hints, monorepo, warnings, errors}` only, and the `language_stack.warnings`/`language_stack.errors` arrays are explicitly empty in Phase 1 (`language_detection.schema.json:47` description: "Phase 1 emits no values here"). (iii) The probe also currently returns `warnings=[]` always at `language_detection.py:491`. **Resolution:** the symlink test is split into three independent surfaces of observation, each on the right layer:
   - **Parser-level (Layer 1):** `pytest.raises(SymlinkRefusedError)` on `safe_json.load(symlink, max_bytes=...)` — the load-bearing structural-defense pin.
   - **Probe-level (Layer 2):** invoke `LanguageDetectionProbe.run(ctx)` directly (per the established pattern at `tests/unit/probes/test_language_detection_extended.py:175-211`); assert `"package_json.symlink_refused" in out.errors` AND `out.confidence == "low"`. This is where the typed-exception → errors[] contract pins.
   - **CLI-level (Layer 3):** invoke `invoke_gather(tmp_path)` (the S5-01 helper); assert `exit_code == 0`; assert the gather output (`repo-context.yaml` + `raw/*.json` + audit record) contains **none of the sentinel-leak markers** (`"leaked-sentinel"`, `"99.99.99-sentinel"`) — the data-leak canary. The closed-world check `out.confidence == "low"` is on the probe-level layer; the CLI-level only checks the no-leak invariant. The three layers are independent observables; a regression in O_NOFOLLOW makes all three fail (Layer 1 most precisely; Layers 2 and 3 broaden the catch).

4. **The structlog-event assertion `probe.symlink_refused` or `probe.parser.cap_exceeded` (whichever Phase 1 emits — confirm at land-time) is empirically wrong: neither event fires on ELOOP.** Grepping `src/codegenie/logging.py` and the parser modules confirms only `probe.symlink.escaped` exists (emitted by `LanguageDetectionProbe._classify_symlink`-driven walker at `language_detection.py:101`, not by the parsers). The parsers raise `SymlinkRefusedError` silently — no structlog emit (`parsers/safe_json.py` does `raise` without logging; `parsers/jsonc.py:154` does the same). `probe.parser.cap_exceeded` fires only on **size** or **depth** cap violations (`parsers/_io.py`, `parsers/_depth.py`), never on ELOOP. **Resolution:** the symlink test's structlog assertion is rewritten as a **dual observable**:
   - Assert `probe.symlink.escaped` event fires exactly once with `path == "package.json"` (the walker observable — pin per `tests/adv/test_symlink_escape.py:34-83`).
   - Assert **NO** `probe.parser.cap_exceeded` event fires for the symlink (the parser doesn't emit; if a future regression replaces the ELOOP check with a size-cap path, the test catches it).
   The original "confirm at land-time" hedge is removed; the empirics are now pinned.

5. **`run_gather` fixture is fictional. The canonical helper is `tests/adv/_helpers.py::invoke_gather(repo: Path) -> GatherResult`, created by S5-01 (AC-7).** Same defect as S5-01 TQ-1 / S5-02 CV-1. **Resolution:** every code sample now imports `from tests.adv._helpers import invoke_gather` and `GatherResult`, with the parsed YAML accessible via `result.context["probes"][probe_name][...]`. New AC pinned to the import.

6. **Missing `Depends on: S5-01` (for `invoke_gather` helper and `adv` marker registration).** `pyproject.toml:178` enables `--strict-markers`; `pyproject.toml:183-185` only registers `bench`. The `adv` marker registration is owned by S5-01 (per `_validation/S5-01` AC-5). Without S5-01 landing first, every test in this story fails at collection with `pytest.PytestUnknownMarkWarning` → `pytest --strict-markers` errors out. **Resolution:** `Depends on:` line updated to `S4-02, S5-01`. New AC pins the dependency (notes the marker is not re-registered here).

7. **AC-3 (tsconfig pathological) asserts `tsconfig.malformed` may surface in `warnings`. No such ID exists in `_WARNING_IDS` or `_ERROR_IDS` for `node_build_system`.** Per `src/codegenie/probes/node_build_system.py:212-233`, the only emittable IDs from the `tsconfig` family are `tsconfig.extends_cycle`, `tsconfig.extends_depth_exceeded` (warnings), and `tsconfig.depth_cap_exceeded` (errors). `_walk_extends` at `node_build_system.py:421` **silently swallows** `MalformedJSONError` / `SizeCapExceeded` / `SymlinkRefusedError` via `break` — no warning, no error appended. A pathological tsconfig with unterminated string therefore produces a slice with `typescript: {compiler_options_path: "tsconfig.json", resolved_compiler_options: {}}` and **no warning/error surfaces** on the gather YAML; only the time-budget assertion is observable at the CLI layer. **Resolution:** AC-3 gather-level assertion rewritten to pin (i) `gather exit 0`, (ii) `result.context["probes"]["node_build_system"]["typescript"]["resolved_compiler_options"] == {}` (the silent-swallow observable), (iii) wall-clock budget < 1.0 s on the `_walk_extends` call. The "either MalformedJSONError or extends_cycle or extends_depth_exceeded" disjunction is moved to the parser-level direct call (where `jsonc.load` deterministically raises `MalformedJSONError("unterminated string")` — pinned exactly, not as a disjunction).

8. **The combined-vector tsconfig fixture is misleading.** With nested block comments at byte 0 + body `{ "extends": ..., "compilerOptions": { "out` (unterminated string), `_strip_comments` consumes the block comments cleanly, then the state machine enters `_S_STRING` on the second `"` of `"out`, runs off the end of input, and raises `_UnterminatedString` → `MalformedJSONError("unterminated string")`. The cycle file (`tsconfig.cycle.json`) is **never reached**: `jsonc.load(tsconfig.json)` raises before the body parses; `_walk_extends` never advances to the `extends` resolution. Combining all three vectors in one file conflates them under a single failure path; only the unterminated-string vector fires. **Resolution:** the pathological test is restructured as **three separate sub-cases** within one test file, each pinning its own vector:
   - **Sub-case A — nested block comments + unterminated string.** Direct `jsonc.load(path, max_bytes=1_000_000)` call; `pytest.raises(MalformedJSONError, match="unterminated string")` (the `match=` is mutation-resistance against a regression where the stripper falls through to `json.loads` on garbage bytes). Wall-clock budget < 0.5 s.
   - **Sub-case B — circular `extends` (jsonc-valid bodies).** Two well-formed JSONC files: `tsconfig.json` with `"extends": "./tsconfig.cycle.json"` and `tsconfig.cycle.json` with `"extends": "./tsconfig.json"`. Direct call: `data, warnings = _walk_extends(tsconfig_path, tmp_path)`; assert `"tsconfig.extends_cycle" in warnings`; assert `data == {}` (no compilerOptions at any depth). Wall-clock budget < 0.5 s.
   - **Sub-case C — depth-exceeded `extends` chain.** Five well-formed JSONC files chained linearly (`a → b → c → d → e → f`, six total, depth 5 exceeds the depth-4 cap at `node_build_system._TSCONFIG_EXTENDS_MAX_DEPTH`). Assert `"tsconfig.extends_depth_exceeded" in warnings` from `_walk_extends`. Wall-clock budget < 0.5 s.
   Splitting catches a mutation that breaks just one vector (e.g., a regression that drops the cycle-detection `visited` set passes Sub-case A but fails Sub-case B); the original combined fixture would have passed all three with a single check.

The eleven harden-tier weaknesses:

- **HV-1 — `/etc/passwd`-substring canary is structurally weaker than sentinel JSON.** Even with O_NOFOLLOW removed, `safe_json.load` reading `/etc/passwd` would raise `MalformedJSONError` whose `__str__` does NOT include the doc bytes (`json.JSONDecodeError.__str__` returns `"Expecting value: line N column M (char K)"` only; the document bytes live on `exc.doc` and the code at `safe_json.py:98` truncates `str(exc)` to 200 chars). The `root:x:0:0` substring would therefore likely never surface in the YAML even with the defense broken — the canary fires false negatives on the broken implementation. **Resolution:** the symlink target is now a **JSON sentinel file** under `tmp_path.parent` (writable, deterministic): `tmp_path.parent / "S5-03-leaked-sentinel.json"` containing `{"name": "leaked-sentinel", "version": "99.99.99-sentinel", "scripts": {"start": "leaked-sentinel"}}`. Without O_NOFOLLOW, `safe_json.load` would parse this successfully and the `framework_hints` walk in `LanguageDetectionProbe._post_walk` would surface the sentinel content in the slice (`primary` field would reflect the sentinel name; `framework_hints` would derive from sentinel deps). The smoking-gun observable is `"leaked-sentinel"` not appearing in `result.context_yaml_text + concat(raw_jsons)`. Mirrors `_validation/S1-02-safe-json-parser.md` AC-5 / S4-02 AC-24 sentinel-exfiltration pattern. The sentinel file is cleaned up in `finally` (and lives under `tmp_path.parent`, not `tmp_path`, so pytest's `tmp_path` cleanup wouldn't remove it; explicit `Path.unlink(missing_ok=True)` required).

- **HV-2 — Closed-world assertion on `errors` for the probe-level layer.** Original AC-1 said `"package_json.symlink_refused" in out.errors`. A pre-existing `package_json.malformed` from an unrelated path would have satisfied an `in` check vacuously. **Resolution:** probe-level AC asserts `out.errors == ["package_json.symlink_refused"]` (exact equality) AND `out.confidence == "low"`. Mirrors S5-01 CV-3 closed-world hardening.

- **HV-3 — Zip-slip test uses string-substring `/etc/passwd` canary instead of S4-02-pattern sentinel-exfiltration.** Per `_validation/S4-02-deployment-probe.md` TQ-3 (BLOCK / SECURITY): "the naive wrong impl (`str(root) + str(resource)` + `.startswith` check) bails before any `open()` happens — the spy never fires and the test passes despite the defense being broken." `Path("/tmp/x") / "../sentinel.yaml"` stringifies to `/tmp/x/../sentinel.yaml` which **does** start with `/tmp/x`. The string-substring `"/etc/passwd" not in blob` assertion has the same defect: a wrong impl that bails before reading `/etc/passwd` passes the substring check trivially. **Resolution:** the zip-slip test reuses the S4-02 sentinel-exfiltration pattern. The fixture writes a sentinel **at `tmp_path.parent / "S5-03-SENTINEL-LEAK.yaml"`** carrying `kind: Deployment` with `spec.template.spec.containers[0].ports[0].containerPort: 31337` (the same `31337` smoking-gun port S4-02 AC-24 uses). The `kustomization.yaml` lists `resources: ["../S5-03-SENTINEL-LEAK.yaml", "./valid-deployment.yaml", "./subdir/../valid-deployment.yaml"]`. The CLI-level observable is `31337 not in result.context["probes"]["deployment"]["exposed_ports"]` — a structurally-impossible-to-fake check. The naive `str(root) + ... + startswith` wrong impl provably leaks `31337` into `exposed_ports`; the correct `.resolve().is_relative_to(root.resolve())` defense rejects it. Mirrors S4-02 AC-24 verbatim.

- **HV-4 — The `subdir/..` canary deserves its own AC, not a paragraph.** Original story buries it in AC-2's `iv` clause. Mutation-resistance against the most likely wrong impl (substring-on-`..`) is the whole point of this canary; it should be a top-level AC. **Resolution:** new dedicated AC for the canary: assert that `./subdir/../valid-deployment.yaml` (which `Path.resolve()`s **inside** `tmp_path`) IS processed — the legitimate-deployment fields land in the slice. A wrong impl that string-matches on `..` rejects this valid resource; the assertion fails immediately.

- **HV-5 — `valid-deployment.yaml` is renamed to align with S4-02 fixture naming.** S4-02 fixtures use `deployment.yaml` (per `_validation/S4-02-deployment-probe.md` AC-24 example). Inventing a new name for the S5-03 fixture creates needless inconsistency. **Resolution:** the legitimate-sibling file is `deployment.yaml` (port 8080); the dot-dot-resolves-inside canary references `./subdir/../deployment.yaml`.

- **HV-6 — Pytest marker registration not asserted.** The story's tests are decorated `@pytest.mark.adv` but the marker is unregistered until S5-01 lands. **Resolution:** new AC explicitly notes the marker is registered by S5-01 (not by this story) and the story `Depends on: S5-01`; the AC asserts a collection-time invariant (`pytest --collect-only tests/adv/test_symlink_*.py tests/adv/test_zip_slip_*.py tests/adv/test_tsconfig_*.py` succeeds with no `PytestUnknownMarkWarning`).

- **HV-7 — Skip-on-Windows guard promoted to AC, not Notes.** Mirrors S5-01 AC-18 / S5-02 AC-9. All three tests carry `@pytest.mark.skipif(sys.platform == "win32", reason="POSIX-only adversarial surface")`.

- **HV-8 — `/etc/passwd`-existence skip-guard removed (no longer needed with sentinel pattern).** The symlink test no longer references `/etc/passwd` anywhere; the sentinel JSON pattern is platform-independent. Removes the original AC's `pytest.skip("/etc/passwd not present")` branch — which itself was a smell (it made the test silently pass on Docker-distroless / restricted CI runners where the structural defense should still be exercised).

- **HV-9 — Wall-clock budget of 6 s too tight; align with S5-01 / S5-02 envelope.** S5-01 budgets 15 s for 4 tests + a 10 s headroom on the 600 MB writer; S5-02 budgets 10 s p95 for 3 tests including subprocess spawn. S5-03's three tests are I/O-light (no 600 MB write, no subprocess); 6 s is achievable on CI but tight on a contributor laptop running other processes. **Resolution:** AC-budget raised to 10 s wall-clock for the three tests aggregate (`pytest -m adv tests/adv/test_symlink_escape_in_declared_inputs.py tests/adv/test_zip_slip_kustomize.py tests/adv/test_tsconfig_pathological.py`). The tsconfig per-sub-case budget stays at < 1.0 s as published in the original story.

- **HV-10 — `kustomization.yaml` schema-version key missing.** S4-02's `_kustomize_parse` reads `apiVersion: kustomize.config.k8s.io/v1beta1` + `kind: Kustomization` to validate the file is a kustomization (otherwise the YAML is treated as raw and the resources list is ignored). The original fixture in the story has these keys correctly; preserved in the hardened version. Notes-only — confirming the implementer's fixture matches the parser's expectations.

- **HV-11 — `mypy --strict` clean assertion.** Mirrors S5-01 AC-17. New ACs for `tests/adv/test_symlink_escape_in_declared_inputs.py`, `tests/adv/test_zip_slip_kustomize.py`, `tests/adv/test_tsconfig_pathological.py` to type-check under the project's `mypy --strict` config.

The synthesizer also resolved **three consistency issues**:

- **CN-A — Existing `tests/adv/test_symlink_escape.py` is the Phase-0 walker-side test (S4-05).** It tests `LanguageDetectionProbe._classify_symlink`-driven walker behavior (symlink whose final component is a symlink-to-outside-repo gets classified as "escaped" and skipped). The two are complementary (defense-in-depth). The hardened story now explicitly says **this story adds `tests/adv/test_symlink_escape_in_declared_inputs.py`** (note the longer name); the Phase-0 file `test_symlink_escape.py` is unchanged. - **CN-B — ADR-0007 colon convention.** Resolution: every WarningId in the story is a bare dotted ID (`kustomization.resource_outside_repo`, `package_json.symlink_refused`, `tsconfig.extends_cycle`, `tsconfig.extends_depth_exceeded`, `tsconfig.depth_cap_exceeded`); no colon-suffixed `:<path>` shape (per S4-01 / S4-02 CN-1 hardening — paths move to `raw/*.json`, not into the ID).

- **CN-C — `pytest.mark.asyncio` vs codebase-standard `asyncio.run`.** Per S4-02 TQ-2 / Rule 11, the codebase pattern is `asyncio.run(probe.run(ctx))` in test bodies (matches `tests/unit/probes/test_node_build_system.py:93`), not `@pytest.mark.asyncio`. The probe-level layer of the hardened symlink test uses `asyncio.run` per this convention.

Four design-pattern observations were filed as Notes (not ACs), per CLAUDE.md Rule 2 ("three similar lines is better than premature abstraction"):

- **DP-1 — Sentinel-fixture helper extraction deferred.** The `tmp_path.parent / "S5-03-*"` sentinel pattern appears once here (symlink target) and once (zip-slip resource). The S4-02 sentinel-exfiltration pattern uses `containerPort: 31337` for K8s; S5-03 reuses that exact convention for the zip-slip test and adds a JSON-package-skeleton sentinel for the symlink test. - **DP-2 — `_walk_extends` test helper exists already.** Sub-case B and C of the tsconfig test directly call `_walk_extends` (an already-exported pure helper at `node_build_system.py:373`). No new test-helper extraction needed — the existing module surface IS the right test seam.

- **DP-3 — Three-layer observation (parser / probe / CLI) for the symlink test is the right shape for defense-in-depth tests.** Each layer pins a distinct invariant; a regression in O_NOFOLLOW fails at the parser layer first (most precisely), then cascades. Notes-only design-pattern observation: future adversarial tests for surfaces that have multi-layer defenses (parser → probe → coordinator → writer) should adopt this layered-observation pattern; surfaces with a single defense layer (e.g., the env-strip in S5-02) don't need it.

- **DP-4 — The cycle-detection `visited: set[Path]` in `_walk_extends` is itself worth a property-based test for Phase 2.** Notes-only: a follow-up Phase-2 story could `hypothesis.strategies.lists(small_paths)` against `_walk_extends` to assert the visited-set invariant; deferred per Rule 2.

The original **6 ACs** are now reorganised into **6 named AC groups with 21 individually-verifiable assertions** plus a **3-item out-of-scope cross-reference list** preventing the executor from accidentally landing the Phase-0 walker test, the S4-02 unit-level zip-slip test (already pinned at probe-PR time), or the property-based `_walk_extends` fuzz (Phase 2).

## Critic findings

### Coverage (CV-)

- **CV-1 [block] — `safe_json.load(symlink)` missing `max_bytes` kwarg.** `parsers/safe_json.py:60` makes it mandatory. Same defect as S5-02 TQ-1 against `safe_yaml.load`. **Resolution:** all sample calls pass `max_bytes=5_000_000` (the 5 MiB `package.json` cap). `jsonc.load(...)` calls pass `max_bytes=1_000_000` (the 1 MiB `tsconfig` cap).
- **CV-2 [block] — Probe-name `language_stack` (top-level) does not exist.** The probe is `language_detection`; `language_stack` is the nested slice key. Original AC-1's `result.context["probes"]["language_stack"]["warnings"]` fails at `KeyError: 'language_stack'`. **Resolution:** Layer-3 (CLI) assertions use `result.context["probes"]["language_detection"]["language_stack"][...]` for slice fields and switch to Layer-2 (probe) for `out.errors` assertions.
- **CV-3 [block] — `package_json.symlink_refused` does not surface in the gather YAML.** `_seam_shallow_merge` (`cli.py:284`) merges only `schema_slice`; `ProbeOutput.errors` is not in the envelope. `language_stack.warnings`/`language_stack.errors` arrays are empty in Phase 1 per the schema spec. **Resolution:** the typed-ID assertion is moved to the probe-level layer (direct `probe.run(ctx)` returning `ProbeOutput`); the CLI-level layer asserts the no-leak invariant only.
- **CV-4 [block] — Structlog event guess wrong on both candidates.** Neither `probe.symlink_refused` (doesn't exist) nor `probe.parser.cap_exceeded` (size/depth caps only, never ELOOP) fires on symlink refusal. **Resolution:** assert `probe.symlink.escaped` fires once with `path="package.json"`; assert no `probe.parser.cap_exceeded` fires for the symlink (mutation guard).
- **CV-5 [harden] — `out.errors` open-world `in` check vacuous-green class.** A pre-existing unrelated error would pass `in` trivially. **Resolution:** closed-world `out.errors == ["package_json.symlink_refused"]` + `out.confidence == "low"` per S5-01 CV-3.
- **CV-6 [harden] — No "valid resource still processed" positive control on the zip-slip test beyond the dot-dot canary.** A regression that broke processing for ALL resources (not just the escaped one) would not be caught. **Resolution:** new AC asserts that `./deployment.yaml` (legitimate sibling, port 8080) IS in `exposed_ports`.
- **CV-7 [harden] — `subdir/..` canary buried in AC clause iv.** Mutation-resistance against the most likely wrong impl deserves its own top-level AC. **Resolution:** new dedicated AC for the canary; failure surfaces immediately on a substring-on-`..` regression.

### Test-Quality (TQ-)

- **TQ-1 [block] — `run_gather` fixture is fictional.** Canonical helper is `tests/adv/_helpers.py::invoke_gather` (S5-01 AC-7). Same as S5-01 TQ-1. **Resolution:** all CLI-level test code imports `from tests.adv._helpers import invoke_gather, GatherResult`.
- **TQ-2 [block] — `/etc/passwd` substring canary is structurally weak.** `JSONDecodeError.__str__` doesn't include doc bytes; the `root:x:0:0` substring would not surface in the YAML even with O_NOFOLLOW dropped — false negative on the broken impl. **Resolution:** sentinel JSON file at `tmp_path.parent / "S5-03-leaked-sentinel.json"` with `"name": "leaked-sentinel"` and `"version": "99.99.99-sentinel"`; Layer-3 asserts `"leaked-sentinel"` not in YAML or `raw/*.json`.
