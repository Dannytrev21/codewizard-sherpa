# Validation report: S1-03 — Makefile + uv lock + bootstrap targets

**Validated:** 2026-05-12
**Verdict:** HARDENED
**Validator version:** phase-story-validator v1
**Story file:** [`../S1-03-makefile-bootstrap.md`](../S1-03-makefile-bootstrap.md)

## Summary

S1-03 wires the project's imperative surface — the `Makefile` plus a committed `uv.lock` — that every reviewer, contributor, and CI job invokes. As written, the story's goal, scope, references, and goal-to-AC trace were structurally sound (no `block`-class issues). The original eight ACs and five-test TDD plan, however, repeated three failure patterns already cataloged on S1-01 and S1-02 and added two new ones specific to Make's recipe surface:

1. **Substring overlap.** Test 3 asserted `"pip install" in body` to verify the fallback branch — but `"pip install"` is a substring of `"uv pip install"`. A single-line `bootstrap: \n\t uv pip install -e ".[dev]"` would have passed both branch checks while silently dropping the fallback. Real mutation slip.
2. **Recipe body invisible to tests.** ACs 4–6 named load-bearing tool invocations (`ruff check`, `ruff format --check`, `mypy --strict src/`, `pytest -q`, `mkdocs build --strict`, `python -m codegenie audit verify`) but no test parsed any recipe body. A lazy `lint:\n\techo lint` implementation would have passed every pre-validation test. This is the **most consequential** gap and the same pattern S1-02's "introspection-only TDD plan" finding flagged.
3. **`uv.lock` existence ≠ `uv.lock` validity.** AC-7 said the lockfile is "generated by `uv lock` against `pyproject.toml`'s current dependency set" but Test 4 only checked file existence and non-emptiness. A lockfile predating S1-01's final `pyproject.toml` (e.g., before `pydantic` was added per ADR-0006) would have passed.
4. **Process AC clause.** AC-8 included "was committed" — the same git-history-only clause already relaxed on S1-01 and S1-02 because the executor's Validator pass works against the working tree.
5. **POSIX-sh invariant only documented.** Implementer notes warned about bash-isms (`[[`, `function NAME()`) but no test caught a regression. macOS default `sh` may be bash (masking the bug locally); Linux CI's `sh` may be `dash` (breaking on push). The "exits 0 on macOS and Linux" goal needed a structural test, not just prose.

We rewrote Test 3 (substring fix), Test 5 (committed to prerequisite-style with no-recipe-body check), and added Tests 6–13 inline as five new test functions plus a recipe-body extraction helper. Three new ACs (one strengthened with a new clause; two new structural ACs) were added; ACs 2, 4, 5, 6, 7 were strengthened in place; AC-3's wording was committed to prerequisite-style (resolving the AC-vs-test mismatch); AC-8's process clause was relaxed. No structural rewrites; the story's goal, scope, and dependency on S1-01 are unchanged. Verdict: HARDENED.

## Findings by critic

### Coverage critic findings — S1-03

#### F1 — Recipe contents of `lint`, `typecheck`, `test`, `fence` not test-verified
- **Severity:** harden
- **What's wrong:** AC-4 (original) named four target/command pairs (`make lint` → `ruff check . && ruff format --check .`, etc.) but the TDD plan never parsed a recipe body. A lazy `lint:\n\techo lint` impl passes every original test. The same applies to `typecheck`, `test`, `fence`.
- **Proposed fix:** Add per-target recipe-body assertions (Tests 6–9) using a `_recipe_body(target, text)` regex helper that extracts the tab-indented lines following a target line.
- **Confidence:** high
- **Source:** AC-to-test trace + mutation thought experiment (mirrors S1-02's "introspection-only TDD plan" finding)

#### F2 — `clean` target body unspecified and untested
- **Severity:** nit
- **What's wrong:** AC-1 requires `clean`. Refactor §5 enumerates what `clean` should remove. Neither the AC nor the TDD plan verifies it. A `clean:\n\t@:` no-op passes AC-1.
- **Proposed fix:** No story-level edit. The implementer notes already mark `clean` as "convenience, not contract." Promoting the body to an AC adds noise without enforcement value (a no-op `clean` doesn't break Phase 0 exit; the convenience is for human ergonomics). Surfaced for awareness; no edit applied.
- **Confidence:** medium
- **Source:** AC-to-test trace; design intent ("convenience, not contract" — implementer notes)

#### F3 — AC-7 lockfile parity with `pyproject.toml` not test-verified
- **Severity:** harden
- **What's wrong:** AC-7 says `uv.lock` is "generated by `uv lock` against `pyproject.toml`'s current dependency set." Test 4 checks only file existence and non-emptiness. A lockfile predating S1-01's final `pyproject.toml` passes. Phase 4+ depends on `[agents]` slot consistency per ADR-0006; a stale lock invalidates `make bootstrap` reproducibility.
- **Proposed fix:** Add `test_uv_lock_is_in_lockstep_with_pyproject_dep_set` — parse both files via `tomllib`; assert the package-name set declared in `[project.dependencies]` ∪ `[project.optional-dependencies.dev]` is a subset of `uv.lock`'s `[[package]]` names (case-insensitive, normalizing `_`↔`-`). Avoids requiring `uv` at test time.
- **Confidence:** high
- **Source:** mutation thought experiment + ADR-0006 dependency-closure contract

#### F4 — AC-8 "was committed" — process clause unverifiable from working tree
- **Severity:** nit (already-cataloged pattern)
- **What's wrong:** Same as S1-01 / S1-02: requires git-history inspection; the executor's Validator pass works against the working tree.
- **Proposed fix:** Drop the clause; restate as "exists and is green."
- **Confidence:** high
- **Source:** AC verifiability check (cataloged pattern)

#### F5 — Goal "exits 0 on macOS and Linux" lacks a structural bridge to tests
- **Severity:** harden
- **What's wrong:** Goal explicitly cross-platforms. Implementer notes warn about bash-isms in `/bin/sh` recipes. No test catches a bash-ism regression. macOS-specific failure mode: default `sh` symlinks to `bash` on many setups, so a `[[ ... ]]` recipe runs locally and breaks on Linux CI's `dash`.
- **Proposed fix:** Add `test_makefile_recipes_contain_no_bash_isms` — scan every required-target recipe body for `[[`, `]]`, and `function NAME()` syntax; assert none present.
- **Confidence:** high
- **Source:** goal-to-test trace + implementer-note hardening

#### F6 — `make audit-verify` recipe content untested
- **Severity:** harden (subsumed by F1 fix)
- **What's wrong:** AC-6 names `python -m codegenie audit verify` as the load-bearing command. No test verifies the recipe body contains it. A lazy `audit-verify:\n\t@:` passes AC-1.
- **Proposed fix:** `test_audit_verify_recipe_invokes_codegenie_audit_verify` — substring assertion on `"python -m codegenie audit verify"` in the recipe body.
- **Confidence:** high

#### F7 — `make docs` recipe content untested; `--strict` flag untested
- **Severity:** harden (subsumed by F1 fix)
- **What's wrong:** AC-5 names `mkdocs build --strict`. The `--strict` flag is load-bearing per `phase-arch-design.md §CI gates` job 5 (mkdocs warnings → errors). A lazy `docs:\n\tmkdocs build` (no `--strict`) silently downgrades the gate.
- **Proposed fix:** `test_docs_recipe_invokes_mkdocs_build_strict` — substring assertion on `"mkdocs build --strict"`.
- **Confidence:** high
- **Source:** mutation thought experiment + arch §CI gates load-bearing flag

### Test-Quality critic findings — S1-03

#### F1 — Test 2 (declared/PHONY) — robust as written
- **Severity:** —
- **What's wrong:** Nothing. The regex is sound; `.PHONY:` aggregation across multiple lines is handled correctly.

#### F2 — Test 3 substring overlap: `"pip install"` ⊂ `"uv pip install"`
- **Severity:** harden
- **What's wrong:** `assert "pip install" in body` is satisfied by a body containing only `uv pip install -e ".[dev]"` because `"pip install"` is a substring of `"uv pip install"`. The fallback branch can vanish silently; the test still passes. Real mutation slip with a lazy implementation.
- **Proposed fix:** Replace the assertion with `assert "python -m pip install" in body` — the string `"python -m pip install"` does not overlap with `"uv pip install"`. Bonus: matches AC-2's exact wording, removing one layer of indirection.
- **Confidence:** high
- **Source:** mutation thought experiment

#### F3 — Test 5 regex doesn't reject `check` recipe-style implementations
- **Severity:** nit
- **What's wrong:** The regex `^check:\s*(.+)$` accepts only prerequisite-style (`check: lint typecheck test fence`) but the original AC-3 wording suggested recipe chaining (`make lint && make typecheck && ...`). A correct recipe-style impl would fail the test even though it satisfies AC-3.
- **Proposed fix:** Commit AC-3 to prerequisite-style (the form Make-idiomatically stops at first failure on); add an explicit assertion that `check` has no recipe body. AC-vs-test mismatch resolved by tightening the AC.
- **Confidence:** high
- **Source:** AC-vs-test cross-reference

#### F4 — TDD plan has no per-target recipe-body assertions
- **Severity:** harden
- **What's wrong:** Same root cause as Coverage F1. A test plan that only checks declared targets and dependency lists cannot prove the recipes invoke the load-bearing tools.
- **Proposed fix:** Add Tests 6–11 (per-target recipe-body assertions). One `_recipe_body(target, text)` helper extracts the tab-indented lines following a target; six tests assert load-bearing substrings.
- **Confidence:** high
- **Source:** Rule 9 (Tests verify intent, not just behavior)

#### F5 — No behavioral bridge (e.g., `make -n` dry-run) — and that's OK
- **Severity:** observational (no edit)
- **What's wrong:** Unlike S1-02 where a subprocess `ruff check` test was needed to bridge config-vs-behavior, here the recipe-body parse is sufficient. `make -n` would require `make` installed at test time and add no signal beyond the parse — the recipes are short, the tools are externally well-defined, and the load-bearing properties are textual (does the right command appear in the body). Skipping the subprocess bridge keeps the tests fast and dep-free.
- **Confidence:** medium
- **Source:** test-cost vs signal tradeoff

### Consistency critic findings — S1-03

#### F1 — AC-3 wording vs. green example mismatch
- **Severity:** harden
- **What's wrong:** AC-3 (original) said `make check` "runs (in order) `make lint && make typecheck && make test && make fence`" — that's recipe-chaining language. The green example in the TDD plan uses prerequisite syntax (`check: lint typecheck test fence`). Test 5 only accepts prerequisite syntax. Three sources of truth, two phrasings, one accepted form.
- **Proposed fix:** Reword AC-3 to commit explicitly to prerequisite-style with no recipe body. Note that Make's prerequisite semantics are equivalent to `&&` chaining for stop-on-first-failure.
- **Confidence:** high
- **Source:** AC-vs-TDD-plan cross-reference

#### F2 — AC-6 uses `python -m codegenie audit verify`, not the entry-point script `codegenie`
- **Severity:** — (intentional)
- **What's wrong:** Nothing. `pyproject.toml` from S1-01 declares the `codegenie` script entry; the story uses `python -m codegenie` for robustness against PATH-activation gaps in the dev workflow. This is consistent with `__main__.py` shipping in S1-01 (per implementer notes) and avoids the "console script not on PATH" trap on dev installs without venv activation.
- **Confidence:** high
- **Source:** implementer-note cross-reference + `pyproject.toml` script-entry awareness

#### F3 — `make test` runs `pytest -q` (no coverage flags) but CI test job runs `pytest -q --cov=src/codegenie --cov-branch --cov-fail-under=85`

So `pytest -q` from the `test` target picks up the coverage gate from `addopts`; `make test` and CI invoke the same effective pytest command. No story-level divergence.
- **Confidence:** high
- **Source:** S1-02 `pyproject.toml` `[tool.pytest.ini_options]` shape

## Verdict rationale

HARDENED. Zero `block` findings — the story's goal, scope, references, and AC-to-goal traceability were sound. Six `harden` findings clustered around three root causes: (1) recipe bodies were named in ACs but never parsed by tests (a lazy `lint:\n\techo lint` would have shipped); (2) Test 3's substring `"pip install"` overlapped with `"uv pip install"`, allowing the fallback branch to vanish silently; (3) AC-7 verified file existence but not lockfile/pyproject parity. Three `nit` findings were resolved with minor edits (AC-3 wording committed to prerequisite-style; AC-8's process clause relaxed) or surfaced-but-not-fixed (`clean` body intentionally not contract-pinned).

After the edits, an obviously wrong implementation — `uv pip install` only with no fallback, or `lint:\n\techo lint`, or `docs:\n\tmkdocs build` (no `--strict`), or a stale `uv.lock` predating `pydantic`'s addition, or a bash `[[ ... ]]` test inside `bootstrap` — would now fail at least one TDD assertion. The story meets the "STRONG bar" for executor handoff.
