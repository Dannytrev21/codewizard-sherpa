### Deviations from "Files to touch"

These are technical necessities; flagging for traceability.

**Deviation 1 — `src/codegenie/cli.py` stub created (NOT in Files-to-touch).**

The story Notes describe the `codegenie.cli` contract as "vacuously enforceable until S4-02 lands `src/codegenie/cli.py`" and AC-7(b)'s test contains an explicit `pytest.skip(...)` when cli.py doesn't exist. In practice, import-linter v2.11 errors out immediately when `source_modules` references a non-existent module:

```
Module 'codegenie.cli' does not exist.
```

That makes AC-10(a) `test_lint_imports_exits_zero_on_current_tree` unsatisfiable without a real `src/codegenie/cli.py`. I landed a minimal placeholder (docstring + `from __future__ import annotations`) that S4-02 will expand into the real CLI surface. The AC-7(b) `pytest.skip` branch becomes dead code but the assertion still runs and validates the stub. AC-10(b) canary's `original = CLI.read_text() if CLI.exists() else None` already handles the stub-present case correctly (saves stub, plants violation, restores stub).

**Deviation 2 — `include_external_packages = true` added to `[tool.importlinter]`.**

Not in the story's example config, but import-linter v2 errors out with:

```
The top level configuration must have include_external_packages=True when there are external forbidden modules.
```

`yaml`, `jsonschema`, `pydantic`, `blake3`, `structlog` are all external (not under `root_packages = ["codegenie"]`). Setting this flag is the documented fix.

**Deviation 3 — `.pre-commit-config.yaml` updated.**

Added `packaging` to the mypy hook's `additional_dependencies` because the isolated pre-commit env doesn't share the project venv. Without this, `pre-commit run --all-files` fails on `Cannot find implementation or library stub for module named "packaging.requirements"`. `make typecheck` works because it runs in-venv, but the pre-commit firewall is the load-bearing local gate per S1-04 — both need to be green.

### Lesson learned (for `_lessons.md`)

When a story declares a contract on a module that doesn't exist yet ("vacuously enforced until later story X"), check what the tool actually does with a missing source module. import-linter v2 hard-errors rather than no-op'ing — the practical resolution is a placeholder stub, NOT deferring the contract.
