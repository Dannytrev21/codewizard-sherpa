### Three friction points worth recording

1. **`from __future__ import annotations` defeats naive `inspect.signature(...).return_annotation` checks** — annotations become strings (`'None'`), not the objects. Fixed with `inspect.signature(fn, eval_str=True)` + accepting `(None, type(None))` to survive a future drop of the future-import.

2. **`list` does not accept attribute assignment.** The validated story used `confirm_spy.return_value = True`/`hasattr(calls, "return_value")` on a `list` fixture; that raises `AttributeError` at runtime. Replaced with a `_ConfirmSpy` class exposing `.calls`, `.return_value`, and `__call__`/`__len__`/`__eq__` — same ergonomics, no runtime trap.

3. **`capsys`-based JSON capture is not viable for this test surface.** Three independent failures cascaded from the validated story's `capsys.readouterr().err + json.loads` pattern:

   a. **pytest rotates `sys.stdout`/`sys.stderr` between fixture setup and the test body.** `monkeypatch.setattr(sys.stdin, "isatty", lambda: True)` lands on the pre-swap `CaptureIO`; the test body sees the post-swap stream whose `isatty` is the default built-in `False`. Repro: a 12-line standalone pytest test with `both_tty(monkeypatch, capsys)` fixture and `assert sys.stdout.isatty() is True` — fails with `<built-in method isatty of CaptureIO object at 0x...>`. Removing `capsys` from the fixture deps makes it pass.

   b. **`structlog`'s `PrintLoggerFactory(file=sys.stderr)` caches the file at config time.** If `configure_logging()` captures `sys.stderr=CaptureIO_A` and pytest later closes `CaptureIO_A`, every subsequent log call hits `ValueError: I/O operation on closed file`. The `_config_cache` in `codegenie.logging` keys by `id(sys.stderr)`, but Python recycles object ids — a previous test's `CaptureIO` can reuse the id, returning the cached config pointing at a closed stream. Tried `cgl._config_cache.clear() + structlog.reset_defaults()` in the autouse fixture; still failed because the BoundLoggerLazyProxy held by the helper module bound on first use.

   c. **The `level` key.** capsys-captured JSON has `"level": "warning"` (from the `add_log_level` processor); `structlog.testing.capture_logs` output has `"log_level": "warning"` (from its `LogCapture` proc). Same value, different key.

   **Resolution.** Switched to `structlog.testing.capture_logs()` — already the codebase's pattern in `tests/unit/test_cli_orchestration.py` and `tests/unit/test_cli_flags.py:test_cache_gc_stub_emits_exact_event_name`. The `level` → `log_level` rename was a single search/replace. AC-22's stated intent ("don't use `caplog.records[i].message`") is preserved — `capture_logs` is NOT `caplog`. Story `Divergences` section documents the swap with the runtime evidence above.

   **Defensive belt-and-braces.** Also added a `_logger()` resolver inside `gitignore.py` so future tests using PrintLogger still get a fresh logger per call (instead of a binding cached at import time). Costs one dict lookup per call; production cost is negligible (one gather invocation per CLI run).
