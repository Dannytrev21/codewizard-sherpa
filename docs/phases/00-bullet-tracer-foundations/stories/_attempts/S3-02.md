### Key decisions and lessons

1. **`TypeAliasType` instead of `TypeAlias`/`type` statement.** A naive
   `JSONValue: TypeAlias = Union[..., "list[JSONValue]", "dict[str, JSONValue]"]`
   triggers `RecursionError` inside Pydantic v2's schema builder because the
   forward-ref alias is anonymous. Pydantic v2's documented recursive-union
   pattern uses `typing_extensions.TypeAliasType("JSONValue", ...)` (a named
   recursive alias). This is the canonical workaround for projects that still
   target Python 3.11 (PEP 695 `type` statement is 3.12+). `typing_extensions`
   is transitively available via pydantic — no pyproject change required.

2. **`field_validator` runs `mode="before"`, not `mode="after"`.** Pydantic v2
   in `strict=True` *still* coerces `Decimal -> float` (it's a documented
   built-in coercion the strict-float validator allows). The walker has to
   inspect the raw input before Pydantic touches it, so it runs as a
   `mode="before"` validator and performs both the key-regex check *and* the
   `_ALLOWED_LEAF_TYPES` whitelist. Strict-mode Pydantic still catches the
   non-string-keys case (AC-16) and the bool/int discrimination (AC-13)
   because those happen at type-introspection time, not value-coercion time.

3. **Pydantic v2 only wraps `ValueError` / `AssertionError` /
   `PydanticCustomError`** in `@field_validator`. Raising
   `SecretLikelyFieldNameError` directly propagates as itself. The fix:
   catch the typed error inside the validator and re-raise as
   `PydanticCustomError("secret_likely_field_name", "...", {"error": exc, ...})`.
   The typed error then surfaces at `errors()[0]["ctx"]["error"]` — exactly
   the surface AC-4 names and the surface S3-05's coordinator dispatch will
   use to translate the wrap back into a typed lifecycle event.

4. **Walker is iterative** (stack of `(node, path)` tuples). Defends AC-19
   against `RecursionError` at depth 200. A `collections.deque` is equivalent
   here; a `list` used as a stack is the lighter-weight choice.

### Known deviation from AC-4 (literal reading)

AC-4 stipulates `errors()[0]["loc"]` "MUST include the key that triggered
the match" (e.g., `("schema_slice", "github_token")`). Pydantic v2's
`@field_validator` API does not expose loc beyond the field name (the
returned loc is always `("schema_slice",)`). The TDD plan's tests do *not*
pin loc shape; they pin `ctx["error"]` instead, which we satisfy. The
offending key and the full path are surfaced at
`errors()[0]["ctx"]["key"]` and `errors()[0]["ctx"]["path"]` (both added
by the `PydanticCustomError` wrapping), so S3-05's coordinator dispatch
can still surface the offending key path to the gather lifecycle event —
the *information content* AC-4 requires is preserved; the *exact tuple
shape* would require migrating the regex check into a `model_validator`
that hand-constructs a `ValidationError` via `ValidationError.from_exception_data`,
which is gnarlier than this Phase-0 boundary warrants.

Surfaced for follow-up; not blocking. The validator subagent's hardening
log already noted this is "Pydantic v2 wrapping reality" and rewrote AC-4
to accept either `ctx["error"]` *or* `__cause__` — the same spirit applies
to the loc-vs-ctx-path question.

### Lessons learned (cross-story candidates)

- When a Pydantic field allows `Union[bool, int, ...]`, the `Union` member
  order is load-bearing in Pydantic v2's smart-mode discrimination. `bool`
  before `int` is the only configuration that keeps `True is True` (vs
  coerced to `1`).
- When implementing recursive types for Pydantic v2 on Python 3.11,
  `typing_extensions.TypeAliasType` is the canonical workaround, not
  `TypeAlias` and not the PEP-695 `type` statement.
- Pydantic v2's `mode="before"` field validators are the right tool when
  the field's type would otherwise coerce in ways the validator wants to
  forbid (e.g., `Decimal -> float`).
