### What worked

- **Single RED marker.** `pytest -x` on the three new test files failed
  immediately with `ModuleNotFoundError: codegenie.probes.registry` —
  the surface contract was unambiguous before a line of green code went
  in.
- **Module-level `_filter` lru_cache helper.** The story's implementer
  note flagged the `lru_cache`-on-method footgun (cached `self` leaks
  the registry). Factoring the cached function to module scope —
  `_filter(probes: tuple[type[Probe], ...], task: str, languages: frozenset[str])`
  — makes the cache key explicit, makes `_filter.cache_info()` observable
  for AC-9's mutant-killer test, and keeps each `Registry` instance
  garbage-collectable. The `_filter.cache_clear()` call in the
  `for_task` lru-cache test makes the assertion deterministic regardless
  of test ordering.
- **Modern `referencing.Registry` for `$ref` resolution.** The story's
  implementer note suggested the deprecated `jsonschema.RefResolver`,
  but `jsonschema >= 4.18` ships the `referencing` library as the
  forward-compatible path. Registering each sub-schema by its absolute
  `$id` (e.g. `https://codewizard-sherpa.dev/schemas/probes/language_detection/v0.1.0.json`)
  makes the envelope's `$ref` a stable absolute URI — no path-relative
  surprises, no base-URI string fiddling.
- **`functools.lru_cache(maxsize=1)` on `_validator()`.** AC-13's
  cache-hit pin is observable directly via `_validator.cache_info()`;
  the `cache_clear()` call at the top of the test gives a deterministic
  baseline regardless of which other tests in the file ran first.
- **Pydantic v2 `model_config = ConfigDict(frozen=True, extra="forbid")`
  cleared `mypy --strict`** without a `pydantic.mypy` plugin entry —
  S1-02's toolchain pinning already includes the plugin via Pydantic's
  package-bundled config.
- **`AuditWriter.record` stub raising `NotImplementedError`** keeps
  S3-06's deliverable boundary clean while letting callers `from
  codegenie.audit import AuditWriter` without an `ImportError`. The
  one uncovered line in `audit.py` is the `NotImplementedError`
  raise — deliberate, S3-06 covers it.

### What didn't, the first time

- **`mypy --strict` flagged `jsonschema` as untyped.** The package is
  not `py.typed`; resolution was to add `types-jsonschema` to both the
  `[project.optional-dependencies] dev` extras list and the pre-commit
  `mypy` hook's `additional_dependencies` (so the hook's isolated venv
  resolves stubs the same way the local venv does). The Makefile's
  `uv.lock` lockstep test then complained the lock file was stale —
  resolved with `uv lock` and committing the regenerated lock.
  Lesson: any new third-party dep (even a stub-only package) needs
  three coordinated edits: `pyproject.toml`, `.pre-commit-config.yaml`
  mypy hook, and a regenerated `uv.lock`.
- **`Resource(contents=…, specification=…)` is the wrong constructor.**
  `referencing.Resource`'s `__init__` doesn't accept those kwargs;
  `Resource.from_contents(s, default_specification=DRAFT202012)` is
  the right factory. mypy's `[call-arg]` error caught it before any
  test ran.

### Lesson for next time

- When the story's implementer notes name a deprecated API
  (`jsonschema.RefResolver`), check whether the supported replacement
  is already in the dep set before writing against the deprecated one
  — the test suite never asks "did you use the deprecated API?", it
  asks "did `$ref` resolve to the right slice?". The replacement
  passes the same tests and won't break under a future
  `jsonschema 5.x` upgrade.
- A new third-party dep is **three** edits, not one: `pyproject.toml`,
  `.pre-commit-config.yaml` mypy hook, and `uv lock`. Forgetting the
  hook means CI passes locally but the pre-commit mypy hook fails in
  the isolated environment — silent until the next pre-commit run.
- Module-level cached helpers expose `.cache_info()` for free. Method
  decorators don't (the descriptor wrapping eats the attribute).
  Whenever a story's AC asks for an observable cache, factor the cache
  to module scope.
