### What worked

- The story's TDD plan was directly executable — substring choices like
  `"python -m pip install"` (rather than `"pip install"`, which overlaps
  `"uv pip install"`) caught the trap before it materialized in code.
- Generating `uv.lock` via `uv lock` (not `uv sync`) produced a lockfile
  whose `[[package]]` table contained every name from `dependencies` and
  `dev` — the lockstep parity test in AC-7 passed on first run.
- The `_recipe_body` regex was correctly scoped to tab-indented lines only,
  so the file-head comment block (which deliberately mentions `[[ ... ]]` to
  document the prohibition) didn't trigger the AC-9 bash-ism check.

### What didn't / surprises

- `ruff format --check tests/unit/test_makefile_targets.py` flagged formatting
  on first run. Re-ran `ruff format` (without `--check`) to fix; tests still
  pass after the reformat. Lesson: run `ruff format` before `ruff format --check`
  on hand-authored test files, especially with multi-line `assert (..., "...")` calls.

### Lesson for future stories

- When the story's TDD plan includes Python-string substring assertions,
  pick substrings that are **anti-subsequences** of obvious near-misses
  (e.g., `python -m pip install` excludes `uv pip install`). This pattern
  is reusable in S1-05 (fence test) and S2-04 (allowlist).
