# Phase 4 — Vuln remediation: LLM fallback + solved-example RAG: Devil's-advocate critique

**Reviewed by:** Devil's-advocate critic subagent
**Date:** 2026-05-12

## Method

I read all three designs and attacked each on its own terms. I do not propose alternatives.

## Attacks on the performance-first design

### Concrete problems

1. **Problem:** `sentence-transformers/all-MiniLM-L6-v2` is a general-purpose semantic-similarity model trained on QA/STS pairs. There is no evidence in the doc that it embeds CVE-advisory text or diff snippets in a way that places "same vulnerability class on same package" closer than "different vulnerability on similar package name." The published benchmark gains the design relies on (§Component 2 trade-offs: "recall@3 ≥ 0.85 on the labeled set with MiniLM") are *asserted* against a fixture set that doesn't exist yet, and the design defaults `τ_hit = 0.86`. MiniLM cosine scores compress hard above 0.7 on lexically similar text; an advisory whose summary repeats the package name twice ("lodash prototype pollution in lodash") will trivially clear 0.86 against any other lodash advisory, including unrelated ones — silent wrong-bump.
   **Why it matters:** Tier-2 *short-circuits the LLM* on `score ≥ τ_hit`. A bad embedding model makes the cache hierarchy a wrong-answer cache, and §Failure-mode row 7 explicitly accepts "the first 3 wrong matches per neighborhood waste a validate cycle (~60 s each). Acceptable." That presumes the validator catches every wrong match. The validator catches `npm ci` and `npm test`, neither of which catches "we bumped the wrong package to the right version" if both packages share a transitive dependency that quiets the failure.
   **Where:** §Component 1 (Tier-2 RAG description), §Risks #1, §Goals table ("RAG hit rate ≥ 55 %").

2. **Problem:** The `SolvedExampleRecorder` writes to the local store the instant Phase 3 validation passes, with no human in the loop. The design literally says "regardless of which tier emitted the plan" (§Component 5). ADR-0009 says humans always merge. In Phase 4 the workflow ships only to a local branch — there is no merge event, so the recorder is writing **un-merged** LLM output into the corpus that gates all future workflows. A single LLM hallucination that happens to pass `npm test` (which proves nothing about correctness — it proves the test suite didn't fail) becomes a permanent few-shot example. The misleading-match negative-example logic (§Failure-modes row 7) makes the *bug* worse: a wrong match writes a *negative* example that biases similarity *for that neighborhood downward*, but the original wrong positive is still in the store.
   **Why it matters:** Phase 11 introduces actual PR opening and human merge as the writeback gate. This design ships a writeback policy in Phase 4 that Phase 11 will have to *roll back*. Worse, by the time Phase 11 ships, the local store has months of pre-merge writebacks and there is no audit trail distinguishing them.
   **Where:** §Component 5 ("SolvedExampleRecorder — async, fire-and-forget"), §Data flow step 9, §Open questions #3.

3. **Problem:** The cross-encoder rerank is alluded to in the lens summary ("(3) **time-to-first-diff**") and concretely added in §Component 1 Tier-2 as a "**lexical overlay (BM25 over advisory text + recipe-failure error string)**." That's BM25, not a cross-encoder. The lens summary's third priority either disappears or is silently swapped for BM25 mid-document. If a real cross-encoder rerank is intended, where does the model come from, what's its digest, what's the offline-pin path? If only BM25, the doc should not invoke cross-encoder language. The lens-summary/Component-1 mismatch is exactly the kind of slip that lands as an unstated dep in code.
   **Where:** §Lens summary line "(3) time-to-first-diff" vs §Component 1 Tier-2 "rerank by a cheap lexical overlay (BM25)."

4. **Problem:** "No streaming" is wrong — the design proudly uses streaming SSE with an incremental JSON parser (§Component 3). Streaming + structured output via Anthropic's server-side `response_format` for Sonnet 4.7 is asserted as a 2026 capability ("Anthropic 2026 supports server-side structured output for Sonnet 4.7"). That feature may not be GA on the model version the test cassettes were recorded against. When the cassette pins a streaming response_format payload and the production SDK rolls back the feature behind a flag, every cassette is a stale snapshot of an API contract that no longer exists. The `StreamCancelled` "charged for partial output only" optimization (§Component 3, §Failure-modes row 5) compounds the fragility: cassettes record *the choice of when to cancel mid-stream*, and any change to the parser's early-cancel heuristic invalidates the cassettes, which were already only byte-replay (see cross-cutting blind spot #1).
   **Where:** §Component 3 "Streaming + structured output", §Failure-modes rows 4–6.

5. **Problem:** `τ_bypass = 0.85`-equivalent (called `τ_hit = 0.86` here) short-circuits to `rag_exact` with **no LLM verification of the match**. The design's defense is "the validator catches wrong outputs" + "auto-raise `τ_hit` for that cluster after 3 fails" (§Risks #1). The 3-fail learning rule has a startup problem: the first three workflows in a freshly seeded corpus that hit a wrong neighborhood each burn one full sandbox run (~60s + `npm ci`/`npm test`) plus whatever damage the wrong patch does to the working tree before `git apply --check` fails. In a portfolio scan, "3 fails per neighborhood" *across each neighborhood* is unbounded. There is no top-level cap on wrong-match retries before the system disables RAG entirely.
   **Where:** §Risks #1 ("first 3 wrong matches per neighborhood waste a validate cycle"), §Failure-modes row 7.

### Hidden assumptions

1. **Assumption:** Anthropic's "ephemeral" prompt cache key is byte-stable across SDK versions, OS locale, and Python serialization order, and Anthropic publishes the cache-key derivation rule.
   **What breaks if wrong:** The §Risks #2 cited "byte-stable system block" is asserted as defendable with a golden-file test, but the cache key isn't computed by Python — it's computed by Anthropic. A locale change (`LANG=C` vs `LANG=en_US.UTF-8`) producing different Unicode normalization on a single em-dash inside the Skills manifest silently drops cache hit rate to 0 % across the entire portfolio. The "$/PR — RAG path: $0" and "Prompt-cache hit rate ≥ 80 %" goals collapse, the workflow cost balloons from $0.011 to $0.051 (4.6×, per the design's own math), and the goal table is silently wrong.

2. **Assumption:** The query-key tuple's `recipe_catalog_blake3` reliably invalidates stale tier-1 cache entries when a new recipe lands.
   **What breaks if wrong:** §Component 6 hashes the *whole* catalog directory. But Phase 15 (recipe authoring) ships LLM-proposed recipes that pass `git apply --check` and validation. Those land in `catalog/` and bump the blake3 *for everything*. Every tier-1 cache entry across the portfolio invalidates on every Phase 15 recipe proposal. The "compounding savings" story collapses precisely at the moment Phase 15 is supposed to make it real.

3. **Assumption:** `chromadb.PersistentClient` running in the same Python process as the orchestrator does not block the event loop during similarity search, because the docs say it uses mmap'd parquet.
   **What breaks if wrong:** chromadb's SQLite-backed HNSW index has historically held the GIL during query at sizes well below 1k. The "in-proc mmap" framing (§Component 4) ignores that the embedding-comparison loop runs in Python. The "single similarity call returns in 5–30 ms for 100 examples" claim is the right order of magnitude *for cold-cache, single-worker dev*, not for the steady-state 8-parallel-workflow profile in §Resource & cost ("Lets a 16 GB worker host ~8 parallel workflows"). Eight workers concurrently calling the same in-process chromadb client through the GIL is contended.

### Things this design missed that a different lens caught

- **No defense against prompt injection on the LLM input path.** §Acknowledged blind spots admits this and punts to the security lens. The security design's untrusted-text fence + canary system is load-bearing; the performance design doesn't even sanitize advisory description text before feeding it to the LLM, which means a CVE description (attacker-controlled bytes in NVD) can rewrite the LLM's instructions, and the structured-output schema check is the *only* defense.
- **API key handling is not described.** The performance design imports `anthropic` directly in `src/codegenie/planner/llm/anthropic_leaf.py` and assumes the SDK picks the key up from somewhere. The security design's egress proxy + uid-jailed agent is a real architecture; the performance design just has `import anthropic`.
- **The leaf agent runs in-process with full access to the orchestrator's address space, `RepoContext`, the audit chain, and `~/.ssh`** (because there is no agent-process isolation). The security design's process boundary is invisible to this design.
- **No structural validation that the LLM-produced patch references a known recipe ID** — the LLM may emit a `ManualPatchEngine` patch that goes directly to `git apply` with whatever attacker bytes injection slipped through. The security design's "structured-plan must reference a catalog recipe ID" gate is absent here.

## Attacks on the security-first design

### Concrete problems

1. **Problem:** The LLM agent is **tool-less by construction** (§Component 4 "the agent gets **no tools** (no file_read, no bash, no anything). Its tool surface is empty"). It also has **no bind mount of the repo working tree, no bind mount of `.codegenie/cache/`**. So how does the LLM ever propose a fix that requires looking at the actual content of `node_modules/some/deep/file.js` or even `package.json`? The design says (§Component 4) the agent receives only `LlmRequest(model, system_prompt, user_prompt, max_output_tokens, ...)` of size ≤ 256 KB. The `user_prompt` therefore must contain the entire repo context the LLM will see. But the design also says (§Trust boundaries diagram) "NEVER receives: repo paths." This is internally contradictory: either the LLM sees the relevant repo source (in which case repo paths and source bytes are in the prompt, defeating the boundary commitment), or it doesn't (in which case the fallback path can't handle anything beyond "bump version X to Y," which is the *recipe* engine's job, not the LLM's).
   **Why it matters:** The roadmap exit criterion is "A breaking-change vuln (e.g., a major-version-bump CVE) is solved end-to-end with the LLM fallback." A major-version bump with breaking call-site rewrites *requires the LLM to see and rewrite call sites*. This design either silently leaks call-site source into the prompt (contradicting its own boundary) or cannot meet the exit criterion at all.
   **Where:** §Threat model / Trust boundaries (line "NEVER receives: API key in prompt body, repo paths, secret env"), §Component 4 (the LlmRequest interface; "tool surface is empty"), vs roadmap §Phase 4 exit criterion.

2. **Problem:** Per-artifact whitelisting via the implied `--allow-flagged-input=<sha256>` and the more general `--allow-*` flag family (`--allow-llm-fallback`, `--allow-cross-repo-rag`, `--allow-stale-feeds`, `--allow-test-network`, `--allow-policy-violations`, `--allow-cost-overrun`) is operationally painful enough that operators will create a portfolio wrapper script that passes them all by default. §Risks #5 explicitly acknowledges this ("Operator misuse of `--allow-llm-fallback` and `--allow-cross-repo-rag`. Blanket-enabled portfolio scripts running with both flags expand the attack surface"), and the mitigation is "documentation + dashboard alerts on flag-usage rates" — i.e., social, not structural. The design's own threat model lists "Operator-induced misuse" as an adversary, and then the design ships *exactly* the UX pattern that creates the misuse.
   **Where:** §Threat model / Adversaries assumed (operator misuse), §Components list of flags, §Risks #5.

3. **Problem:** The roadmap exit criterion is "**A breaking-change vuln is solved end-to-end with the LLM fallback and recorded into the solved-example store. Re-running the same case hits RAG, not LLM.**" The security design (§Components / `RagAccepter`, §Data flow step 17, §Open questions #9) puts the *only* RAG write path behind a separate operator command `codegenie rag accept <run-id> --merge-sha <sha> --reviewer <email>`, gated on human merge of an actual PR. **But Phase 4 does not open PRs** — Phase 11 does. So in Phase 4 there is no merge SHA, no PR URL, no reviewer email available to the accepter, because Phase 11 is six phases away. §Open question #9 surfaces the tension and *defends the choice*: "The exit criterion implies the recording happens within the workflow. I argue post-merge is the right time." The synthesizer would have to either (a) accept that Phase 4's exit criterion is met only by an out-of-band manual step that fakes a merge SHA (because there's no real merge), or (b) override the design's policy. The design as written does not meet its own roadmap criterion.
   **Where:** roadmap §Phase 4 exit criterion vs §Component 3 `RagAccepter` + §Data flow step 17 + §Open questions #9.

4. **Problem:** SPKI pinning is mentioned in the lens summary as load-bearing for the egress proxy ("speaks TLS to `api.anthropic.com`"), but the implementation in §Component 5 is silent on pin rotation. Anthropic rotates `api.anthropic.com` certificates without notice (CDN-issued LE certs rotate every ~60 days). A hard SPKI pin breaks the proxy on rotation; a soft pin defeats the threat-model rationale. The design has neither the rotation runbook nor a fallback path to "fail closed on pin mismatch" vs "fail open and notify."
   **Where:** §Trust boundaries diagram (TLS to Anthropic), §Component 5 EgressProxy.

5. **Problem:** Offline `sentence-transformers` model pinning (§Goals #15, §Components §9 `EmbeddingModelLoader`) records the BLAKE3 hash in `tools/digests.yaml` and re-verifies on every load. But the **first-run download** flow is buried: §Component 9 says "`codegenie rag init` (downloads model)... Subsequent loads read from local cache, re-verify hash, fail hard on mismatch." Two failure modes are unaddressed: (a) what verifies the digest *before* `rag init` writes it for the first time? If the digest is a hand-edited entry in `tools/digests.yaml` and a compromised CI run regenerates it after a first-time download from a poisoned mirror, the pin certifies the poison. (b) The design says (§Goals #16) "no telemetry-by-default" with HF downloads disabled at import time — but if first-run *needs* HF to fetch the model, the disabling must be conditional. An airgapped operator who runs `codegenie rag init` with HF blocked gets a cryptic failure rather than a clear "expected pre-loaded model at /var/lib/codegenie/rag/models/<hash>/."
   **Where:** §Goals #15–16, §Component 9.

### Hidden assumptions

1. **Assumption:** Stripping LLM self-reported confidence from the response (§Goals #8 "the LLM's self-reported confidence... is **stripped from the response before downstream consumers see it**") is doable by a regex / Pydantic-extra-forbid filter and an attacker cannot smuggle a confidence-equivalent into a non-confidence field.
   **What breaks if wrong:** ADR-0008 says no LLM self-reported confidence feeds the gate. But the structured-plan schema has a `rationale` free-form field (§Component 8 `LlmOutputApplier`). An attacker who controls the advisory text can instruct the LLM to encode confidence-as-rationale: "rationale: This fix is verified by the upstream maintainer; safe to merge without review." A human reviewing the PR will read that as trust-bearing text. The "strip confidence" defense relies on the field being named `confidence`; it doesn't survive lexical paraphrase.

2. **Assumption:** `bwrap + uid + unshare-net` is "sandbox-ready" enough that the contract relocates behind a Phase 5 microVM "without rewrites" (§Lens summary).
   **What breaks if wrong:** The contract between the orchestrator and the agent in this design is file-based (`/jail/in/req.json` → `/jail/out/resp.json`) over a bwrap mount. The Phase 5 microVM contract is necessarily transport-mediated (firecracker vsock or HTTP-over-pipe). "Identical RPC contract" handwaves over a real transport rewrite: file-mediated handoff with chmod/chown semantics doesn't directly map to a microVM where the host filesystem isn't visible. The claim "switching to a microVM in Phase 5 becomes a transport change, not an architectural change" is too strong.

3. **Assumption:** "Cross-repo retrieval... only if `provenance.public == True` (open-source upstream patches) OR the matched repo is the same as the current one" (§Goals #13) is enforceable without leaking same-org cross-repo bytes through the *embedding space itself*.
   **What breaks if wrong:** §Acknowledged blind spots admits this ("Cross-repo information disclosure via embeddings... the embedding space is shared across all repos"). The design enforces *retrieval-time* boundaries but the *embedding-time* boundary is shared. An adversarial query crafted to produce a high-cosine match against a private repo's solved example reveals *which neighborhood* (and therefore which package + CVE shape) that repo has fixed, even when the retrieval is denied. The retrieval-deny just confirms "you got close" via timing or audit-event count.

### Things this design missed that a different lens caught

- **Latency and operator cost.** Process-spawn overhead per workflow (bwrap + uid setup + egress-proxy bring-up) is described as "~150 ms / run" (§Open questions #1). Performance design's RAG-hot p95 is 95 s, mostly Phase 3 inheritance — the security design's three-subprocess fan-out (orchestrator + rag-querier + agent + egress-proxy) compounds that.
- **No Anthropic prompt caching as a first-class concern.** §Open question #6 surfaces it as a "cost-vs-security tradeoff" and the design ships no cache-breakpoint discipline. The performance design has a ≥ 80 % cache-hit-rate goal worth ~$0.04 per call; the security design pays full price every time.
- **No streaming.** §Acknowledged blind spots calls this out, but it means an LLM completion that's heading toward an unsupported transform burns the full max_tokens before the orchestrator can cancel.
- **No engine-contract integration.** The security design talks about `LlmOutputApplier` and recipe IDs but does not reuse Phase 3's `RecipeEngine` ABC; it bolts a separate "FallbackRouter" on. The best-practices design's "RagLlmEngine as a third RecipeEngine" is structurally cleaner — even if (see next section) it conflates two things.

## Attacks on the best-practices design

### Concrete problems

1. **Problem:** `RagLlmEngine` is registered as a third `RecipeEngine` alongside `NcuRecipeEngine` and `OpenRewriteEngineStub`. The Phase 3 final-design explicitly defines `RecipeEngine.apply(recipe: Recipe, repo: Path, ctx: ApplyContext) -> RecipeApplication(diff: bytes, files_changed, engine_stdout, engine_stderr, exit_code)` — a contract for a **deterministic patch producer**: same `(recipe, repo, ctx)` in, same diff bytes out, exit_code drawn from a small process-shell vocabulary. The `RagLlmEngine` cannot honor this contract:
   - Same inputs produce different outputs (LLM nondeterminism; even temperature=0 is not bit-stable across SDK versions or model patches).
   - `engine.apply(ctx)` for an LLM engine can fail in *qualitatively new* ways: low-confidence-but-syntactically-valid, hallucinated-but-passes-parse, cost-cap-hit-mid-call, prompt-injection-result. `RecipeApplication.exit_code` doesn't model any of these; the design's §Components §1 maps them to "exit codes 5, 9, 11" in the *CLI*, not in the `RecipeApplication`.
   - The Phase 3 `RecipeSelector` "iterates in order; first non-skip reason wins" (§Best-practices §Architecture) — meaning a "matched" returned by `RagLlmEngine` short-circuits any subsequent engine. But `RagLlmEngine.applies()` always returns True (§Data flow step 2c "engine is the fallback; always applies if available"). So the selector's chain-of-responsibility *only works* because the LLM engine is registered last, which is a positional invariant nowhere encoded in the registry — anyone reordering `@register_recipe_engine` calls breaks Phase 4.
   **Why it matters:** The whole pitch of this design is "Phase 4 is one new engine, not a new abstraction." If the engine cannot satisfy the engine contract, the abstraction *is* new — it's just hidden by overloading a name. Phase 7 (distroless migration) will look at this and either (a) follow the precedent and conflate its own deterministic + LLM paths under one engine name, or (b) introduce the right abstraction now and have to ask "why didn't Phase 4 do this?"
   **Where:** §Lens summary first bullet, §Conventions honored "Extension by addition," §Component 1 `RagLlmEngine`, §Component 1 tradeoffs "Engine has no async path," Phase 3 final-design §"Component 2 RecipeEngine ABC."

2. **Problem:** The design declares "**No LangGraph in Phase 4**" (§Lens summary, §What I deprioritize: "LangGraph wrapping. Phase 6's job"). The Phase 4 roadmap entry says verbatim: "**`langgraph` imported minimally — just enough to wrap the leaf agent invocation.**" The best-practices design reads this as "honoured by *importing nothing* — `langgraph` does not appear in Phase 4's `pyproject.toml`." That is not honouring the roadmap; it is overriding it. The roadmap chose a "minimal import" specifically to make Phase 6's migration *not* a from-scratch wrapping. The performance design follows the roadmap; the best-practices design deviates. Phase 6 inherits a clean wrapping target if it lands now; otherwise Phase 6 must invent the wrapping shape and prove it composes with the engine. The "Phase 6 wraps" claim is a promise this design cannot keep on Phase 6's behalf.
   **Where:** §Lens summary "No LangGraph in Phase 4," roadmap §Phase 4 tooling.

3. **Problem:** §Conventions honored §Extension by addition explicitly accepts two edits to Phase 3 committed contracts: ADR-P4-001 (`Recipe.engine` Literal extends from `{ncu,openrewrite}` to `{ncu,openrewrite,rag_llm}`) and ADR-P4-002 (`RemediationOrchestrator` gains a writeback conditional). The CLAUDE.md commitment is "Adding Java, Python, or a new task type must be new probes + new Skills, never edits to existing probes or the coordinator." The design defends both edits as "ADR-gated additive edits," but ADR-gating does not make a contract edit additive. Specifically:
   - `Literal["ncu","openrewrite"]` → `Literal["ncu","openrewrite","rag_llm"]` is a *modification* to a published Pydantic schema. Phase 3's contract-snapshot test (referenced in §Conventions: "protected by Phase 3's contract snapshot test, which regenerates as part of the Phase 4 PR") *regenerates*, which is the design admitting the snapshot test no longer protects what it was supposed to.
   - The `RemediationOrchestrator` gains a new conditional branch. Phase 3 final-design says the orchestrator is six sequential function calls (§501 "Applies it. Stage 5 transform + engine"). Phase 4 adds a seventh conditional call. Phase 5 will add retries; Phase 6 will swap the orchestrator entirely. The "one conditional edit" precedent is the foot in the door.
   **Where:** §Conventions honored "Extension by addition," CLAUDE.md "Extension by addition," Phase 3 §Component 2 ABC contract.

4. **Problem:** Two new top-level packages — `src/codegenie/rag/` and `src/codegenie/llm/` — added on top of Phase 3's `transforms/` and `recipes/`. The design celebrates this as "Smaller new-package footprint than Phase 3's 2; deliberate" (§Goals). Counting packages added so far: Phase 0 lands `cli/`, `probes/`, `coordinator/`, `cache/`; Phase 1 grows them; Phase 2 adds `skills/` and probe-layer modules; Phase 3 adds `transforms/`, `recipes/`; Phase 4 adds `rag/`, `llm/`. The package graph keeps growing one or two roots per phase, and Phase 5 will need `sandbox/`, Phase 6 will need `state_machine/`, Phase 9 will need `temporal/`. The "fewer is better when fewer fits" claim (§Goals) is contradicted by the trajectory. The two packages introduced here are both *runtime-component packages*, not *concern-aligned packages*: `rag/` is "where chromadb lives," `llm/` is "where anthropic lives." Package boundaries should reflect domain concepts, not dep-isolation walls.
   **Where:** §Lens summary "Two new top-level packages," §Goals "New top-level packages: 2."

5. **Problem:** §Component 2 `LeafLlmAgent` + `AnthropicClient` pins the model as `"claude-opus-4-7-20260415"` (the current production model as of 2026-05-12) and treats it as a hard literal: "Pinning is non-negotiable... Bumping the model is an ADR amendment." Every test cassette in `tests/cassettes/` (§Test plan / VCR cassette discipline) records this model name in the request body. When Anthropic ships Claude 5 (which the cassette-freshness script in §Test plan notes: "If the pinned model in `~/.config/codegenie/llm.yaml` changes, a CI script reports every cassette whose recorded model differs — the PR that bumps the model must re-record all cassettes or explain why not"), the entire test corpus is invalid in one PR. The "re-record all cassettes" remediation is a full test-suite re-recording bottlenecked through one engineer who can review every cassette diff (which §Test plan / VCR cassette discipline already requires per-PR review: "`LABEL_REQUIRED: cassettes-reviewed` GitHub label gates merge"). A model upgrade becomes a multi-week PR.
   **Where:** §Component 2 LlmRequest model field, §Test plan / VCR cassette discipline (cassette freshness script).

### Hidden assumptions

1. **Assumption:** Anthropic's prompt-caching API (`cache_control={"type": "ephemeral"}`) is stable across the `anthropic` SDK pin window (§Component 2 "anthropic (~latest 0.x)" — *no version pin*).
   **What breaks if wrong:** §Risks #6 says the SDK is "pinned to a minor; the client's caching-related calls are guarded by an integration test that asserts response carries `cache_creation_input_tokens` and `cache_read_input_tokens` fields." But §New dependencies says `"anthropic" (~latest 0.x)` — no explicit minor pin. The SDK's caching response shape has changed twice in 2025–2026. An unpinned dep + a strict response-schema test means every dependabot bump on the SDK breaks Phase 4 in CI.

2. **Assumption:** The `RagLlmEngine.applies()` always returning True at the bottom of the selector chain is a stable invariant.
   **What breaks if wrong:** Phase 7 (Chainguard distroless) introduces a `DockerfileBaseImageSwapTransform` and "likely OpenRewrite-shaped or Docker-specific engine" per Phase 3's §"Phase 7" forward-look. If Phase 7 introduces an LLM-fallback engine for migration too (which it must, by the same logic that drove Phase 4), there are now *two* engines whose `applies()` returns True at the bottom of the chain, and the selector iterates in order: whichever was registered first wins. The "always-last" positional invariant doesn't survive a second task class.

3. **Assumption:** Phase 13's full Budget Enforcer middleware can subsume Phase 4's `LlmInvocationGuard` ($5.00 per-invocation hard wall) without contract changes.
   **What breaks if wrong:** §Component 6 surfaces this as a deliberate minimum: "Phase 13's full Budget Enforcer subsumes this." But Phase 13 enforces *per-workflow*, *per-task-class*, *tiered direct/amortized/overhead* (per ADR-0027). The Phase 4 guard fires *per invocation* on an estimated cost (chars ÷ 4 × max_tokens × rate). Phase 13's contract needs a hook point *before* a call is made; Phase 4's guard *is* that hook point, but only for the single invocation. A multi-call workflow (Phase 6) will need the same hook taking running totals — but Phase 4 ships the guard with no running-total interface. Phase 13 either rewrites the guard or wraps it; either way, Phase 4's API is not the right shape.

### Things this design missed that a different lens caught

- **Prompt-injection defense is acknowledged-and-deferred** (§Acknowledged blind spots #2 "No prompt-injection defense in the few-shot RAG path"). The security design's canary token + per-run fence-id + structural-plan recipe-ID allowlist are absent. The mitigation listed — "the writeback path only persists patches that passed Phase 3's strict-AND validation gate" — relies on Phase 3's `npm test` to catch *malicious* outputs, which is not what `npm test` does.
- **No agent process isolation.** The leaf agent is "a plain function with a typed input and a typed output" (§Lens summary); the Anthropic SDK runs in the orchestrator's address space. The performance design accepts this for speed; the security design rejects it; the best-practices design doesn't engage with the choice.
- **No defense for the embedding-model first-fetch path** — the security design's BLAKE3 hash verification + airgap-mode is missing. §New dependencies says "`huggingface_hub` (pinned minor): SHA-pinned model download" but the design ships no hash-mismatch failure mode.
- **No cost ledger schema beyond per-run JSONL.** §Conventions honored / Cost observability emits `cost.llm.invoked` events but Phase 13's "three-tier aggregation key from `design.md §3.3`" requires direct/amortized/overhead fields that Phase 4 does not emit. Phase 13 will read incomplete entries and have to backfill.

## Cross-design observations

### Where do the three disagree?

| Dimension | Performance picks | Security picks | Best-practices picks | What's at stake |
|---|---|---|---|---|
| Agent process boundary | In-process, no isolation | bwrap + uid jail + egress proxy + RAG-querier subprocess | In-process plain function | Whether prompt injection escapes the agent process. Performance: assumes Phase 5 fixes it. Security: pre-pays Phase 5's microVM contract. Best-practices: ignores. |
| Solved-example writeback timing | Async fire-and-forget at Phase 3 validation pass | Operator-gated `codegenie rag accept` after human merge | Sync at Phase 3 validation pass, no human gate | Whether the corpus is a record of *what passed validation* (performance, BP) or *what humans merged* (security, ADR-0009). The roadmap exit criterion lives in this disagreement. |
| Embedding model | `all-MiniLM-L6-v2` (384-d) | `all-MiniLM-L6-v2` (384-d, hash-pinned) | `BAAI/bge-small-en-v1.5` (384-d, snapshot-pinned) | Two designs share a default that one design rejects; the rejection rationale (better retrieval quality) is not benchmarked. |

### Which disagreement matters most for *this* phase?

The **writeback timing** disagreement matters most. The roadmap exit criterion is "A breaking-change vuln is solved end-to-end with the LLM fallback and recorded into the solved-example store. Re-running the same case hits RAG, not LLM." The performance design writes back on Phase 3 validation pass (no human in the loop). The best-practices design writes back on Phase 3 validation pass plus engine-type check (no human in the loop). The security design writes back **only** on operator-invoked `codegenie rag accept` (gated on human merge of a real PR). Phase 4 ships locally, where there is no PR to merge. The security design therefore does not — by its own machinery — meet the roadmap exit criterion in Phase 4. The performance and best-practices designs meet the criterion by writing back un-merged LLM output into the corpus that gates every future workflow, which Phase 11 (which introduces real PRs and human merge) will then have to retrofit. Either we accept that Phase 4's writeback is structurally different from Phase 11's writeback (and document the migration), or Phase 4's "compounding savings" story is built on an un-audited corpus. The synthesizer must explicitly pick which way the criterion is met and document the Phase 11 migration plan.

### Where do all three quietly agree on something questionable?

1. **VCR cassettes (`pytest-recording`) as sufficient for deterministic CI.** All three designs commit to cassette-based testing for every test that touches Anthropic. VCR is *byte-replay*, not semantic-replay: the cassette is keyed off the exact request body bytes, which include the model name, system prompt, few-shot text, user prompt, and `cache_control` markers. Any prompt-template edit — including changes that are *functionally identical* (whitespace, key reordering inside the system block, additional Skills loaded into the manifest) — invalidates every cassette that hit that path. All three designs accept that a cassette miss in CI is a hard fail (performance: `VCR_BAN_NEW_CASSETTES=1`; security: `--record-mode=none`; best-practices: `--record-mode=none` + `LABEL_REQUIRED: cassettes-reviewed`). In practice this means every PR that touches a prompt requires re-recording every affected cassette, which requires a live API key, which requires a single engineer with access. The "deterministic, free CI" promise rests on a workflow that creates a bottleneck PR by PR.

2. **All three treat solved-example writeback as a single decision point with no Phase 11 migration plan.** Performance writes back fire-and-forget at validation pass. Best-practices writes back sync at validation pass when `engine_used == "rag_llm"`. Security writes back only on operator `rag accept` after merge — but Phase 4 has no merge. None of the three designs include a section titled "How Phase 11's real-merge writeback policy composes with this Phase 4 local writeback." The performance design's §Open question #3 ("Should `SolvedExampleRecorder` write be made durable... earlier?") is the closest, and it punts to Phase 9 (Temporal), skipping the Phase 11 question entirely. The corpus accumulated in Phase 4 will be present when Phase 11 ships; either Phase 11 trusts it (in which case the human-merge gate is theatrical), or Phase 11 wipes it (in which case Phase 4's "compounding savings" exit criterion was a local artifact).

3. **All three feed `RepoContext` (or substantial slices of it) into the LLM prompt with no exfiltration defense beyond egress byte caps.** `RepoContext` contains the full file inventory, dep graph, traces (which may include filenames, file content excerpts), policy violations, and Skills text. The performance design (§Component 3 "Query block: the advisory + repo-context slice") and the best-practices design (§Data flow step 4 "user message (per-run RepoContext slice + advisory)") send a "slice" of `RepoContext` to Anthropic with no schema-pinned definition of what's in the slice. The security design (§Component 4) caps the prompt at 256 KB and ships it via the egress proxy, which byte-caps egress at 5 MB/run — but a 256 KB prompt *is* the exfiltration channel: an injected LLM rewrites a fix to embed the secret in the diff (`// CONFIG_VALUE=<exfiltrated>`), the diff gets to `git apply --check`, and the bytes are now in the working tree's audit log. None of the three designs strips secrets from `RepoContext` before prompt construction. Phase 2's Skills loader includes the org's conventions catalog and exception registries — content that should not leave the host. The "RepoContext slice" is the missing schema in all three designs.

## Roadmap-level critiques

1. **Phase 5 / 6 / 7 / 11 / 13 / 15 composition:**
   - **Phase 5 (microVM + Trust-Aware gates).** Performance has no agent isolation; Phase 5 must retrofit. Security has bwrap + uid jail with a file-IO contract; Phase 5 swaps to firecracker but the file-IO assumption travels poorly to a microVM that doesn't share the host FS. Best-practices has no isolation; Phase 5 must retrofit. **Security has the cleanest migration only if the file-IO contract is replaced with a vsock contract — which is more than a "transport change."**
   - **Phase 6 (LangGraph state machine for the vuln loop).** Performance imports LangGraph minimally (one-node `StateGraph`); Phase 6 wraps cleanly. Security does not import LangGraph at all; Phase 6 must invent the wrapping. Best-practices explicitly refuses LangGraph; Phase 6 must invent the wrapping *and* override this design's stated policy. **Performance has the cleanest Phase 6 migration; best-practices the worst.**
   - **Phase 7 (distroless migration as second task class).** All three designs ship `RagLlmEngine`-shaped (or `ManualPatchEngine`-shaped) engines without considering how a second task class's LLM-fallback engine composes with the first. Best-practices' "always-last positional invariant" breaks first (two engines whose `applies()` returns True). Security's `FallbackRouter` is task-class agnostic by accident — it works for migration too, but adds a second `--allow-*` flag.
   - **Phase 11 (real PR opening + human merge).** Discussed above — writeback policy divergence is the load-bearing failure across all three designs.
   - **Phase 13 (AgentOps cost ledger).** Phase 4 introduces cost ceiling enforcement first. Performance's `--max-llm-cost-usd=0.50` per-workflow cap with token-prediction enforcement is the closest to Phase 13's per-workflow Budget Enforcer (ADR-0025). Security's per-workflow token-count cap + egress byte cap in the proxy is structurally different — Phase 13 cannot consume "bytes egressed" as a primary cost signal. Best-practices' `LlmInvocationGuard` is per-invocation with no running-total interface and is the worst fit. **None of the three designs ship the `cost-ledger.jsonl` entry schema in a shape Phase 13 can consume without a migration.**
   - **Phase 15 (agentic recipe authoring from solved examples).** The Phase 4 solved-example store is the seed corpus for Phase 15. None of the three schemas explicitly capture: (a) *which Phase 3 recipe failure mode* triggered the LLM fallback (best-practices' `recipe_failure_reason` field is the closest); (b) *which solved-example was used as few-shot* (so Phase 15 can detect example clusters that consistently solve the same problem shape — the trigger for recipe authoring); (c) *the engine_used trajectory* (Ncu attempted → failed → OpenRewrite attempted → failed → RAG hit → LLM with few-shot → succeeded). Without (a)+(b)+(c) as first-class metadata, Phase 15 cannot cluster solved examples for recipe-promotion candidacy without re-walking the audit log. **Best-practices' schema is the most complete; performance's is the most under-specified; security's `provenance` dict is opaque.**

2. **Reliance on earlier phases that didn't establish what's needed:**
   - Phase 2's IndexHealthProbe (B2) is the precedent for the best-practices design's `SolvedExampleHealthProbe`. The B2 contract is fine; the analog is reasonable. **However**, B2 is a Phase 2 layer-B probe whose freshness signal feeds Phase 3 gates (per Phase 3 §54 "lockfile.parse_ok, lockfile.policy_violation_count == 0, ... → confidence"). The `SolvedExampleHealthProbe` would feed *Phase 5* gates per the best-practices design ("This probe's `confidence` will eventually be a Phase 5 gate input"), but Phase 4 surfaces it without gating on it. So Phase 4 ships a probe that is dead weight in Phase 4 and only earns its place in Phase 5. The Phase 1 probe-contract precedent isn't broken, but Phase 4 is adding probe machinery for a future gate.
   - Phase 2's BLAKE3-chained JSONL audit log is presupposed by all three designs (security explicitly extends; performance and best-practices implicitly extend). Phase 2 final-design must already define the chain's event-type extensibility model — if it does not, this is a Phase 2 → Phase 4 dependency that has to be back-filled.
   - Phase 3's `RecipeSelection.reason` enum is the chain-of-responsibility pivot. All three designs read it. Phase 3's final-design defines the enum as `Literal["matched","no_engine","range_break","peer_dep_conflict","unsupported_dialect","catalog_miss"]` (verified, §30 of Phase 3 final-design). Phase 4 designs use this enum without extending it. Good — but if any of the three designs *needed* a new reason (e.g., "llm_required_but_disabled"), it would require a Phase 3 contract amendment. None of the three designs explicitly check this, which is a latent fragility.

3. **Load-bearing commitment violations from production/design.md §2 and CLAUDE.md:**
   - **§2.1 "No LLM anywhere in the gather pipeline."** All three designs respect this — the LLM lives in the planner/transform path, not the gather path.
   - **§2.2 "Facts, not judgments."** Performance writes negative examples with `mismatch_cluster_id` — facts-shaped. Security strips LLM self-confidence — facts-shaped. Best-practices logs LLM self-confidence under `cost-report.yaml#diagnostics` but excludes from gates — facts-shaped, but the diagnostic is *one rename away* from being read as a judgment.
   - **§2.3 "Honest confidence" / ADR-0008.** All three respect strict-AND objective signals at the gate. Performance accepts a 3-fail empirical neighborhood-threshold raise (§Risks #1) — this is a *learned* confidence, which is on the borderline of ADR-0008 ("no LLM self-confidence" — but ADR-0008 does not forbid system-learned confidence calibration; the design is in the gray zone).
   - **§2.4 "Determinism over probabilism for structural changes."** Best-practices' "the LLM produces a candidate patch; Phase 3's machinery verifies it deterministically" is the cleanest articulation. Performance's `ManualPatchEngine` and security's `LlmOutputApplier` are structurally similar. All three honor the commitment in spirit; the disagreement is whether the LLM-engine contract should *be* the deterministic-engine contract (best-practices: yes; the others: no).
   - **§2.5 "Extension by addition."** Best-practices admits two coordinator edits (ADR-P4-001 and -002). CLAUDE.md says "never edits to existing probes or the coordinator." The defense ("ADR-gated") is operationally honest but contractually a violation. Performance adds `ManualPatchEngine` as a new `RecipeEngine` sibling and wraps the Phase 3 coordinator (§Component 7 "Phase 3 coordinator stays unchanged. We add a new orchestrator entrypoint `remediate_v2` (Phase 4) that *wraps* Phase 3's coordinator") — this is the cleanest fit to §2.5. Security adds a "Stage 3.5 FallbackRouter" between Phase 3 stages — that's an edit in the orchestrator's call graph, which is an edit to the coordinator's shape regardless of where the file lives.
   - **§2.7 "Progressive disclosure for context."** All three serialize prompts/responses to disk and reference by path in the report — good. The concern: performance's "embedding cache for the query string (avoids re-embedding repeat advisories)" lives in `.codegenie/cache/planner/embeddings/` outside the per-run dir, which is *index* not *evidence* and doesn't violate the principle.
   - **§2.8 "Humans always merge" / ADR-0009.** Performance and best-practices write back to the RAG store *before any human merge could possibly happen* (Phase 4 has no PR). This is the most concrete §2/ADR violation in the document set. Security gates on `rag accept` after merge, which honors ADR-0009 but at the cost of not meeting the roadmap exit criterion in Phase 4. The synthesizer cannot satisfy ADR-0009 *and* the roadmap exit criterion *and* Phase 4's local-only scope without making a structural choice none of the three designs makes cleanly.
