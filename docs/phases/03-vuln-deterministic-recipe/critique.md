# Phase 3 — Vuln remediation: deterministic recipe path: Devil's-advocate critique

**Reviewed by:** Devil's-advocate critic subagent
**Date:** 2026-05-15

## Method

I read all three designs end-to-end, plus the roadmap Phase 3 spec, `docs/production/design.md` §2 (load-bearing commitments), ADRs 0029–0034 (+0011, 0008, 0014, 0009, 0012, 0028), and the final-designs of Phases 0, 1, 2, 5, and 6.5. ## Attacks on the performance-first design

### Concrete problems

1. **Problem:** The design ships an **in-process `PluginRegistry` as a module-level mutable singleton** (`_REGISTRY: dict[PluginScope, PluginManifest] = {}` populated by `@register_plugin` at import time) and explicitly forbids per-workflow reconstruction. That is import-time side effects in the very place the project commits to **no side effects in constructors / module import time** (design-patterns-toolkit §"Side effects in constructors / module import time"). Worse, the registry is **per-process global mutable state** — every test that imports a plugin module dirties it for every subsequent test in the same interpreter.
   **Why it matters:** Phase 6.5's `TaskClassRegistry` (see `docs/phases/06.5-per-task-class-eval-harness/final-design.md`) already had to wrestle with registry-pollution-across-tests and adopted a registry instance with fresh-fixture-per-test discipline. The performance design will repeat that pain with no acknowledgement. The "no reload, restart the worker" line is a hand-wave: pytest does not restart the worker between tests.
   **Where:** `§Components / 1. PluginRegistry — process-singleton` and the boast "Mutability is gone — adding a plugin at runtime requires a process restart" (last line of the singleton component).

2. **Problem:** **OpenRewrite is unilaterally relegated to "Phase 7 deliverable."** The roadmap names OpenRewrite *first* and `npm-check-updates` "as a simpler first cut." The design's defense — "JVM is a 4-second tax for a 60-line jq" — is *only* true if (a) the npm CVE surface is, in fact, 95% lockfile resolution (asserted, never proven) and (b) the JVM never ships before Phase 7. **Phase 5 is *already designed* to consume `RecipeApplication` shapes from Phase 3/4**; if Phase 3 ships a pure-Python recipe path that the Phase 5 design (which uses a YAML gate catalog and a `gate_isolation_class` annotation) presumes is reusable, then re-introducing OpenRewrite in Phase 7 forces *two* recipe-engine semantics to coexist *retroactively*. Phase 7's exit criterion is "zero edits to existing files." A new OpenRewrite engine that needs to share infrastructure with a pure-Python one is an edit.
   **Why it matters:** This is a roadmap-vs-design conflict; the design admits it (`§Architecture` line "This contradicts the roadmap's wording") and discharges it by promising the bench will pick. The synthesizer cannot accept "bench picks later" as the answer — Phase 7 isn't bench-able from Phase 3.
   **Where:** `§Architecture` paragraph 1; `§Components / 5. NpmRecipeEngine` paragraph "Why not OpenRewrite"; `§Open questions #1`.

3. **Problem:** The **Bundle Builder hedges fallback queries** (`asyncio.Semaphore(4)` + "TCCM-declared `fallback` runs concurrently with the primary; first high-confidence wins"). This is **non-determinism leaking into the recipe-decision path**. The roadmap commits to "Determinism guarantee: running the same plugin against the same repo at the same content-addressed cache state produces the same diff and the same event sequence." If two adapters race, the *order of completion* depends on scheduler vagaries — and `IndexHealthProbe`'s confidence reading might race with the primary's confidence reading. The design's own test #11 (`Bundle cache hits are content-addressed: running the same workflow twice produces identical Bundle digests`) only passes because the *cache* is checked first; on cold cache, hedging means the bundle is whichever path returned first.
   **Why it matters:** ADR-0008 (objective-signal trust score) + commitment §2.4 (determinism over probabilism for structural changes) both demand byte-identical bundles for byte-identical inputs. Hedging trades determinism for tail-latency; that trade is not the perf lens's to make unilaterally.
   **Where:** `§Components / 2. BundleBuilder` bullet "Confidence-driven fallback" + `§Risks #4`.

4. **Problem:** The **`asyncio.Semaphore(4)` bound is justified by "empirical SSD knee" without an actual benchmark** — the design says "Four is the knee" with no fixture, no measurement, no run. The same paragraph then says `min(4, os.cpu_count())` as the default. On a Phase 0 GitHub-hosted runner with `cpu_count()=2`, the bound collapses to 2 — which the Phase 2 critique (`docs/phases/02-context-gather-layers-b-g/final-design.md` line 343, "GitHub-hosted runners have `cpu_count()=2` where per-tier sizing degenerates") already identified as a starvation hazard for the existing single-semaphore design. The performance design imports the same problem.
   **Why it matters:** The constant is the load-bearing performance argument, but the value is invented. If the synthesizer accepts the design and CI shows `bench_bundle_builder_cold > 250ms`, the explanation will be "the SSD knee doesn't exist on `ubuntu-latest`" — and the design has no fallback rationale.
   **Where:** `§Components / 2. BundleBuilder` bullet "Parallelism"; `§Open questions #5`.

5. **Problem:** The design **skips LangGraph entirely in Phase 3** (`§Architecture` line 2: "No LangGraph subgraph in Phase 3") and ships "the subgraph topology as a sequence of typed step functions that Phase 6 wraps in LangGraph nodes 1-to-1." Phase 5's final design *already* assumes Phase 4's `FallbackTier.run` is structured around `RecipeApplication` objects with a `prior_attempts` kwarg, *and* assumes a `RemediationOrchestrator` exists that Phase 5 wraps for retry. **There is no `RemediationOrchestrator` in the performance design** — there is a plain async function chain. Phase 5's retry-feedback loop (which re-enters Phase 4 with `prior_attempts: list[AttemptSummary]`) needs an orchestrator to be the call-site that passes them. The performance design's "Phase 6 wraps the step functions" is a *Phase 6* statement, but Phase 5 — which is *already designed* — needs Phase 3's orchestrator now.
   **Why it matters:** Phase 5's `tests/integration/gates/test_stage6_retry_recovers.py` is the exit-criterion gate. It calls into "Phase 3's `RemediationOrchestrator`." If Phase 3 doesn't ship one, Phase 5's exit criterion is unmeetable. The Phase 6 LangGraph wrapper is too late.
   **Where:** `§Architecture` paragraph 2; the diagram label "in-process, no LangGraph in Phase 3."

6. **Problem:** The design's **`UniversalFallbackPlugin` is placed at `plugins/universal--*--*/`** — literal `*` in the directory name. The design's own `§Components / 7` says "Resolution to this plugin is the same `dict.get` as any other; the fallback path costs no more than the happy path. Crucially, the resolver does **not** do 'did any plugin match? if not, fallback'." But every other concrete plugin must match before this one wins on `precedence: 0`, so the lookup is *not* a single `dict.get` — it's a sort + first-match walk. The performance claim ("O(1) even on cache-miss tuples: there is always a match") is true only if the resolver is a precomputed scope-tuple → plugin map, which the design says it is — but that map cannot have entries for arbitrary `(task, lang, bt)` triples; it must be a wildcard-fallback walk. The design contradicts itself: "pre-computed matcher (built once at boot — O(plugins) walk amortized)" — that's not `O(1)`; that's `O(plugins)` per workflow.
   **Why it matters:** The performance claim is wrong by the design's own description, and the contradiction is in adjacent paragraphs. A synthesizer pulling "fallback is `dict.get`" forward will discover at integration time that resolution is `O(plugins)`.
   **Where:** `§Components / 1. PluginRegistry` internal design vs `§Components / 7. UniversalFallbackPlugin`.

7. **Problem:** The **Phase-3 event-log format (`.codegenie/events/<workflow_id>.jsonl.zst`) is asserted to be Phase-9-projectable, but Phase 9 (per ADR-0034) lands a Postgres event log "for workflow-spanning concerns" while Temporal owns workflow-internal events.** The design ships per-workflow JSONL files containing what would be Temporal workflow-internal events (e.g., `PluginResolved`, `RecipeApplyOutcome`, `WorkflowEnded`). When Phase 9 ships, those events are owned by Temporal's history; the Postgres side-channel is for *workflow-spanning* events (cost ledger, KG writes). The design's JSONL is structurally aligned with the wrong substrate.
   **Why it matters:** "Phase 9's projector reads our files unchanged" is a false promise. Phase 9 will read Temporal history for workflow-internal events and Postgres for portfolio events. The Phase-3 JSONL is neither.
   **Where:** `§Components / 6. EventAppender` bullet "Why JSONL+zstd, not Postgres / Redpanda / Kafka"; `§Risks` discussion.

### Hidden assumptions

1. **Assumption:** That **`npm install --package-lock-only --ignore-scripts --prefer-offline` actually works deterministically and fully without `node_modules` present.** The design's data flow line `T+346/526` claims 5–6 s for this step, hardlink-copying only `package.json` and `package-lock.json`.
   **What breaks if it's wrong:** npm's `--package-lock-only` does in fact require resolving the full dep tree; if any transitive dep has a peer-dep constraint that requires inspecting the real `node_modules`, npm will produce a *different* lockfile shape than a developer running `npm install` normally — and reviewers will reject the PR as "noisy diff." The fixture (`fixtures/vuln-repos/express-cve-2024-21501`) is one repo; the design assumes its 1.2 GB `node_modules` is unrepresentative on the deterministic path. It probably is, for most CVEs; but the *exit criterion* says "passes the repo's own tests" — and tests run against real `node_modules`.

2. **Assumption:** That **`mmap` of the `.scip` blob is safe under asyncio.** The design admits this ("Mitigation: SCIP file is small (≤ 10 MB); pages-touched ≪ total") but the "if we later index a 500 MB SCIP, we move the mmap into a `to_thread`" hand-wave doesn't compose with Phase 8's mention of 5k-file repos with much larger SCIP indexes.
   **What breaks if it's wrong:** Slow page faults stall the event loop, which silently *also* stalls the Bundle Builder's hedge timer — a 200 ms stall converts confidence-driven hedging into "primary always wins because the hedge couldn't even start." That's a silent correctness regression masquerading as a perf win.

3. **Assumption:** That **content-addressed cache key `blake3(plugin_id || plugin_version || primitive || args_hash || repo_ctx_digest || scip_digest)` covers all reproducibility-affecting inputs.** It does not include the `vuln-index.sqlite` digest (the design admits this in `§Open questions #7`). A CVE feed refresh that re-classifies a CVE produces a different *correct* answer but the same cache key — the bundle returns the stale answer.
   **What breaks if it's wrong:** "The remediation may still be correct" (the design's own line) is exactly the kind of fail-silently posture commitment §2.3 (Honest confidence) and Rule 12 (Fail loud) reject. A stale cache hit on a re-classified CVE produces a PR with the *wrong* patch and no warning.

### Things this design missed that a different lens caught

- **The security design caught: subprocess invocation of `npm install` is the central trust boundary** and needs `bwrap`/`sandbox-exec` jails, env stripping, and `--ignore-scripts` enforcement at both flag and env-var level (because npm has historically had bugs where one or the other was honored). The performance design says "`--ignore-scripts` is non-negotiable (security boundary, but also performance)" and stops there — no env-allowlist, no `bwrap`, no jail. A malicious dep's `postinstall` script running on the orchestrator host is a credential-leak vector the performance design just trusts the npm CLI to enforce.
- **The best-practices design caught: the plugin contract must be *testable against a second plugin* in Phase 7 with zero kernel edits.** The performance design ships `RecipeEngine` Protocol with one implementation (NpmRecipeEngine) and one future implementation (JVM, deferred). It does NOT ship a synthetic third plugin in tests to exercise the contract. Phase 7's "zero edits" requirement will discover at integration time that the engine Protocol has subtle assumptions baked in (e.g., `apply(repo: Repo, plan: RecipePlan) -> Diff` — `Repo` and `RecipePlan` are not defined, but they'll have JSON-shaped assumptions baked into `NpmRecipeEngine`).
- **The security design caught: `package.json` parse needs size/depth caps.** The performance design parses `package.json` with `orjson` and `npm install --package-lock-only` and trusts the files. A 32 MiB malicious `package.json` (or a `package-lock.json` with a 24-level nested dependency tree) is a DOS vector the security lens guards against; the performance lens just lets it through with "we trust npm."

## Attacks on the security-first design

### Concrete problems

1. **Problem:** The design **stops at `RequiresSandboxValidation` and explicitly defers `npm test` to Phase 5** (`§Lens summary`: "the runtime validation of the patched repo — `npm test`, `node` actually executing — is deferred to Phase 5's microVM"). **But the roadmap Phase 3 exit criterion reads, verbatim, "the system writes a working patch diff on a local branch that — when applied — installs cleanly *and passes the repo's own tests*."** The security design knows this is an exit-criterion conflict (`§Acknowledged blind spots`: "the biggest scope choice... my reading: 'the patch is provably installable and Phase 5 will run the tests'"). That reading is **wrong**. The roadmap says "passes the repo's own tests" — there is no Phase-5-handoff ambiguity in the wording. Phase 5's exit criterion (per `docs/phases/05-sandbox-trust-gates/final-design.md`) is the *retry-loop* gate, not the *first* gate. Phase 3 must run tests; Phase 5 wraps the retry.
   **Why it matters:** A design that doesn't meet the exit criterion is a design that fails on contact with the synthesizer. The security lens's defense ("any test execution runs the repo's code, and the repo's code can include malicious test files") is real, but the resolution is not "skip the criterion" — it's "run tests inside a weaker sandbox that Phase 3 designs *now*." Note: Phase 5's `final-design.md` line 53 calls out "Stage 6: Validate → wrapped by Phase 5" — meaning Phase 3 ships Stage 6 (validate, including tests) and Phase 5 wraps it with the retry envelope.
   **Where:** `§Lens summary` paragraph 1; `§Components / 3.4 SubprocessJail` no `npm test` mention; `§Data flow step 11`.

2. **Problem:** **The macOS-only "offline mode" via `codegenie cve-prefetch` introduces an unjailed network step *outside* the install jail**, then has the jailed npm read from a cache that step populated. This is exactly the trust-boundary inversion the lens claims to forbid: TB-D ("registry response is parsed as JSON with the same size/depth/charset caps as CVE feeds") sits *outside* TB-C (the install jail), and the bytes that crossed TB-D — the dependency-closure tarballs — are now writable inputs to TB-C with no further verification beyond "SHA-512 against the registry metadata response," which the design itself notes is circular ("the SHA-512 validation is circular if the metadata is also compromised").
   **Why it matters:** The design's own Risk #1 acknowledges this. But the design ships it as the *default macOS path*. macOS is named-in-spec ("Phase 3 is single-repo, local, CLI-invoked" and developer-loop is macOS-heavy). The security lens has chosen a default that bypasses its own primary defense.
   **Where:** `§Components / 3.4 SubprocessJail` paragraph "macOS offline mirror flow"; `§Risks #1`.

3. **Problem:** **The Capability mechanism is enforced by "a custom flake8/ruff rule fails CI if any module other than `codegenie.cap` imports a Capability constructor directly,"** and the design says "Pydantic's `model_validate` raises `CapabilityForgeryAttempt` if called outside `CapabilityMint`." Detecting whether `model_validate` is called from `CapabilityMint` requires runtime stack inspection (Pydantic itself doesn't know the caller). The mechanism is asserted but not designed. The design also says capabilities have "private `__init__`" — but Pydantic BaseModels don't have private constructors in the conventional sense; `BaseModel.__init__` is what `model_validate` reaches.
   **Why it matters:** The capability pattern is the design's most aggressive type-system claim ("mypy `--strict` plus a custom flake8/ruff rule fails CI if any module..."). It's presented as a load-bearing defense. In practice the runtime check would require an inspect-based mechanism the design doesn't describe; the static lint is the entire enforcement, and lints can be `# noqa`'d, run on the wrong file set, or simply omitted from `pre-commit`. The "capability cannot be forged" claim does not survive scrutiny.
   **Where:** `§Components / 3.5 codegenie.cap` paragraph "Forgeability defenses."

4. **Problem:** **The JVM containment uses `Java SecurityManager` — which is removal-flagged in the JVM roadmap** (the design admits this: "SecurityManager is deprecated upstream; we accept a future 'upgrade the JVM containment' debt"). It then layers `bwrap`/`sandbox-exec` *outside* the JVM as the real defense. So the SecurityManager is "defense in depth inside the JVM" — but the same paragraph then says "the bwrap layer is the real defense; SecurityManager is defense-in-depth." If `bwrap` is the real defense, why ship SecurityManager at all? The configuration cost (custom `.policy` file, `-Djava.security.manager=default`, plus `--add-opens` allowlists) is non-trivial code that the design itself admits is dead-on-arrival in Java 25+.
   **Why it matters:** This is YAGNI dressed as defense-in-depth. The design is paying the cost of two defenses and counting on the second one. Risk #3 says "if we cannot get a working policy at all, fall back to pure-Python recipes only and treat OpenRewrite as Phase 5-or-later territory" — at which point the JVM containment is unused. The synthesizer should reject the deprecated-API ceremony.
   **Where:** `§Components / 3.6 OpenRewriteRunner` "JVM containment" + `§Risks #3`.

5. **Problem:** **`SandboxedPath.create` requires `Path.resolve(strict=True)`** which **follows symlinks** — and the design then says "Symlinks created mid-operation (TOCTOU): every write goes through `os.open(O_NOFOLLOW)`." But `resolve(strict=True)` *resolves through* symlinks (it's how it gets the canonical path). If an attacker creates a symlink between `resolve` and `open`, **the resolve already passed** — the `O_NOFOLLOW` open will fail, but the design says `SandboxedPath` is already minted at that point. So you have a constructed `SandboxedPath` whose `self.absolute` claims to be in-jail (because resolve thought so) but the actual file behind it is unreachable via `O_NOFOLLOW`. The semantics of "you have a `SandboxedPath` ⇒ the path is safe" don't survive TOCTOU: every consumer must re-handle the `ELOOP` exception, which makes the type-discipline argument moot.
   **Why it matters:** The design pitches `SandboxedPath` as a smart-constructor that makes the unsafe state unrepresentable. Under symlink TOCTOU it actually represents a path that *was* in-jail at validation time but isn't safe to open now. That's not "unrepresentable"; that's "deferred re-check."
   **Where:** `§Components / 3.3 SandboxedPath` adversarial inputs.

6. **Problem:** **The plugin loader's "snapshot `sys.modules`, import plugin, diff snapshot" mechanism rejects any plugin that mutates `sys.modules` outside its own namespace.** Plugins import standard library modules (`json`, `pathlib`, `typing`), Pydantic, `nodesemver`, etc. Every one of those imports adds entries to `sys.modules`. The design says "any plugin that mutates `sys.modules` outside its own namespace fails the load" — but `import json` *is* a mutation of `sys.modules` outside the plugin's namespace. The mechanism as described would reject every real plugin.
   **Why it matters:** If "outside its own namespace" means "outside `plugins.vulnerability_remediation_node_npm.*`" then the rule is unimplementable; if it means something narrower (like "no new top-level packages registered by this plugin"), the design doesn't say so. The defense is asserted but not engineered.
   **Where:** `§Components / 3.2 Resolver` paragraph "Isolation."

7. **Problem:** **The plugin signature mechanism is `plugins/PLUGINS.lock` — a checked-in file mapping `{plugin_id: sha256_of_dir_tree}` — and the design's Risk #2 admits "a PR that updates both the plugin and the lockfile passes the runtime check; the trust is in the review process, not in cryptography."** This is **`PACKAGES.lock` cosplay**: it looks like a supply-chain signing mechanism but is just a checksum that travels with the artifact it checksums. Any compromised committer can update both. The design names "CODEOWNERS on `plugins/PLUGINS.lock`" as mitigation but CODEOWNERS is enforced by GitHub, not by the runtime — a malicious push to a branch the CI runs against bypasses CODEOWNERS entirely.
   **Why it matters:** The whole "tamper-evident plugin registry" claim collapses to "a second reviewer is required." The synthesizer should treat this as no signing rather than weak signing.
   **Where:** `§Components / 3.2 Resolver` "Adversarial inputs neutralized — Signature check" + `§Risks #2`.

### Hidden assumptions

1. **Assumption:** That **Phase 3's threat model includes a malicious upstream CVE feed** (the design lists this as Adversary #1).
   **What breaks if it's wrong:** If the threat model is "Phase 3 runs on a developer laptop against a trusted internal feed," every byte of `CveAdvisory.parse(...)` size/depth/charset caps is dead code. The defense-in-depth budget is huge — `§Resource & cost profile` admits 15–70 s total wall-clock — much of which is the prefetch + jail boot. The performance lens (8 s p50) would call this 5–10× overpaid for a threat that may not apply at Phase 3 scale. The CVE-feed-supply-chain risk is a real threat, but Phase 3 is single-repo local and the operator can keep the feed fresh themselves; the design assumed the threat without arguing it.

2. **Assumption:** That **deferring `npm test` to Phase 5 is a defensible reading of the exit criterion.** (Design names this as the "biggest scope choice.")
   **What breaks if it's wrong:** The whole architecture is built around `RequiresSandboxValidation` as the terminal event. If the synthesizer reads the criterion as "Phase 3 runs tests in *some* sandbox," the design has to grow a Phase-3-local `npm test` jail (different from the `npm install` jail), with its own threat model and its own latency budget. The design has no test-running jail at all.

3. **Assumption:** That **macOS sandbox-exec will remain a viable substrate.** The design itself admits ("Risk #4: sandbox-exec works but uses an undocumented profile language Apple has signaled may go away").
   **What breaks if it's wrong:** macOS dev loop becomes "no install jail at all" — and the Linux/CI path becomes the only secure one. The design then has no developer-loop story.

### Things this design missed

- **The performance design caught: hot-vs-cold-cache ratio is ~1.05× on a workflow whose floor is npm.** The security design's 15–70 s total range is dominated by the prefetch + jail-boot tax, not by npm itself. The lens never asks "is the security cost per workflow worth it relative to *what specific attacks*?" Phase 3 is single-repo, local, developer-laptop. The threat model lists "compromised CVE feed" as Adversary #1 — but at Phase 3, the feed has to be manually `codegenie cve-feed sync`'d, not pulled inline. Many of the defenses are over-budgeted for the actual phase.
- **The best-practices design caught: Phase 6.5 backfill compatibility.** The security design's event log is BLAKE3-chained Postgres-event-log-precursor JSONL. Phase 6.5's bench backfill (`tests/e2e/plugins/test_replayable.py` in best-practices) reads the *same* JSONL stream to lift cases. The security design does not mention Phase 6.5 — it mentions Phase 9. If Phase 6.5's projector needs raw bytes (e.g., the patch diff), the security design's "stdout written to `.codegenie/runs/<run_id>/<step>/stdout.bin` (0600), not inlined" makes that backfill harder.
- **The best-practices design caught: a "synthetic third test plugin" exercises the plugin contract from two-and-a-half angles.** The security design ships only the concrete plugin + universal fallback. If the plugin contract is wrong (the most expensive thing to discover late), Phase 7 will be the discovery moment. The synthetic test plugin would have caught it in Phase 3.

## Attacks on the best-practices design

### Concrete problems

1. **Problem:** **The design picks `npm-check-updates` (`ncu`) as the canonical recipe, defends it as "boring, widely-used, 1M+ weekly downloads."** But `ncu`'s job is to *suggest version bumps* — it queries the registry to discover available upgrades. Phase 3 already knows the target version from the CVE feed (`CveAdvisory.fixed_versions`). Invoking `ncu --upgrade --target=patch --filter <pkg>` is asking `ncu` to figure out something Phase 3 already knows; worse, it requires the subprocess to **talk to the npm registry** to enumerate available versions, which the performance design avoids and the security design jails. The "boring tool" argument cuts both ways: `ncu`'s contract is "what versions are available?" — that's not Phase 3's question.
   **Why it matters:** All three designs converged on `ncu` or pure-Python lockfile edits, but the best-practices design specifically rejects pure-Python ("a hand-rolled equivalent would be ~800 LOC of error-prone semver math") and rejects OpenRewrite. The convergence is a *blind spot*: nobody asked "do we even need an external dep-bump tool when we know the target version?" A direct edit of `package.json` + `npm install --package-lock-only` is what the performance design picked, and it's the same answer with one fewer external dep.
   **Where:** `§Components / Vulnerability remediation plugin` paragraph "Boring tool first" + `§Open questions #2`.

2. **Problem:** **The design ships event types that include `PluginsLoaded`, `PluginResolved`, `BundleBuilt`, `BundleEntryPromoted`, `RecipeMatched`, `RecipeApplied`, `RecipeSkipped`, `RecipeFailed`, `InstallSucceeded`, `InstallFailed`, `LocalBranchWritten`, `RequiresHumanReview`, `AdapterDegraded`** — thirteen variants — and asserts "Phase 9 lifts the same event types into Postgres unchanged." But ADR-0034 says Phase 9 ships a **Postgres side-channel for workflow-spanning concerns** and **Temporal workflow history for workflow-internal events.** `PluginResolved`, `RecipeApplied`, `InstallSucceeded`, `LocalBranchWritten` are all workflow-internal. They will live in Temporal history in Phase 9, not in the side-channel Postgres event log. The "lifts unchanged" promise is wrong.
   **Why it matters:** All three designs make this mistake (see Cross-design observations). The best-practices design is the one that ships the most extensive event taxonomy under that wrong assumption, so the type-rot will be most concentrated here.
   **Where:** `§Components / EventLog writer`; `§Conventions honored / Event sourcing canonical primitive.`

3. **Problem:** **The design ships LangGraph as a Phase 3 dependency** ("the only framework-coupled code is the LangGraph subgraph wiring, which Phase 6 fleshed out the runtime for" in `§Goals`; "build_subgraph() — LangGraph" in the architecture diagram; subgraph nodes 1–5 in `§Internal design`). **The roadmap explicitly assigns LangGraph state-machine wiring to Phase 6 (per `docs/roadmap.md` line ~70: "Stage 4 — Execution... SHERPA subgraph: red-phase → apply-change → green-phase → commit"; ADR-0002 names LangGraph as runtime engine).** The performance lens correctly flagged "no LangGraph in Phase 3" as a deliberate avoidance. The best-practices design imports a heavyweight framework one phase early.
   **Why it matters:** LangGraph checkpoint writes are ~50 ms per node per write. With 5 nodes and checkpoints between each, that's 250 ms of overhead in Phase 3 for a feature Phase 6 owns. Worse, LangGraph couples the plugin to a runtime that Phase 5 will then *wrap* — if Phase 5 also wraps LangGraph state transitions, the dual-wrapping (Phase 5 retry envelope + Phase 6 LangGraph state machine) creates a topology Phase 5's design did not anticipate.
   **Where:** `§Architecture` diagram; `§Components / Vulnerability remediation plugin` internal design.

4. **Problem:** **`PluginScope.matches` accepts `language: TaskClass | Literal["*"]` etc., but `PluginScope.parse` returns `Result[PluginScope, ParseError]`** — and the design simultaneously says `TaskClass = NewType("TaskClass", str)`, `Language = NewType("Language", str)`, `BuildSystem = NewType("BuildSystem", str)`. `NewType` is a no-op at runtime; it's a type-checker hint. So `Language("typescript")` is just `"typescript"`. The `Literal["*"]` arm of the union is the only thing the type checker can distinguish — and `Literal["*"]` overlaps with `Language` (since `Language = NewType("Language", str)` reduces to `str`). `Language | Literal["*"]` collapses to `str` at runtime and `str | Literal["*"]` (i.e., `str`) for the type checker. The "newtype + Literal" claim is type-checker theater.
   **Why it matters:** The design defends this in `§Open questions #1`: "We could have made `*` a sum-type variant (`Concrete(value) | Wildcard()`) — slightly more illegal-state-unrepresentable — but `Literal['*']` reads better in YAML." It then files this under "documented as a known weakness." But ADR-0033's whole point is to *make illegal states unrepresentable*. A YAML aesthetic doesn't beat an ADR.
   **Where:** `§Components / PluginScope` interface; `§Patterns deliberately avoided.`

5. **Problem:** **The design's `Plugin` Protocol declares `def cve_feed_parsers(self) -> dict[CVEFeedName, CVEFeedParser]`** — and this is part of the *Plugin* contract, not the *VulnerabilityRemediationPlugin* contract. **Phase 7's `distroless-migration--node--npm` plugin has nothing to do with CVE feed parsers.** Phase 15's agentic-recipe-authoring plugin won't either. The plugin contract carries vuln-remediation-specific knowledge into the kernel — exactly the anti-pattern ADR-0031 was created to refuse ("a 'pluggable' design where the kernel still has a hardcoded list of plugin names").
   **Why it matters:** Phase 7 implementers will either (a) `return {}` from `cve_feed_parsers` (vestigial method, design smell), (b) reinterpret the method (silent contract drift), or (c) refactor the `Plugin` Protocol (Phase 3 contract change — Phase 7's "zero edits" exit criterion fails). The contract is wrong on day 1.
   **Where:** `§Components / Plugin (src/codegenie/plugins/protocols.py)` interface declaration.

6. **Problem:** **The design exports 8 names from `src/codegenie/plugins/` and 4 from the plugin's `api`** (the goal is `≤ 8`). The exported list includes `register_plugin`, `PluginRegistry`, `default_registry`, `Plugin`, `PluginManifest`, `PluginScope`, `PluginResolution`, `resolve_plugin`. **There is no `Adapter` Protocol, no `RecipeProtocol`, no `Bundle` model in that list — yet the design uses all three as public types in `§Components`.** Either the count is wrong (8 is undercounted; real public surface is closer to 15) or the components are violating the "small public surface" goal by being internally public-but-not-exported. Both readings are bad.
   **Why it matters:** ADR-0029/0031 designate `RecipeProtocol`, `Adapter`, and `Bundle` as cross-plugin shared types. They have to be public. The "8 names" metric is fake.
   **Where:** `§Goals / Public API surface (count)`; `§Components` (compare).

### Hidden assumptions

1. **Assumption:** That **Phase 6's LangGraph runtime is ready to run inside the plugin in Phase 3.** The design says "the only framework-coupled code is the LangGraph subgraph wiring, which Phase 6 fleshed out the runtime for" — past tense, as if Phase 6 has already shipped.
   **What breaks if it's wrong:** Phase 6 *has not shipped at Phase 3 time* (per the roadmap order — Phase 3 ships first; Phase 6 is "LangGraph state machine for the vuln loop"). So the plugin's `build_subgraph()` either imports LangGraph as a Phase-3 dep (with Phase 6's runtime semantics undefined) or stubs it out. The design imports it.

2. **Assumption:** That **`run_external_cli` is the Phase 2 chokepoint and adequate for Phase 3's npm/git/ncu invocations.** The design says "CLI tools `npm`, `npm-check-updates`, `git`, `jq` via Phase 2's `run_external_cli` allowlist (one entry added per binary)."
   **What breaks if it's wrong:** Phase 2's `run_external_cli` is designed for *read-only probes* (semgrep, syft, grype, gitleaks). It's not designed for `npm install` which writes to a `node_modules` tree, mutates a lockfile, and produces 300+ MB of side effects. The chokepoint may need a wider mode (writable working dir, larger time budget, larger memory) that Phase 2 didn't ship.

3. **Assumption:** That **`fcntl.flock` per Phase 6.5's pattern works for the event log writer.** Phase 6.5's audit writer used `fcntl.flock` for *eval-run audit records* — small writes, single file per run. The plugin event log writes many events per workflow per file.
   **What breaks if it's wrong:** Per-workflow-file means no contention; but if Phase 11 ever shares an event log across workflows (which it will, for the cost ledger projection), `flock` semantics change. The design picks per-workflow-file but the open question #6 admits "single file simplifies cross-workflow queries (which are Phase 9 territory anyway)" — meaning Phase 9 will move to a single file and `flock` semantics need re-examination.

### Things this design missed

- **The performance design caught: `vuln-index.sqlite` digest belongs in the bundle cache key.** Best-practices ships parsers per feed but doesn't ship a `VulnIndex` at all — every CVE lookup is a fresh JSON parse of the GHSA/NVD/OSV file. At portfolio scale this is a 50–200 ms parse per workflow that the performance design knocks down to 3 ms with an indexed sqlite. The best-practices defense ("Phase 9 introduces retention; document the gap explicitly") is for *event log* growth, not for vuln-feed parse cost.
- **The security design caught: the `git push` capability must not exist in Phase 3.** The best-practices design says "humans always merge" and "never `git push`" in `§Architecture / Internal design` of the plugin — but does not enforce it at the type level. The security design ships `GitLocalOpsCapability(repo, branch_prefix)` where "pushing is not a Phase 3 capability and minting one fails." A best-practices design that asserts "we don't push" without a capability barrier is a string-typed enforcement that a future plugin author can violate by adding one line.
- **The security design caught: `--ignore-scripts` enforcement at both flag and env-var level.** The best-practices design says it shells `npm` via `run_external_cli` but doesn't specify whether `--ignore-scripts` is on. If it isn't, postinstall scripts run on the orchestrator host.
- **The performance design caught: the *single most important* probe — `IndexHealthProbe` (B2) — flows into the adapter's `confidence()`.** The best-practices design has `NodeScipAdapter.confidence()` returning `Degraded(reason=ScipIndexStale(...))` based on `IndexHealthProbe` but does not stress-test the freshness signal in CI (the Phase 2 design ships a *stale-fixture* for B2). The best-practices test plan has unit tests for adapter `confidence()` but no end-to-end test that exercises a deliberately-stale SCIP fixture against the plugin — Phase 2's exit criterion was *exactly* "at least one real staleness case in CI." Phase 3's first plugin should be the first consumer that proves the signal propagates.

### Which disagreement matters most for *this* phase?

**`npm test` in Phase 3 — yes or no.** This is the only disagreement that touches the exit criterion directly. The performance design implicitly does not run tests; the security design explicitly defers them to Phase 5; the best-practices design runs `npm ci` (install verification) but not `npm test`. **The roadmap's exit-criterion sentence is: "the system writes a working patch diff on a local branch that — when applied — installs cleanly and passes the repo's own tests."** Read literally, Phase 3 must run the tests. Phase 5's final design (already merged) presumes Phase 3 ships a `RemediationOrchestrator` whose Stage 6 (Validate) Phase 5 *wraps* — and Phase 5's `final-design.md` line 51 says Stage 6 already exists in Phase 3 and is "wrapped by Phase 5" for the retry loop. So Phase 3 is supposed to ship Stage 6 (a validate stage that includes tests). If the synthesizer accepts the security design's deferral, **Phase 5's exit criterion (retry-1-fail + retry-2-recover) cannot be tested** because there's no first-attempt to retry.

The second-most-important disagreement is **recipe engine choice** — and the second-most-important *because* it surfaces a roadmap-vs-design tension the synthesizer must adjudicate (the roadmap says OpenRewrite; none of the three really ship it). The third is **LangGraph in Phase 3 yes/no** — two designs say yes, one says no, the roadmap puts LangGraph at Phase 6.

### Where do all three quietly agree on something questionable?

1. **All three converge on a non-OpenRewrite default recipe.** The roadmap explicitly names "OpenRewrite recipes (or `npm-check-updates` as a simpler first cut for cases where OpenRewrite is overkill)." All three designs demote OpenRewrite: performance defers it entirely; security keeps it as a fallback behind JVM jail; best-practices defers it to "later." **The roadmap may be stale, or all three may be missing what Phase 7 (distroless migration) will need.** Phase 7 owns structural refactors (`FROM` line edits, multi-stage build splits, layer caching) — those are *exactly* the OpenRewrite/AST-LST surface. If Phase 3 ships zero JVM infrastructure, Phase 7 will be inventing it from scratch under a "zero edits to existing files" constraint. The synthesizer should ask: is the OpenRewrite path being correctly identified as a Phase-7 concern, or being silently dropped from the roadmap?

2. **All three say "Phase 9 will lift the Phase-3 event log unchanged."** None of the three reads ADR-0034 carefully enough to notice that Phase 9 ships *both* Temporal workflow history (for workflow-internal events) and a Postgres side-channel (for workflow-spanning events). Phase 3's `PluginResolved`, `RecipeApplied`, `LocalBranchWritten` are all workflow-internal — they belong in Temporal history, not in the side-channel Postgres event log. **The "lifts unchanged" promise is wrong across all three designs.** Type rot will hit first at the Phase 9 boundary: either Phase 9 ships two storages and only some Phase 3 event types lift to one, or the event taxonomy has to be re-partitioned. The synthesizer should flag this commitment for Phase 9 review.

3. **All three call `plugins/universal--*--*/` "the right directory name"** without seriously testing the alternative (`plugins/_fallback/` with a manifest-declared wildcard scope). The literal `*` in a directory name has real costs: shell quoting on macOS/Linux, glob expansion when typing the path, file-listing tool friction, and (the security design notices but doesn't act on) "name-as-scope" couples the discoverability mechanism to the scope semantics. If the scope language ever grows a fourth dimension (e.g., region, deployment-class), every fallback directory has to be renamed. The convergence on `--*--*` is shared rather than reasoned.

## Roadmap-level critiques

1. **Phase 5 already designed; Phase 3's `RemediationOrchestrator`/`TrustScorer`/`ApplyContext` is a hard contract.** `docs/phases/05-sandbox-trust-gates/final-design.md` lines 53, 311, 327, 346, 642 all reference Phase 3 artifacts by name: `RemediationOrchestrator`, `TrustScorer`, `ApplyContext`, `Transform` ABC, `remediation-report.yaml`, BLAKE3 audit chain. **The performance design ships none of `RemediationOrchestrator`, `TrustScorer`, or `Transform`** — it ships async step functions, `BundleBuilder`, `RecipeMatcher`, `NpmRecipeEngine`, `PrEmitter`. **The security design ships `LocalGitOps`, `CapabilityMint`, `SubprocessJail`** — none named `RemediationOrchestrator` or `TrustScorer`. **The best-practices design ships `PluginRegistry`, `BundleBuilder`, `resolve_plugin`** — also no `RemediationOrchestrator` or `TrustScorer`. **None of the three named the artifacts Phase 5 already commits to consuming.** This is the most concrete cross-phase break. Phase 5's `StrictAndGate` thin adapter calls "Phase 3's existing `TrustScorer`" — *that scorer does not exist in any of the three designs.*

2. **The "first plugin doubles as proof the loader works" exit criterion (ADR-0031) is only really tested when there are two plugins.** All three designs ship `vulnerability-remediation--node--npm` + the universal fallback. The best-practices design admits this in `§Open questions #7` ("Whether to ship a third 'synthetic test plugin' in `plugins/_examples--*--*/`"). **The universal fallback isn't really a contract test — it has no TCCM derived queries, no adapters, no recipes.** It exercises the *resolver fallback path* but not the *plugin contract surface*. Phase 7 (distroless migration) will be the first time the contract gets exercised by something real; that's exactly the bake-test the ADR was created to avoid (`§Phase 7 becomes "author distroless-migration--node--npm"... requires zero edits to any file outside plugins/distroless-migration--node--npm/`). The synthesizer should consider whether Phase 3 needs the synthetic third plugin or not.

3. **All three import or imply LangGraph in some form, but ADR-0002 + roadmap Phase 6 own LangGraph.** Performance design is most defensible (says explicitly: "No LangGraph subgraph in Phase 3"). Security design uses LangGraph-shaped vocabulary (nodes, subgraph entry) without naming the framework. Best-practices imports it (`build_subgraph() -> StateGraph`). **If Phase 6 changes how subgraphs are wired, Phase 3 will need re-wiring.** The phase ordering (Phase 6 ships LangGraph; Phase 3 ships before Phase 6) means Phase 3 should ship something Phase 6 can wrap, not Phase 6's substrate prematurely.

4. **The roadmap's Phase 3 line "OpenRewrite recipes (or `npm-check-updates` as a simpler first cut)" is unstable** between the roadmap text and what Phase 7 will need. If Phase 7 = distroless migration involves AST-LST manipulation of Dockerfiles, OpenRewrite recipes make sense. If Phase 7 = something else, the OpenRewrite scaffolding is dead weight. The synthesizer should treat the OpenRewrite line as **an open contract between Phase 3 and Phase 7**, not as a Phase-3-only choice.

5. **Commitment §2.4 (Determinism over probabilism for structural changes) — the performance design's hedged-fallback violates this.** The Bundle Builder's race between primary and fallback adapter is *probabilistic* in scheduler timing. Same input → not necessarily same Bundle. The performance lens's defense ("threshold-hedge" alternative in `§Open questions #4`) doesn't change the fundamental conflict. Either Phase 3 commits to deterministic adapter dispatch (declared fallback fires only on low confidence) or commitment §2.4 is silently dented.

6. **Commitment §2.6 (Organizational uniqueness as data, not prompts).** All three designs honor this — TCCM, plugin manifest, recipe inventory, all YAML. Convergence; nothing to attack.

7. **Commitment §2.7 (Progressive disclosure).** All three honor this; Bundle holds references not contents. Convergence; nothing to attack.

## Design-pattern critiques (cross-cutting)

### Misapplied patterns (with concrete victim)

- **Performance's `RecipeEngine` Protocol is labelled Strategy but ships with one implementation.** The "second on the horizon" is the JVM/OpenRewrite engine — explicitly Phase 7. Strategy with one current implementation is **premature pluggability**. The design defends it ("Phase 7 ships the second implementation, the Phase-3 benchmark plan provides the data") but the benchmark plan is also Phase 3, with no actual JVM impl to bench. **Consequence:** the Protocol shapes itself around assumptions (the `(Repo, RecipePlan) -> Diff` shape, the `RecipeOutcome` variants) that may not survive contact with OpenRewrite's actual API.

- **Security's `Capability pattern` is labelled "make-illegal-states-unrepresentable" but the enforcement is a flake8/ruff rule and a Pydantic `model_validate` exception that requires stack inspection to actually fail forgery attempts.** As described, a determined plugin author writes `MyCapability.model_validate({"_minted_by": "x"})` and Pydantic constructs it. **Consequence:** the capability is forgeable in practice; the type-level safety claim is brand-not-substance.

- **Security's `Hexagonal Port + Adapter` is labelled around `SubprocessJail` (Linux `bwrap`, macOS `sandbox-exec`, future Firecracker).** But the security design admits Phase 5's microVM is a *third* substrate (Firecracker), and Phase 5's final design ships Firecracker as a real backend distinct from `bwrap`/`sandbox-exec`. So the "Port" is two implementations now and one different implementation later — fine as Strategy, but the "Port" framing requires the core to not know which adapter is active, while Phase 5's design ships `gate_isolation_class: "microvm" | "shared_kernel"` *that propagates downstream*. The core does know. **Consequence:** the hexagonal label is aspirational; the design carries the isolation class into the verdict, which is the *opposite* of hexagonal.

- **Best-practices' `Plugin` Protocol declares `cve_feed_parsers(self) -> dict[CVEFeedName, CVEFeedParser]` — vuln-remediation-specific knowledge in the kernel contract.** This is the very "kernel still has a hardcoded list of plugin names" anti-pattern ADR-0031 enumerates. **Consequence:** Phase 7's distroless plugin either returns `{}` (vestigial vuln knowledge in every plugin) or refactors the Protocol (Phase 7 "zero edits" fails).

- **Best-practices' `RecipeProtocol` ships with 2 implementations (NcuBumpRecipe, DirectDepPinRecipe).** This is *barely* not premature — 2 is the minimum that argues for an abstraction. But `DirectDepPinRecipe` is not described anywhere in the design; it's named once in the recipe-directory tree and never mentioned again. If `DirectDepPinRecipe` doesn't actually exist as code in Phase 3, the "two implementations" defense collapses to one. **Consequence:** the Protocol may also be premature pluggability with `DirectDepPinRecipe` as a fictional second implementation.

- **All three label their plugin loader "Plugin / Registry" but the kernel does mention plugin specifics** (security checks `capabilities-needed` against an allowlist; best-practices' `Plugin` Protocol carries CVE-feed-parser knowledge; performance walks `@register_plugin` decorators). **Consequence:** the kernel *isn't* zero-knowledge of plugins; the contract surfaces leak.

### Missed patterns (with concrete location)

- **All three missed `Newtype` on `WorkflowId`.** All three flow a `workflow_id` through CLI → loader → resolver → bundle → recipe → install → git → event log as `str` (or, in security's case, as `WorkflowId` but never used to discriminate against other UUID-typed identifiers — `BundleId`, `RunId`, `EventId`). A `WorkflowId`/`BundleId` swap is a type-checker no-op in all three.

- **Performance design missed `Tagged union` on `RecipeEngine.apply` outcome.** The design says "the diff produced is identical" between engines but doesn't ship `RecipeOutcome = Matched(Diff) | Refused(RefusalReason)` as the engine's return type. `Diff in memory` (the design's wording) is `bytes` or a typed model — unspecified.

- **Security design missed `Smart constructor` on `BranchName`.** The design says branch names are "validated against `^[a-z0-9-]+$` after canonicalization" — but the validation is asserted in prose, not in a `BranchName.parse(...) -> Result[BranchName, ValidationError]` smart constructor. If an attacker-influenced field (CVE id) feeds into the branch name and the regex check is forgotten in one call site, the type carries no evidence.

- **Best-practices design missed `Make-illegal-states-unrepresentable` on `PluginScope`.** It admits this explicitly (`§Open questions #1`) and ships `Literal["*"]` because "reads better in YAML." This is *one of the highest-value ADR-0033 applications* and the design declined it for aesthetic reasons. Phase 7 distroless will have a `(distroless-migration, node, npm)` triple where wildcards still need to coexist; **the `Wildcard()` variant being missed in Phase 3 propagates to every future plugin scope match.**

- **All three missed `Capability pattern` on the recipe-engine's filesystem-write ability.** Best-practices and performance both let the recipe engine read/write the repo directly without a capability mechanism. Only security ships `FsReadWriteCapability(jail)` — but doesn't fully type-enforce it. The capability *would* be the right pattern here; nobody implements it convincingly.

- **All three missed `Open/Closed` on the recipe-engine selection.** All three have a hardcoded "default engine" choice (perf: `NpmRecipeEngine`; security: pure-Python `NpmDepBumpRecipe`; best-practices: `NcuBumpRecipe`). Adding a fourth engine requires editing the registration order or the matcher. The matcher *is* the registry per design, but the **precedence** logic is unspecified — a new engine has to pick a precedence value that doesn't collide, and the existing engines' precedence values aren't published.

### Pattern claims that don't survive scrutiny

- **Performance's "Hexagonal-ceremony around `subprocess.run`" rejection** (`§Anti-patterns avoided`): "_run_npm_install(...) is a function, not a `NpmRunnerPort` with two Adapters. We have one adapter (npm). Wait for the second before extracting." Good rejection — *unless* Phase 5's Firecracker adapter wraps subprocess calls, at which point Phase 3's `_run_npm_install` is the call-site Phase 5 needs to wrap. The rejection is correct in isolation; it's wrong in roadmap-context.

- **Security's "TB-A capability tokens cross this line"** trust boundary requires that no module outside `codegenie.cap` constructs a Capability. The enforcement is a flake8/ruff rule. Lints can be skipped (`# noqa`), bypassed (`importlib.import_module("codegenie.cap")` and then `getattr(..., "NpmInstallCapability")(...)`), or run only on a subset of files. The trust boundary is asserted, not engineered.

- **Best-practices' "≤ 8 names exported from `src/codegenie/plugins/`"** — the count is fake. `Adapter`, `Protocol`, `Bundle`, `RecipeProtocol`, `RecipeOutcome`, `AdapterConfidence`, `CVERecord`, `Event`, `BundleEntry`, `BundleQueryHandle`, `PluginsLoaded`, `PluginResolved`, `BundleBuilt`, `BundleEntryPromoted`, `RecipeMatched`, `RecipeApplied`, `RecipeSkipped`, `RecipeFailed`, `InstallSucceeded`, `InstallFailed`, `LocalBranchWritten`, `RequiresHumanReview`, `AdapterDegraded`, `FallbackResolution`, `ConcreteResolution`, `FallbackReason`, `SkipReason`, `NotApplicableReason`, `RecipeError`, `InstallError`, `BranchName`, `BundleId`, … are all imports a plugin author or Phase 5 consumer must reach for. The "small public surface" metric is undercounted by an order of magnitude.

- **All three claim "Event sourcing"** with per-workflow JSONL files. Event sourcing isn't a *file shape*; it's the property that state is reconstructable as a fold over events. Best-practices' design includes a `replay()` method on `EventLog`; security has `codegenie audit verify`; performance has the digest-chain narrative. **Only best-practices ships a property test that asserts replay round-trips state** — the other two assert event sourcing without proving it. Performance's "Phase 9 will replay these into Postgres unchanged" is a promissory note.

### Anti-patterns from the toolkit's "flag on sight" list

- **Pattern soup (performance).** The performance design names: PluginRegistry, BundleBuilder, RecipeMatcher, RecipeRegistry, NpmRecipeEngine, RecipeEngine, ValidationProbe, EventAppender, UniversalFallbackPlugin, RepoContextLoader, CveLookup, PrEmitter, VulnIndex, ScopeMatcher, PluginResolver, ResolvedPlugin. That's 16 named components in a single phase, each a candidate pattern application. Many earn their names; several feel like Builder/Matcher/Resolver ceremony. The naming density is itself a smell.

- **Premature pluggability (performance and best-practices).** Performance: `RecipeEngine` Protocol with one impl + one Phase-7-future. Best-practices: `RecipeProtocol` with `NcuBumpRecipe` + the never-described `DirectDepPinRecipe`. The toolkit's flag is unambiguous ("Strategy with a single implementation = unnecessary indirection. Wait for the second implementation before extracting").

- **Stringly-typed identifiers (best-practices).** `TaskClass = NewType("TaskClass", str)`, `Language = NewType("Language", str)`, etc. — `NewType` reduces to `str` at runtime; the type-checker discriminates statically but a JSON-deserialize path (`PluginScope.parse(yaml_dict["scope"])`) can mint a `Language` from any string without validation. The "smart constructor returns `Result[T, ParseError]`" defense covers the load path but not the runtime mint path (e.g., `Language("typescirpt")` — typo, no error).

- **Boolean flags (best-practices, performance).** Best-practices: `RecipeProtocol.applies(cve, context) -> bool` — `bool` for what `§Open questions #5` admits should be `Applies | NotApplies(reason) | Maybe(needs_more_context)`. Performance: `IndexStaleAtWorkflowStart event emitted and continue with degraded confidence` — `continue` is a bool; the typed alternative ("the workflow runs with `confidence: degraded` propagated to the verdict") is named in security's design but not performance's.

- **Tag-and-dispatch without sum type (performance).** The performance `RecipeMatcher` walks "the candidate list (4 recipes in Phase 3: `NpmSemverBumpRecipe`, `NpmPeerDepConflictRecipe`, `NpmTransitiveOnlyRecipe`, `NpmMajorBumpRefuseRecipe`)... The first `Matched` wins." There's no `RecipeKind = Literal["semver_bump", "peer_dep_conflict", "transitive_only", "major_bump_refuse"]` sum type controlling dispatch; the design says `@register_recipe(ecosystem=Ecosystem.NPM, kind=RecipeKind.SEMVER_BUMP)` but the kind enum is asserted, never specified. The "first Matched wins" semantics implies dispatch-by-trial, which is not the same as dispatch-by-tag.

- **Capability passed through ten frames as a parameter (security).** The security design's `apply_npm_dep_bump(capability: NpmInstallCapability, target: SandboxedPath, package: PackageId, to_version: SemverVersion)` is the function-level capability injection. Multiplied across recipe nodes, install nodes, validate nodes, git nodes, the capability set passes through every frame. The design's comment "this *is* the point — the verbosity is the audit trail at compile time" defends this — but the toolkit's flag is "Capability passed through ten frames as a parameter.
