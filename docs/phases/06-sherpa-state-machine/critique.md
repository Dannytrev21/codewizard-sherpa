# Phase 6 — SHERPA-style state machine for the vuln loop: Devil's-advocate critique

**Reviewed by:** Devil's-advocate critic subagent
**Date:** 2026-05-12

## Method

I read all three designs and attacked each on its own terms. I do not propose alternatives. My job is to surface what the synthesizer needs to see before it merges.

## Attacks on the performance-first design

### Concrete problems

1. **Problem:** The design literally tells the synthesizer it has unilaterally rewritten Phase 5's contract — risk-1 + open question 1 + acknowledged blind spot 4 all admit performance favors interpretation (a) ("re-apply the same `RecipeApplication` and let the gate re-evaluate") while Phase 5's final-design exit-criterion #19 explicitly says "Phase 4's `FallbackTier.run` is invoked with `prior_attempts` and produces a *different* `RecipeApplication`." The §Data-flow walkthrough at t≈70s reads "the same recipe is re-applied with `prior_attempts` carried in `ApplyContext`" — that is **not what Phase 5 exit-criterion #19 says ships**, and Phase 5 §3-step-10 of its own data flow says the orchestrator re-enters Phase 4 on retry and gets a *different* patch.
   **Why it matters:** This is the load-bearing piece of Phase 5's exit criterion. If Phase 6 ships interpretation (a), the Phase 5 retry parity test (`test_retry_semantics_parity.py` — performance §Test #4) cannot pass byte-for-byte because Phase 5's `attempts.jsonl` contains *distinct patch bytes* across the two attempts (Phase 5 final-design §511) and performance's path produces *identical patch bytes*. Performance has put a test in their plan that, run today, would fail their own design.
   **Where:** §Data flow step 8; §Risks risk-1; §Open questions Q1; §Test plan item 4 (parity test); §Acknowledged blind spots item 4.

2. **Problem:** The "checkpoint write returns when queued, not when persisted" trick (§Component 3) directly contradicts the exit criterion. Roadmap Phase 6 says "Mid-run kill + resume works **without state loss**." Performance §Failure modes row 1 says "at most one in-flight node re-runs" — that is state loss the moment the in-flight node has *visible side effects* (a recipe-application patch is written to disk, the Phase 5 audit chain is extended, a sandbox VM was booted and a cost was paid). The 50 ms "next gate boundary" durability bound means there is a 50 ms window where Phase 5's BLAKE3 audit chain has appended an event for which no checkpoint exists.
   **Why it matters:** Phase 5's audit chain is BLAKE3-linked and append-only. If the next checkpoint claims a `prev_chain_head` older than what the audit chain actually contains, Phase 5's `AuditChainCorrupted` raise (Phase 5 §339 `chain verification test at startup`) fires and the workflow is dead. Performance has set up the checkpointer and Phase 5's chain to drift; the failure mode is detected, but the recovery story ("salvage tool reads Phase 5 RetryLedger") in §Failure modes row 4 is hand-waved as "reconstructs minimal state to resume" with no defined contract.
   **Where:** §Component 3 ("WAL mode + NORMAL fsync ... node returns before fsync"); §Failure modes rows 1 and 4; goal 5 ("off the hot critical path via background flush").

3. **Problem:** Goal-4 ("p50 ≤ 2 ms / p95 ≤ 8 ms" per-node overhead) is asserted with **zero measured evidence** — §Acknowledged blind spot 1 admits "No actual portfolio-scale benchmarks ... derived from Phase 4/5 measured latencies plus LangGraph's published per-node overhead." LangGraph does not publish a per-node overhead number. The canary test (§Component 6, §Test 5) asserts the budget; nothing measures the actual LangGraph runtime overhead at Phase-6 design time. A CI failure on day one of merge is a real risk.
   **Why it matters:** Goal 1 (≥ 40 wf/hr/worker) depends on Goal 4 + Goal 5 + Goal 14 all holding. If LangGraph 0.x's actual per-node overhead is, say, 15 ms p95 on the target hardware (it has been historically slow with Pydantic-validated state), the entire performance lens collapses — and the synthesizer has nothing to fall back to because Phase 6 is supposed to be the perf-critical envelope.
   **Where:** §Goals row 4; §Component 6; §Acknowledged blind spots row 1.

4. **Problem:** Module-level compile (`VULN_LOOP: CompiledGraph = _build().compile(checkpointer=...)`) bakes the checkpointer path into a module-level singleton at import time (§Component 1, §Data flow step 1). Best-practices and security both treat the checkpointer as a per-invocation argument; performance ships it as a process-level constant. Two consequences: (a) test isolation in `tests/perf/test_canary_overhead.py` requires the canary node test to mutate the singleton (which performance does not address), and (b) the `--checkpointer-db <path>` CLI flag (§Component 7) cannot actually re-wire the compiled singleton without re-compiling — but performance just said "saves ~80 ms/wf cold start" by not re-compiling.
   **Why it matters:** Either the CLI flag works and the cold-start win is fictional, or the cold-start win is real and the CLI flag is a no-op. The design ships both claims.
   **Where:** §Component 1 ("Compiled once at import time"); §Component 7 ("All commands accept `--checkpointer-db <path>` for test isolation").

5. **Problem:** The sensitivity classifier (§Component 2: `sensitivity_class(state) -> Literal["low","medium","high","critical"]`) is asserted to read CVE severity from `advisory_ref` and a "repo-criticality flag from `repo_context_path` *manifest*" — but **no manifest field for repo-criticality exists in Phase 1 or Phase 2's `RepoContext` schema**. Performance has invented a field that does not exist and built routing behavior on it (the interrupt/auto-fail decision in §Component 5 hangs off it). The "Default `sensitivity_class` for vuln remediation in Phase 6 = `high` (everything in the vuln loop hits human eyes)" hack in §Component 5 reveals the classifier is functionally a constant.
   **Why it matters:** If sensitivity always returns `high`, the auto-fail path (low/medium sensitivity → fail without HITL) is dead code — and Phase 5's exit-criterion semantics (gate exhaustion → `escalate`, full stop) are simply replicated, not extended. The whole sensitivity classifier is a future-Phase-7-thing pretending to be a Phase-6-thing.
   **Where:** §Component 2 ("Sensitivity classifier is pure"); §Component 5 ("Sensitivity → interrupt mapping").

### Hidden assumptions

1. **Assumption:** That the dict-return-from-node discipline (returning `{changed_field: value}`) is a *performance* optimization that LangGraph honors as a delta.
   **What breaks if it's wrong:** LangGraph's reducer-merging applies the dict, but then the next checkpoint serializes *the full state*, not the delta. The 1.5 ms `model_dump_json` cost cited in risk-4 is paid on full state every time. The "checkpoint cost grows with the *delta*, not the *whole state*" claim in §Lens summary is incorrect unless the checkpointer is configured to write deltas — LangGraph's stock `AsyncSqliteSaver` does not by default; it writes the full checkpoint per step.

2. **Assumption:** That single-writer SQLite at 800 writes/s/worker (Goal 6) is achievable with `synchronous=NORMAL` and WAL on a developer-grade SSD.
   **What breaks if it's wrong:** The 6× headroom claim in §Component 3 evaporates and the queue saturates at one worker. Performance treats this as a non-issue ("forward-compatible to Postgres in Phase 9") but Phase 6 testing happens on macOS dev laptops, where aiosqlite + WAL behavior under sustained load is *explicitly* not benchmarked (§Acknowledged blind spots row 3).

3. **Assumption:** That the Phase 5 `GateRunner` body can be "lifted verbatim" into LangGraph nodes (§Component 4) without breaking Phase 5's `for attempt in range(1, max_attempts + 1)` loop's variable capture and the `with_prior_attempt` immutable-ctx contract.
   **What breaks if it's wrong:** The dual-implementation parity test (§Test 4) is the only safety net, and §Risk-1 admits the *retry feedback semantics* between the two paths may differ. If they differ, byte-identical `RetryLedger` output is impossible and the test cannot pass. Performance keeps Phase 5's sync `GateRunner.run` "for sync-only callers (CLI smoke tests, Phase 5 regression tests)" — but that means two retry implementations live in production code indefinitely, exactly the kind of drift Phase 5's critique landed on (per Phase 5 final-design §331).

### Things this design missed that a different lens caught

- **Tamper-evidence on the checkpointer DB.** The security design's `AuditedSqliteSaver` (security §Component "AuditedSqliteSaver") catches an entire class of attack performance ignores. Performance §Failure modes row 4 ("SQLite file corruption ... `LoopAborted("checkpoint db corrupt")`") detects *corruption* but not *forgery* — an attacker who edits the DB to flip `verdict: fail → pass` (security adversary 1) is invisible to performance's design. Performance ships a `LoopAborted` for the wrong failure mode.
- **Schema-versioning of the checkpoint blob.** Best-practices ships `schema_version: Literal["v0.6.0"]` (best-practices §State ledger versioning) and security ships `schema_version = blake3(...)` (security §Checkpointer security). Performance §Failure modes row 7 mentions `CheckpointSchemaIncompatible` but does not specify how the persisted version is encoded or checked — the schema is just "v1" with no migration plan.
- **Resume payload integrity.** Performance's resume is `VulnLoop.ainvoke(Command(resume=ApprovalDecision(...)), config={thread_id})` (§Component 5), unauthenticated, with no signature on the `ApprovalDecision`. A `codegenie loop resume <thread_id> --decision approve_and_retry` from any local user advances any paused workflow. Security catches this (their HITL CLI requires Ed25519 signature); performance does not even mention it.

## Attacks on the security-first design

### Concrete problems

1. **Problem:** The whole HITL operator-key story — `~/.config/codegenie/operator.key`, `codegenie operator init`, `expected_approvers.json`, Ed25519 signing, gate-control HMAC envelope signing — is **massively out of scope for Phase 6 as the roadmap defines it**. Roadmap Phase 6 (line 117): "`langgraph`, `pydantic`, `aiosqlite` for the checkpointer, `langgraph-cli` for graph inspection." Roadmap Phase 11 (not Phase 6) owns PR-comment dispatch and approval ingestion. Security has imported Phase 11–16 of the roadmap into Phase 6 and called it "the minimum security posture."
   **Why it matters:** A `codegenie operator init` CLI, an operator-key rotation flow, an HSM upgrade story in Phase 16, a gate-control HMAC handshake "continued from Phase 5" (which Phase 5's final-design does not actually establish — Phase 5 §rejects the daemon outright), and three signature primitives at three layers add ~600 LOC of crypto plumbing to a phase that the roadmap scopes to "LangGraph + Pydantic + SQLite." The Phase-6 budget (best-practices target 1200 LOC) is blown three times over before any state machine code is written.
   **Where:** §HITL interrupt/resume protocol; §Goals "HITL authentication model"; §Architecture (`operator_key.py`, `cli_hitl.py`, `expected_approvers.json`); §Risks 1.

2. **Problem:** The `@side_effect` decorator + replay-from-chain machinery (§Components, §Architecture `sideeffects.py`) duplicates work Phase 9 (Temporal envelope, roadmap line 151) is supposed to own — "All retries are framework-level — application code contains no retry loops" — and uses a non-standard mechanism (audit-chain lookup of `sideeffect.started` / `sideeffect.completed`) that conflicts with both LangGraph's checkpointer model and the Temporal Activity idempotency model. Security explicitly notes this in §Architecture but ships it anyway.
   **Why it matters:** Phase 9 will throw all of `sideeffects.py` away. The `recovery_required=true` interrupt path (security §Components "@side_effect decorator") that escalates partial side effects to humans is unique to Phase 6, untestable in Phase 9 (because Temporal Activities don't half-complete in the same way), and trains operators on a workflow that will not exist in 4 phases. This is the kind of investment Phase 0's "extension by addition" promise (CLAUDE.md, commitment §2.5) was supposed to prevent.
   **Where:** §Architecture `sideeffects.py`; §Components "@side_effect decorator"; §Failure modes row "Side-effect started but not completed."

3. **Problem:** The `@reducer` + field-level `read_acl` / `write_acl` machinery (§Goals "State-ledger invariants", §Components "@reducer decorator") is **fighting LangGraph's idiom**. LangGraph's reducer-merging is *automatic*; nodes return updated state and LangGraph reconciles. Security replaces this with `ReducerCall("add_attempt", summary)` returns and an `apply_reducer(state, call)` after-node hook. This is not idiomatic LangGraph; the "framework, not against it" principle that best-practices §Conventions calls out (and that ADR-0002 explicitly endorses — "use LangGraph's mature tooling for free") is broken.
   **Why it matters:** The runtime ACL check (`enforce_field_acl(state_before, state_after, node_name)`) requires the node to never directly return a `LoopState` — but LangGraph's node contract is `state -> state` or `state -> dict`. Security's `ReducerCall` is a third return type that requires a custom LangGraph wrapper. Every node is now wrapped, every test mocks a custom dispatcher, and the `langgraph-cli` topology rendering (which security purports to support in §Components "Operations") cannot introspect a graph whose nodes return `ReducerCall` instead of state. Best-practices §Component 2 explicitly rejected this shape ("Returning dicts (not full state) is a SHERPA-correctness footgun ... mitigated by a pytest plugin").
   **Where:** §Components "@reducer decorator + reducers"; §Goals row "State mutations are produced via typed reducer functions"; §State ledger schema "every field carries Field(json_schema_extra={...})".

4. **Problem:** Banning `random`, `time`, `os`, `datetime` (except `fromisoformat`) inside `loop/edges/` (§Components "@pure_edge decorator", §Threat model "Non-determinism-pivoted adversary") is correct in principle but security has not noticed that one of their own edge functions in §Conditional edges (`route_after_validate`) constructs `AttemptSummary` comparisons over `s.attempts[-1]` and `s.attempts[-2]` — if those `AttemptSummary` objects contain timestamps (they do, per Phase 5 — `attempts.jsonl` BLAKE3-chained JSON lines have wall-clock fields), the edge function indirectly reads time. The fence-CI check is shallow (AST `import` only); it cannot catch transitive reads.
   **Why it matters:** The 10k-Hypothesis-state property test (§Goals "Replay-safety property") tests determinism on synthetic states, but production states carry timestamps from upstream. The "replay-equivalent" guarantee is asserted but not enforced for the data the edges actually consume.
   **Where:** §Components "@pure_edge decorator"; §Conditional edges "route_after_validate"; §Test plan "Property tests".

5. **Problem:** "JSON-only serializer; MessagePack rejected" (§Goals, §Checkpointer security) is a *justified* but *unmotivated* security claim — MessagePack ext types are an attack surface "for naive consumers." The naïve consumer in Phase 6 is `langgraph-cli`, which security has *already* firewalled off to read-only file copies (§Threat model "`langgraph-cli` pivot"). With the CLI already isolated and the file at `0600`, the JSON-vs-MessagePack debate is choosing the slower serializer to defend against a threat that doesn't exist in Phase 6's threat model. The size-cap claim (`≤ 64 KB per checkpoint blob`) further constrains state shape in ways performance and best-practices do not (their VulnLedger easily exceeds 64 KB once `prior_attempts` accumulates structured Phase 5 `AttemptSummary` objects with their `evidence_paths` dicts).
   **Why it matters:** Security ships a hard `CheckpointTooLarge` raise (§Failure modes) that will fire on a normal 3-retry vuln workflow once the `attempts` tuple reaches 3 entries with full Phase 5 `AttemptSummary` payloads. This is an unforced architectural cap.
   **Where:** §Goals (size cap, JSON-only); §Checkpointer security "Serializer"; §Failure modes "Pydantic validation fails on a deserialized checkpoint blob."

6. **Problem:** `frozen=True` on `LoopState` (§Goals "State-ledger invariants" 1) is incompatible with LangGraph's `add_conditional_edges` semantics — LangGraph's checkpointer creates new instances via Pydantic's `model_validate`/`model_dump_json` round trip, which technically works with frozen models, but every node mutation via `model_copy(update=...)` allocates a new instance. Best-practices §Component 1 explicitly chose `frozen=False` because "LangGraph applies node-returned updates by constructing a new instance." Security's `frozen=True` either forces every node to construct the new instance manually (which they enforce via `@reducer` — fine in their world but disconnected from LangGraph's reducer-merging) or means `LoopState` is never actually frozen at runtime.
   **Why it matters:** Two of the three designs have made a hard choice on `frozen`. The synthesizer cannot average — one path makes LangGraph reducer-merging work natively and accepts the discipline cost; the other replaces LangGraph reducer-merging with a custom dispatcher and claims `frozen=True`. The synthesizer must pick.
   **Where:** §Goals "State-ledger invariants" 1; §Components "LoopState".

### Hidden assumptions

1. **Assumption:** That the Phase 5 BLAKE3 audit chain has an unambiguous "current head" that the orchestrator can `O_APPEND` to atomically with a SAVEPOINT'd SQLite write (§Components "AuditedSqliteSaver").
   **What breaks if it's wrong:** Phase 5's `RetryLedger` writes one JSON line per attempt — those lines come from `GateRunner.run` (sync, in-process). Phase 6's `AuditedSqliteSaver.put` extends the chain again, in the same process, but Phase 5 wrote bytes to a different file (`attempts.jsonl`) than security's `audit/<run-id>.jsonl`. Whether these are the same file or different files is **unspecified**. If they're different files, there are two BLAKE3 chains with no defined cross-link; if they're the same file, `O_APPEND` from two different writers (Phase 5 inside `validate_in_sandbox`, Phase 6 in the checkpointer wrapper) races on every gate evaluation.

2. **Assumption:** That `schema_version = blake3(canonical_sorted_json(LoopState.model_json_schema()))` is stable across Pydantic version bumps and Python versions.
   **What breaks if it's wrong:** Pydantic v2 has changed `model_json_schema()` output between minor versions (e.g., `2.5 → 2.6` reorderings, `2.7` adding `$defs` differently). A `pip install --upgrade pydantic` recomputes the schema version on a checkpoint file's resume and triggers `SchemaDrift`, refusing to resume **every** in-flight workflow on every developer's machine after a routine Pydantic bump. The "single biggest operational hazard" (security risk 3) is one `dependabot` bump away from firing constantly.

3. **Assumption:** That tests can inject a real Ed25519 keypair fingerprint into the production verifier without it being a backdoor (§HITL "Mocked-human responses in tests cannot be a prod backdoor").
   **What breaks if it's wrong:** The `test_no_test_keys_in_production_approvers.py` static check walks "`tools/` and `~/.config/codegenie/operators.json` candidates in the repo." `~/.config/codegenie/operators.json` is *not in the repo* — it's on the operator's machine. The check cannot see prod fingerprints; it only ensures tests don't ship prod fingerprints. The reverse — a prod operator's machine accidentally including a test fingerprint via copy-paste of an example config — is unguarded.

### Things this design missed

- **The performance design's compile-once-per-process win.** Security's `build_vuln_loop_graph(...)` is called on every `codegenie remediate` invocation (§Architecture "Phase 3 RemediationOrchestrator"). The 80 ms cold-start cost performance identified is paid on every CLI run. Security never measures or even mentions cold-start.
- **The best-practices design's golden-graph snapshot test.** Security has no concept of topology snapshot — a future PR can re-wire the graph (move `await_human_review` to between `apply_transform` and `validate_in_sandbox`, say) and nothing in security's test plan catches it as long as the property tests on `route_after_validate` still pass. Best-practices' `vuln_graph.json` golden test (best-practices §Components 8 "Golden-graph snapshot test") would catch this; security doesn't have it.
- **A pre-existing Phase 5 contract: the `--max-attempts-override` CLI flag.** Phase 5 final-design ships `--max-attempts-override <int>` with `--operator-ack`. Security's "no override flag" stance in §Conditional edges "Second-consecutive failure" notes Phase 5's flag "survives" but says "the value is locked into `LoopState` at entry and cannot be changed mid-run" — but if a paused workflow needs a higher cap mid-resume, security has just locked it out. Phase 5 designed for this; security undid it.

## Attacks on the best-practices design

### Concrete problems

1. **Problem:** `VulnLedger.retry_count: int = 0` "represents lifetime attempts on this workflow" and is **incremented in record_attempt but never reset except on HITL "continue"** (§Conditional edges "Where the three-retry counter lives"). The conditional edge `route_after_attempt` reads `retry_count >= max_attempts`. But because `retry_count` is monotonically increasing across the workflow's entire life and `max_attempts=3`, **a single recipe-path failure followed by a single RAG-path failure followed by a single LLM-path failure burns the budget** — the workflow escalates to human after one attempt per engine, not three attempts per engine. ADR-0014 is "three retries *per gate transition*" (ADR-0014 title); best-practices has made it "three attempts *per workflow lifetime*."
   **Why it matters:** This directly violates ADR-0014. The exit criterion ("HITL fires when trust gates fail twice in a row") becomes "HITL fires when total attempts reach 3 across any combination of engines." The retry parity test against Phase 5 (Phase 5's loop is per-gate with `max_attempts=3`) cannot pass — Phase 5's `RetryLedger` for a recipe→RAG→LLM-with-3-attempts run would have 5 entries; best-practices' graph would escalate after 3.
   **Where:** §Conditional edges "Where the three-retry counter lives"; §Components "Conditional-edge predicate contract" `route_after_attempt`.

2. **Problem:** The "every node returns `state.model_copy(update={...})`" enforcement (§Components 2 "convention enforced by ruff + a CI introspection test") plus the docstring `Reads:` / `Writes:` AST-validation contract (§Node contract) is **complexity-budget tunnel vision**. The CI test that parses module docstrings to build a field-level access map (~80 LOC of AST walking) is itself a maintenance burden the design doesn't price: every new node author has to author *and maintain* the docstring contract, the AST walker has to support every legitimate `state.<name>` access pattern (chained access? `getattr(state, name)`? tuple destructuring?), and a passing-but-wrong docstring is unfalsifiable. Best-practices §Component 2 cites this as the load-bearing safety mechanism but ships zero edge cases for the AST walker.
   **Why it matters:** This is the kind of "lint that becomes truth" Rule 9 (Global) warns against — a test that encodes intent ("nodes declare what they read/write") but verifies behavior trivially (does `state.x` syntactically appear?). A node author who reads `state.recipe_selection.recipe` and forgets to declare `recipe_selection` in `Reads:` produces a passing test if they happen to also access `state.recipe_selection` somewhere else (or vice versa). The test claims more than it can prove.
   **Where:** §Node contract "How a node declares the fields it reads vs writes"; §Components 2 "The function body MUST end with `return state.model_copy(update={...})`."

3. **Problem:** The route-back-to-engine logic (§Components 3 `route_after_attempt`: `return f"retryable_{state.last_engine}"`) wires three different "retryable_recipe / retryable_rag / retryable_llm" edges into `record_attempt`. But `state.last_engine` is set inside the engine node (`apply_recipe` sets `last_engine="recipe"`, etc.) — meaning the route-back graph has a topology that's **valid only if the last engine's node actually wrote `last_engine`**. If a node fails and the checkpoint captures pre-write state, `state.last_engine` is `None` and the f-string produces `"retryable_None"`, which is not in the conditional-edges mapping — KeyError at routing time.
   **Why it matters:** §Conditional edges section asserts the edge "reads booleans and counters from the state ledger that were written by Phase 5's signal collectors." Best-practices §Components 3 asserts the `assert state.last_engine is not None` will catch this — but `assert` is the wrong primitive (raises `AssertionError`, not a routing decision) and assertions can be disabled with `python -O`. The topology has a soft-fail mode the design treats as impossible.
   **Where:** §Components 3 "route_after_attempt"; §Architecture diagram "retryable_recipe / retryable_rag / retryable_llm" branches.

4. **Problem:** `frozen=False` on `VulnLedger` is justified as "LangGraph's update merging works idiomatically" (§Components 1) — and then the "no in-place mutation" rule is enforced by **AST lint, not by Pydantic** (§Components 1 "Tradeoffs accepted"). But the AST lint to detect in-place mutation (e.g., `state.events.append(...)` or `state.prior_attempts.append(...)` on the *list value* of a field — Python's reference semantics on mutable values) is dramatically more complex than the simple `model_copy` check the design ships. A `state.events.append(e)` is syntactically `getattr` + method call, not assignment; the design's "AST lint" cannot catch it without dataflow analysis.
   **Why it matters:** Two designs picked `frozen=False` and `frozen=True` for the same reason (LangGraph compatibility) — best-practices chose `frozen=False` and then admitted it must lint a behavior pure Pydantic would have prevented. The lint claim is hand-waved.
   **Where:** §Components 1 "frozen=False intentional"; §Test plan "Layer 0 — Static checks" (no entry for in-place-mutation detection).

5. **Problem:** The "supervisor stays out of Phase 6" decision (§Open question 1) directly creates the refactor Phase 8 has to do. Phase 8 (roadmap line 137) is "Hierarchical Planner + pre-rendered hot views" — and the planner is what routes between vuln, distroless, and future subgraphs. Best-practices says `cli/remediate.py` calls `build_vuln_graph()` directly, "Phase 7 (distroless) adds a sibling builder; Phase 8 mints the supervisor." But Phase 7's exit criterion (roadmap line 133) is "Both task classes run from the same orchestration. The diff for this phase touches *only* new files — no Phase 0–6 source code is modified." If Phase 7 must not edit Phase 6 source, then either (a) `cli/remediate.py` is not Phase-6 source (best-practices made it a Phase 6 *edit* via ADR-P6-001) and Phase 7 can edit it, or (b) Phase 7 adds its own CLI entry — splintering the user surface. Best-practices has set up Phase 7 to either violate its own roadmap exit criterion or fragment the CLI.
   **Where:** §Open questions Q1; §Architecture "Phase 0–5 source code: unchanged. The cli/remediate.py edit is the only Phase-0–5 touch, and it is a one-line dispatcher swap captured by ADR-P6-001."

### Hidden assumptions

1. **Assumption:** That `langgraph-cli` can render the `build_vuln_graph()` output stable-deterministically into an SVG that survives `langgraph` minor-version bumps (§Goals 18 "CI job that re-renders and diff-checks the committed SVG").
   **What breaks if it's wrong:** `langgraph-cli` rendering is not a documented stable contract. SVG output includes node coordinates that depend on the layout algorithm version — every `langgraph-cli` upgrade re-flows the SVG. The CI gate flips red on a Renovate PR that bumps `langgraph-cli` from 0.0.x → 0.0.y. Best-practices §Risks 2 acknowledges "LangGraph idiom drift" but does not extend that risk to the SVG snapshot.

2. **Assumption:** That `graph.get_graph().to_json()` is itself a stable, deterministic API across LangGraph versions (§Goals 13 "Golden-graph snapshot test").
   **What breaks if it's wrong:** Same issue. LangGraph's `to_json` output schema includes runtime-generated IDs (node identifiers, edge labels' canonicalization). Best-practices hand-waves "after deterministic key sort." Whether the underlying dict has stable IDs is not asserted, and the design has no fallback if it doesn't.

3. **Assumption:** That AsyncSqliteSaver is the right primitive for sync nodes (every node is `def`, not `async def` per §Components 2 "every node in Phase 6 is sync").
   **What breaks if it's wrong:** `AsyncSqliteSaver` is *async*. Sync nodes inside a graph compiled with an async checkpointer either (a) require the whole `ainvoke` path (which best-practices' `cli/remediate.py` edit does call), introducing an event loop the sync node bodies don't expect, or (b) cause LangGraph to bridge sync/async in unspecified ways. Best-practices §Component 2 explicitly says "sync nodes in async graphs creates a stack-jumping debugging trap" — and then ships sync nodes in an async checkpointer graph.

### Things this design missed

- **Cost ledger integration.** Performance §Goals 10 surfaces "0 tokens for Phase 6 envelope" and integrates with Phase 4's `LlmInvocationGuard`. Best-practices §Goals 16 mentions "Phase 6 emits `cost.graph.run` cumulative ledger entries" but does not specify the integration with the Phase 5 cost ledger or the Phase 4 token ledger — three ledgers cited; no schema for any.
- **Side-effect replay safety.** Security caught this (the `@side_effect` decorator). Best-practices §Acknowledged blind spots 4 "Mid-node failure semantics with the checkpointer" hand-waves it away ("we trust this; we test it indirectly via the replay tests") and admits the `multiprocessing` kill test is "the strongest evidence; if it passes for one node, the LangGraph contract says it passes for all." That's the inductive leap Rule 12 (Fail loud) explicitly bans — best-practices trusts an unverified contract.
- **An actual measurement that the 30-minute new-developer onboarding target (§Goals 4) is met by anyone.** This is a vibe-target, not a verifiable target.

### Which disagreement matters most for *this* phase?

**The retry counter semantics row.** ADR-0014 is "three retries *per gate transition*" — the title of the ADR. Phase 5 final-design ships `for attempt in range(1, max_attempts + 1)` *per gate*, and its exit-criterion #19 demonstrates retry-1 fail + retry-2 recover *within a single gate evaluation* with Phase 4 re-entry. Best-practices has made `retry_count` monotonic across the workflow lifetime, which means a recipe-fail + RAG-fail + LLM-fail-attempt-1 workflow escalates after 3 attempts of any kind — not after 3 attempts at the same gate. Security and performance both keep per-gate semantics but disagree on where the counter lives (state field vs cycle-edge state). Best-practices' interpretation **silently breaks ADR-0014 and the Phase 5 retry parity test**. The synthesizer must pick a per-gate interpretation and reject best-practices' per-workflow shape — or surface that ADR-0014 has changed and write an ADR amendment, which is what global rule §7 (surface conflicts, don't average them) requires.

### Where do all three quietly agree on something questionable?

1. **All three defer Postgres (ADR-0016) and assume Phase 9 is the right place to swap.** None of the three verifies that SQLite's single-writer concurrency model (one writer per file or one writer across all workflows) actually meets the throughput target *for any concrete workload*. Performance budgets 800 writes/s/worker without measurement; security defers throughput to Phase 9; best-practices documents `database is locked` as a known limitation. The roadmap-level cost of this shared blind spot is real: Phase 9's Temporal envelope cannot simply "swap one constructor call" if the checkpoint blob shape (security's 64 KB cap, performance's path-references-only discipline, best-practices' Phase 3/4/5 typed sub-models inline) differs from what Postgres's `JSONB` column likes.

2. **All three assume `interrupt()` is the right HITL primitive without engaging with what Phase 11 will actually do.** Phase 11 (roadmap, not yet detailed) is where real PR comments and approvals land. Performance ships a mocked `codegenie loop resume`; security ships a complete signed-envelope flow; best-practices ships a typed Pydantic resume contract. None ask "what is Phase 11's actual signal source going to be" — a GitHub webhook? A Slack interactive button? An MCP server poll? The shape of the `HumanDecision` Pydantic model is being defined now without input from the system that will produce it.

3. **All three treat `langgraph-cli` as a debug visualization tool and budget zero for its operational role.** The roadmap lists `langgraph-cli` as Phase 6 tooling, not Phase 9 tooling. None of the three designs ships a Phase-9 plan for what happens to `langgraph-cli` when checkpoints move to Postgres — security firewalls it from production; performance ships a one-shot `dump-topology`; best-practices commits an SVG. If `langgraph-cli` becomes the *operator inspection tool* in Phase 9–11, none of the three has thought about its access model.

## Roadmap-level critiques

1. **Does this phase set up problems for phases 7–9?**

   **Phase 7 (extension by addition):** Best-practices' decision to edit `cli/remediate.py` (ADR-P6-001) is the only Phase-0–5 source touch in any of the three designs, and it directly conflicts with Phase 7's exit criterion ("The diff for this phase touches *only* new files — no Phase 0–6 source code is modified"). Performance's module-level `VULN_LOOP: CompiledGraph` is even worse — Phase 7's distroless subgraph cannot register itself without either modifying the module-level constant or shipping a separate compile path (which is also a Phase 6 edit). Security's `cli_hitl.py` adds operator infrastructure Phase 7 will inherit but cannot extend without editing.

   **Phase 8 (hierarchical planner + Redis hot views):** All three defer the supervisor to Phase 8 (correctly per ADR-0018), but none of the three defines the seam at which the supervisor will dispatch to `VulnLoop` / `build_vuln_graph()` / `build_vuln_loop_graph()`. The three names alone are a problem. The supervisor will route on `intent.task_type`, but the three designs each name their entry point differently; Phase 8 has to pick one or rename them all.

   **Phase 9 (Temporal):** Security's `@side_effect` decorator is the explicit conflict — Phase 9's Temporal Activity model owns side-effect idempotency. Performance's "checkpoint returns when queued" relies on a 50 ms next-gate-boundary durability window that Temporal's Activity-level retries make moot but also incompatible (Temporal expects deterministic Activity inputs; performance's "delta-encoded checkpoint" makes activity replay reproducibility a question the design doesn't answer). Best-practices' `frozen=False` + `model_copy(update={})` is the most Temporal-compatible; security's `ReducerCall` and performance's dict-returns are not.

2. **Does it rely on something phases 0–5 didn't actually establish?**

   - **Performance assumes the Phase 5 `GateRunner.run` body can be "lifted verbatim" into nodes.** Phase 5 §330 calls out that the `for` loop *body* is lifted; the body's interaction with `with_prior_attempt` and `RetryLedger.record` is what survives. Performance's "5 nodes (build_spec → execute_sandbox → collect_signals → evaluate_gate → branch)" decomposes Phase 5's monolithic `GateRunner.run` into five chunks Phase 5 does not actually expose. Whether `build_spec` is a public Phase 5 function or an internal step of `GateRunner.run` is unspecified by Phase 5's final design.
   - **Security assumes a per-orchestrator gate-control HMAC handshake "continued from Phase 5"** (§Goals "HITL authentication model"). Phase 5's final design explicitly rejected the `codegenie-gated` daemon (Phase 5 §330: "Security's `codegenie-gated` daemon over AF_UNIX is over-engineered"). There is no Phase-5 HMAC handshake to continue.
   - **Best-practices assumes Phase 5's `GateRunner` exposes a `.run-one` method** (§Architecture "thin wrapper over Phase 5 GateRunner.run-one"). Phase 5 ships `GateRunner.run` (the looped version); a single-attempt entry point would be a Phase 5 edit best-practices' "no Phase 0–5 source touch" promise forbids.

3. **Does it violate any load-bearing commitment from production/design.md §2 or CLAUDE.md?**

   - **Commitment §2.1 "No LLM in the gather pipeline."** None of the three violates this directly, but security's `apply_decision` reducer fence-wraps operator `notes` to feed to "Phase 4's `FenceWrapper`" — meaning Phase 6 is participating in LLM input construction. This is an "LLM at routing time" by another name unless the notes never reach an LLM, in which case the fence is theater.
   - **Commitment §2.4 "Determinism over probabilism for structural changes."** Best-practices' `route_after_attempt` returning `f"retryable_{state.last_engine}"` is a string-construction-at-routing-time pattern that is hard to statically verify; security's `Literal[...]` return types are the more disciplined shape but disagree with best-practices.
   - **Commitment §2.5 "Extension by addition."** Best-practices' `cli/remediate.py` edit + performance's module-level compile + security's Phase-5-amendment HMAC handshake all bend or break this. Best-practices is most honest about it (ships an ADR-P6-001 capturing the edit); performance and security do not document the implicit Phase-5 edits.
   - **CLAUDE.md Rule 3 "Surgical Changes."** Security's design is the largest violator — it ships operator-key infrastructure that the roadmap explicitly assigns to Phase 11/16. The "tamper-evidence layer is Phase 6's responsibility" claim in §Lens summary is the design's own justification for scope creep; the synthesizer should weigh whether that claim is load-bearing or rationalization.
   - **CLAUDE.md Rule 9 "Tests verify intent, not just behavior."** Best-practices' docstring-vs-AST contract test is the canonical example of a test that *appears* to verify intent but actually verifies syntactic appearance. Performance's canary-overhead benchmark is the most honest of the three test plans (asserting a real measurable property); security's adversarial tests are the most intent-aligned but ship against a threat model that may not be Phase 6's threat model.
   - **CLAUDE.md Rule 12 "Fail loud."** Performance's "non-fsync'd checkpoint may be lost on hard crash between nodes. Acceptable because the next node either re-derives state or restarts from the last fsync'd frame" (§Component 3 "Tradeoffs accepted") is exactly the kind of silent-degradation Rule 12 forbids. "We never lose more than one node's worth of work" is not the same as "we never lose work" — and the exit criterion says "without state loss."
