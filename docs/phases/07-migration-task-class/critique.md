# Phase 07 — Add migration task class (Chainguard distroless): Devil's-advocate critique

**Reviewed by:** Devil's-advocate critic subagent
**Date:** 2026-05-12

## Method

I read all three designs and attacked each on its own terms. I do not propose alternatives. My job is to surface what the synthesizer needs to see before it merges.

The central tension is extension-by-addition: ADR-0028 plus Phase 6's exit-criterion #14 require that **no Phase 0–6 source file is modified**. Every design claims it satisfies this. I went back to the actual Phase 0–6 final-design.md files and pulled the contract surfaces: `Probe` ABC (Phase 2 / ADR-0007), `RecipeEngine` + `RecipeSelection.reason` closed `Literal` (Phase 3), `FallbackTier.run(...)` signature (Phase 4), `@register_signal_kind` open registry + closed `Literal["shared_kernel","microvm"]` for `gate_isolation_class` (Phase 5), `VulnLedger.last_engine: Literal["recipe","rag","phase4_llm"]` + `cli/loop.py` not editable + `cli/sherpa.py` named as the future dispatch home (Phase 6). Three of these are seams Phase 5/6 actually built for Phase 7; two of them (the `RecipeSelection.reason` enum and the `Probe` ABC's input field set) are closed against the kind of extension the security and best-practices designs need.

## Attacks on the performance-first design

### Concrete problems

1. **Problem:** `ShellInvocationTraceProbe` is declared a *probe* with `applies_to_tasks = ["distroless-migration"]` and run inside `run_in_sandbox` "at gather time" (lens summary §"prebuilt fixture image at gather time, not against the candidate image"), but it does `docker buildx build --load` and then `docker run --rm` of the *target's* image. **That is executing the repo's own code at gather time.** Phase 2 specifically forbade this; that's why C4 (`RuntimeTraceProbe`) was deferred and ships with `applies() = False`. The performance design imports a Phase-5-class workload into Phase-2's deterministic-gather pipeline and pretends it's still a probe.
   **Why it matters:** Phase 2's whole threat model rests on "no target-binary execution at gather time." A gather-time probe that runs an attacker-influenceable entrypoint inside a `bwrap`/subprocess sandbox (Phase 2's posture) is a sandbox-class downgrade from where the security design rightly places this work. The cache-hit rate the performance lens depends on (sub-2 ms on a `(dockerfile_blake3, image_digest, strace_wrapper_digest)` key) presupposes the probe is cheap — it isn't; the *cold* path is a `docker buildx build` plus a `docker run`.
   **Where:** Component 2 (`ShellInvocationTraceProbe`), data-flow step 1 ("Cold gather (one-time, amortized…) ShellInvocationTraceProbe runs once").

2. **Problem:** Goal #2 — "≥ 12 workflows/hr mixed portfolio, single-worker, warm" — silently assumes the buildkit + grype + dive subprocesses fit within Phase 5's single chokepoint, and that the chokepoint has been *re-tuned* to accept docker daemon work. But Phase 5 shipped DinD-on-macOS as `gate_isolation_class: "shared_kernel"` and Firecracker on Linux as a real second backend. Running `docker buildx build` inside Firecracker requires either DinD-in-Firecracker or a buildkit daemon socket exported into the microVM — both of which Phase 5 deferred. The performance design lists buildx-in-sandbox as a fact ("dive and shell_presence collectors run inside Phase 5's sandbox chokepoint") but never resolves *which* backend, and the wall-clock numbers (10–25 s warm build) are only credible for DinD-on-macOS. The Linux/CI numbers will be 2–5× worse, and `tests/integration/test_phase7_*` runs in CI.
   **Why it matters:** All eleven Goal targets (≥6/hr cold, ≥30/hr warm, p95 wall-clocks 180–540 s) are reported as single numbers — they're actually two distributions and the design hides the cost of the worse one.
   **Where:** Goals table; §"Resource & cost profile" → "Wall-clock per run (4-vCPU CI / M-series Mac, distroless task only)".

3. **Problem:** `MigrationLedger` is declared a fresh schema with `extra="forbid"` and `schema_version: Literal["v0.7.0"]`, but Phase 6 shipped `VulnLedger.last_engine: Literal["recipe","rag","phase4_llm"]`. The performance design says recipes/RAG/LLM-fallback are reused from Phase 4 ("Phase 4's RagLlmEngine on a distroless solved-example corpus shard") — so the engine names are the same — but a parallel ledger duplicates the closed `Literal` without explaining how Phase 8's supervisor (which Phase 6 promised would dispatch on `task_type`) reads one engine name across two schemas. Open Question #1 in the design defers this to the synthesizer. The supervisor that's supposed to consume both ledgers does not yet exist; Phase 7 is choosing a shape for Phase 8 without Phase 8 being there to push back.
   **Why it matters:** Two ledgers means two checkpoint-migration paths the moment either ledger's `Literal` bumps. Goal #15 ("0 tokens inside Phase 7's package boundary") is preserved, but the cost surfaces in Phase 8 as schema-merge work — a deferral disguised as a clean split.
   **Where:** Component 8 (`build_distroless_loop()` — the LangGraph factory); Open question 1.

4. **Problem:** The "Buildx layer cache as content-addressed disk artifact" plan (Component 4) assumes `--cache-from=type=local,src=$BUILDX_CACHE --cache-to=type=local,dest=$BUILDX_CACHE,mode=max` is **concurrency-tolerant**. The design literally asserts this ("buildkit's local-cache export is concurrency-tolerant"). That's a claim, not a fact — buildkit's local cache export has had race conditions filed against it as recently as 2024 (manifest-list corruption when two writers race on the same blob), and the design's own Failure-modes table row 2 ("Buildkit layer cache corruption") concedes this can happen. The design's mitigation is "Retry once with `--cache-from` only" — but cache corruption masks CVE-bearing layers, exactly the risk row 2 names, and a single retry doesn't help if both workers writing concurrently produce the same corrupted state.
   **Why it matters:** Goal #8 ("≥85% pulled-layer cache hit rate") is the load-bearing throughput assumption, and the cache is also the largest cross-workflow shared mutable state in the entire stack. Phase 5 explicitly *didn't* ship a verdict cache for exactly this reason ("Phase 5 does not ship a verdict cache (Phase 9's territory)"). Phase 7 ships one for image bytes, with weaker reasoning.
   **Where:** Component 4 ("Buildx layer cache + grype-db cache"); Risk #2 ("Buildkit cache as shared mutable state").

5. **Problem:** `cli/sherpa.py` is introduced as a "pure router ~30 LOC". But Phase 6's exit criterion explicitly named `cli/sherpa.py` as the file Phase 7 should add — it's the right home — *and* Phase 6 already shipped `cli/loop.py` with `run`, `resume`, `inspect`, `replay`, `migrate-checkpoint`, `render` subcommands. The performance design's `codegenie sherpa run` duplicates `loop run`, `loop resume`, `loop inspect`, `loop replay` ("Subcommands: `run`, `resume`, `inspect`, `replay` — same as Phase 6's `loop` subcommand surface"). Now operators have two CLI commands that do the same thing on different ledger types. A `MigrationLedger` workflow can't be resumed via `codegenie loop resume`, and a `VulnLedger` workflow can't be resumed via `codegenie sherpa resume`. The dispatch is *not* a thin router; it's a fork.
   **Why it matters:** Phase 8 is supposed to *replace* `codegenie sherpa run` with a supervisor. If sherpa's subcommand surface forks first, Phase 8 inherits two CLI surfaces to merge instead of one.
   **Where:** Component 9 (`codegenie sherpa run` — the dispatch CLI); §"Data flow" step 2.

### Hidden assumptions

1. **Assumption:** The grype DB cache invalidation can be controlled by a `flock(2)` advisory lock on a sentinel file.
   **What breaks if it's wrong:** macOS BSD flock and Linux fcntl-based flock have different semantics for shared-mode acquisition and lock inheritance across fork — the design's own §"Acknowledged blind spots" admits "cross-platform fixture coverage is partial". On a portfolio-scale CI runner, the "first arriver wins" semantics quietly become "everyone tries to update, everyone gets a different staleness", and the `cve_delta` signal becomes a coin flip.

2. **Assumption:** `imagetools inspect --raw` is a "manifest-only roundtrip (small KB-sized JSON, no layer bytes)" (Component 1).
   **What breaks if it's wrong:** `docker buildx imagetools inspect` *can* return multi-MB manifest lists for multi-arch images. The "Acknowledged blind spots" admits multi-arch is deferred — but the inspect call doesn't know that and will return a multi-arch manifest list anyway. The cache key `(image_ref, registry_mirror_id)` doesn't include `--platform`, so a `linux/amd64`-only gather later in the day reuses a `linux/arm64,linux/amd64` resolved digest with no warning.

3. **Assumption:** Phase 5's `task_class=migration` overlay flag is a "tiny additive flag, identical pattern to Phase 5's `test_execution=True` overlay".
   **What breaks if it's wrong:** Phase 5's `test_execution` overlay was specifically designed for tests inside an already-built image — it relaxes egress for `npmjs.org` etc. A `task_class=migration` overlay needs *different* relaxations (`cgr.dev`, `docker.io`, plus `docker.sock` access for buildx). That's not an overlay flag pattern; that's a new sandbox profile masquerading as an overlay.

### Things this design missed that a different lens caught

- **The security design's `applies_to_lifecycle` field surfaces a real concern the performance design papers over**: a probe that runs a target-built image is structurally different from a Phase-2 gather-time probe. The performance design just sets `applies_to_tasks = ["distroless-migration"]` and calls it gated. Gating by task isn't the same as gating by *when in the pipeline*.

- **The best-practices design's closed-`Literal` audit (its Open Question #6) is exactly the audit the performance design owes the reader.** Performance's `MigrationLedger` defines `last_engine` (Phase 6 closed `Literal`) without saying whether the value space is shared or forked. Performance owes a table.

- **The security design's `dive` threat model** (`dive` has been CVE-bearing twice in five years; runs on host) is invisible in the performance design. Performance runs `dive` inside Phase 5's chokepoint, but assumes "no extra invocation" because dive's output is reused for `shell_presence`. That's a correctness-vs-defense trade the design never explicitly makes.

## Attacks on the security-first design

### Concrete problems

1. **Problem:** The design introduces `applies_to_lifecycle = ["gate"]` as "an additive field on the `Probe` ABC that defaults to `['gather']` for backward compatibility" (§"Trust boundaries" + Open Question 1). **That is an edit to `src/codegenie/probes/base.py`.** Per ADR-0007 and Phase 2's final-design §"Probe contract preserved (ADR-0007)" — "Zero edits to `src/codegenie/probes/base.py`. Zero edits to `ProbeContext`'s public field set in Phase 0/1" — adding a new class-level attribute with a default value to the ABC is the canonical example of what that commitment forbids. Phase 2's `consumes_peer_outputs` was achievable as a class attribute *only because Phase 2 was the phase that landed it*; Phase 7 cannot replay that maneuver against an already-frozen ABC. The design tries to have it both ways: "the new probes are tested against the *existing* Phase-2 `Probe` ABC unmodified; the `applies_to_lifecycle` field is a `ClassVar` default that pre-existing probes inherit-as-gather without code change" — but pre-existing probes inheriting a default *requires the default to exist on the base class*, which is the edit.
   **Why it matters:** This is the load-bearing extension-by-addition violation in the whole phase. The design's own goal #5 ("a CI test computes BLAKE3 over every source file under …`/probes/{layerA..G}`… and compares against `tools/phase-frozen-digests.yaml`") would *fail* on `probes/base.py` the moment this field is added. The design has a CI check that detects its own contract violation.
   **Where:** §"Lens summary" final paragraph ("Contradiction-to-prior-design surfaced"); §"Components → ShellInvocationTraceProbe" interface; Open Question #1; goal #5 ("Extension-by-addition enforcement").

2. **Problem:** The coordinator is also edited, silently. Goal #5's premise is that no Phase 0–6 file changes — but to "refuse to schedule a `gate`-lifecycle probe at gather time" (the design's words), Phase 2's `coordinator.py` must learn to *read* `applies_to_lifecycle`. Phase 2's coordinator dispatches probes by their `applies()` boolean; it does not branch on a lifecycle string. Adding the branch is exactly the kind of edit Phase 2 paid for `consumes_peer_outputs` (one ADR-gated coordinator branch). Phase 7 cannot pay for a second one without amending the ADR-0028 freeze.
   **Why it matters:** The design says "the coordinator's contract is extended (additively)" (Component 2 → ShellInvocationTraceProbe). That's not additive; it's an edit. And the security lens's BLAKE3 file-freeze CI catches it on `src/codegenie/coordinator.py`.
   **Where:** Component 2 (`ShellInvocationTraceProbe` → Interface paragraph "coordinator's contract is extended").

3. **Problem:** The strict-AND signal set in goal #10 introduces a signal `image_size_post / image_size_pre ≤ 0.8` and calls it "strict-AND." Phase 5's final-design §"Signal kind extensibility" / §"Why this choice over the alternatives" was emphatic: signals registered via `@register_signal_kind` are *binary* (`passed: bool`) — they translate to `TrustSignal(kind, passed, details)` and feed Phase 3's existing `TrustScorer.score(...)`. A ratio comparison is not a binary signal; it's a soft threshold. The performance design correctly flags `dive`'s `efficiency_pct >= 0.85` as "advisory at first — gate emits `confidence: low` rather than hard-fail" — the security design ignores that and treats a soft ratio as a strict-AND boolean, which silently *strengthens* gate failure semantics on images that legitimately grow (e.g., a Chainguard distroless that pulls in extra glibc support).
   **Why it matters:** Goal #11 ("No LLM in the recipe path") plus the always-fail-on-size-regression signal means a legitimate distroless migration of a small Alpine base to a larger glibc Chainguard runtime fails the gate every time, and falls through to retry-2 (LLM fallback) for purely-size reasons. The $/PR target in any sibling design that survives gets blown by the wrong gate.
   **Where:** Goal #10, sixth bullet ("`image_size_post / image_size_pre ≤ 0.8`").

4. **Problem:** The Secret Broker (`codegenie-secretd`) is a new daemon process with its own Unix user, AF_UNIX socket, systemd unit, on-disk encrypted credential file, separate audit chain (`audit/secretd.jsonl`), and tamper-detection on startup (Failure-modes row "Secret Broker tamper"). **This is service-shaped infrastructure dropped into the Phase 0–6 local POC.** Phase 0–6 are explicitly a "local Python CLI POC" per `CLAUDE.md`; the entire roadmap has Postgres/Temporal/Redis arriving in Phases 8–9 and production credential handling arriving in Phase 16. The design's "Risks #1" admits "Recommendation: in Phase 16 hardening, move the broker to a TEE (SGX/SEV) or to a per-workflow ephemeral KMS-leased credential" — but the broker itself is shipping *now*, in the phase whose explicit exit criterion is "the diff for this phase touches only new files."
   **Why it matters:** A new long-running daemon is the antithesis of a local POC. It also breaks the "Filesystem-backed everything" convention from `CLAUDE.md` ("Single Python project, no services, no databases. Filesystem-backed everything."). The design knows this — its §"Acknowledged blind spots" includes "Operators on bare laptops without a desktop keychain pay the `age` cost" — but never asks whether the local POC needs a credential broker at all when the existing operator-side `~/.docker/config.json` (called out as the cleanest answer in best-practices Open Question #7) already handles this case.
   **Where:** §"Trust boundaries" (Secret Broker box); Component "SecretBroker"; Risks #1; Failure-modes "Secret Broker compromised".

5. **Problem:** The `cosign verify` step (workload step 4: "cosign verify cgr.dev/chainguard/<base>@<digest> against policy") demands a pinned OIDC issuer regex in `tools/cosign-policy.yaml`, plus the Chainguard public key in `tools/chainguard-publisher.pub`, plus the `CveImageMapIngester` signature ceremony — all of which presuppose a Chainguard *publication infrastructure* the roadmap doesn't reference and Chainguard itself does not document as a stable contract. The roadmap (Phase 7 entry) names only `dockerfile-parse`, `dive`, `docker buildx`, "Chainguard registry credentials", and "A CVE-to-image-recommendation lookup table." Nothing about a Chainguard-signed image-map publisher, a Chainguard OIDC issuer, or a `chainguard-publisher.pub` artifact. The design invents an entire supply-chain trust hierarchy that doesn't exist upstream.
   **Why it matters:** Operators cannot complete the setup if the artifacts don't exist. The design's E2E test ("E2E migration of a vulnerable Node.js service") requires `cosign verify` to succeed against a real Chainguard image — without a published OIDC policy aligned with what Chainguard actually signs, this test either passes by accident (the policy is too permissive) or fails permanently (the policy is too strict and rejects valid Chainguard signatures).
   **Where:** Component "CveImageMapIngester"; Goal #7 ("CVE-to-image lookup table integrity") and #8 ("Image-signature verification SLA").

### Hidden assumptions

1. **Assumption:** Phase 5's `TrustGate.register_signal(name, schema_field, predicate)` exists as an extension API.
   **What breaks if it's wrong:** Phase 5's final-design names `@register_signal_kind` as the open registry decorator and `ObjectiveSignals` as the Pydantic model widened by ADR-P5 amendment per signal kind. There is no `TrustGate.register_signal(...)` callable function in Phase 5. The security design's `MigrationSignalScorer` "registers via `TrustGate.register_signal(...)` at startup" (Component "MigrationSignalScorer") — that API was made up. Phase 7 either edits Phase 5 to add it (frozen file) or uses `@register_signal_kind` and pretends that's the same thing (it isn't — the decorator registers a *collector*, not a scorer predicate).

2. **Assumption:** A migration gate's wall-clock can be raised from Phase 5's 5 min cap to 10 min "for migration gates only."
   **What breaks if it's wrong:** Phase 5 didn't ship a wall-clock policy keyed on task class. The design treats this as additive ("lifetime: ≤10 min wall-clock (raised from Phase 5's 5 min)" in the microVM box) but it requires Phase 5 to learn a `max_wall_clock_by_task_class` map. That's an edit. The CI BLAKE3 file-freeze gate fails.

3. **Assumption:** A `runtime_shell_count == 0` strict-AND with a 30s wall-clock will not produce a high false-positive rate on legitimate images.
   **What breaks if it's wrong:** §"Acknowledged blind spots" admits "Some images legitimately invoke shell" and the synth is left to decide whether to ship an allowlist. The design's stance is "no allowlist; humans always merge" — but in practice the gate auto-fails legitimate migrations, retry-1 reproduces the failure (the recipe didn't change the entrypoint), retry-2 invokes the LLM to "fix" a non-bug, retry-3 escalates. Every legitimate-but-shelly migration costs three retries plus an LLM call.

### Things this design missed

- **The performance design's $/PR target is incompatible with this design's strict-AND on `image_size_post / image_size_pre ≤ 0.8`** (problem 3 above). A gate that consistently false-positives feeds Phase 4's LLM, which costs money. The security design budgets zero attention to this interaction.

- **The best-practices design's Open Question #6 — closed `Literal` audit across Phase 3/4/5/6 — is the right framing for whether `applies_to_lifecycle` is "additive."** The security design adds the field and asserts additiveness without doing the audit; best-practices flags the audit and admits it's the most important pre-implementation review item.

- **The performance design's wall-clock numbers (180–540 s p95)** are completely absent from the security design — which only says "Gate wall-clock p95: ~400s" (§"Resource & cost profile"). Where does 400s come from? The design doesn't show its work. With cosign verify + buildx + dive + image-runner + a 30s eBPF trace, that's a lower bound, not a p95.

## Attacks on the best-practices design

### Concrete problems

1. **Problem:** The design relies on `FallbackTier.run(..., task_type="distroless_migration", ...)` (Component 5 → `replan_with_phase4` node), and Open Question #3 explicitly flags this: "does its `run()` take a `task_type` keyword arg or is it implicit?" The Phase 4 final-design.md is concrete: `run(advisory, repo_ctx, recipe_selection, *, run_id, include_pending: bool, auto_promote: bool) -> FallbackTierResult`. **There is no `task_type` parameter.** Phase 4 was designed entirely for vuln remediation; the only sense in which it "generalizes" is the assertion in Phase 4's own forward-integration section that "Phase 7 (distroless migration as second task class): new `DockerfileBaseImageSwapTransform` + new engines; `RagLlmEngine` shape generalizes by adding a new task-class prompt (`migration_distroless.v1.yaml`)". That generalization requires Phase 4 to learn about task types — which means *editing* Phase 4. The design admits it; the synthesizer is being asked to handle it; but Phase 7's exit criterion says no Phase 0–6 edits.
   **Why it matters:** Without Phase 4 routing on task type, the RAG-fallback / LLM-fallback paths described in this design (and in the performance design) don't actually exist. The promised "deterministic → recipe → RAG → LLM" chain is broken at the third stage.
   **Where:** Component 5 (`build_distroless_loop()` → `replan_with_phase4` node); Open Question #3.

2. **Problem:** The design adds `RecipeSelection.reason = "unsupported_image_dialect"` as a "new enum value (additive)" (§"Lens summary" final paragraph, §"Failure modes & recovery" row "Image dialect doesn't match"). Phase 3's final-design.md, decision #5, defines this as **a closed `Literal`**: `Literal["matched","no_engine","range_break","peer_dep_conflict","unsupported_dialect","catalog_miss"]`. Adding `"unsupported_image_dialect"` to a closed `Literal` is a source edit to Phase 3 — exactly the kind of contract violation Open Question #6 names as the "most likely surprise." The design caught itself: its Open Question #6 lists `Recipe.engine: Literal["ncu", "openrewrite"]` as needing `"dockerfile"`, and `RecipeSelection.reason` as needing `"unsupported_image_dialect"`. **But the design also picks (b) — "ship a new value and call it additive" — for the in-text components.** Either the closed `Literal` is the contract violation the design admits it is, or the design needs to use `"unsupported_dialect"` (which already exists in Phase 3) and never mention `"unsupported_image_dialect"`. It does neither.
   **Why it matters:** This is the design's load-bearing self-contradiction. Best-practices is the lens explicitly attacking the closed-`Literal` problem, *and* writing new closed-`Literal` values it claims are additive.
   **Where:** Failure-modes table row "Image dialect doesn't match"; Open Question #6.

3. **Problem:** Component 5 introduces a `nodes_distroless/` directory parallel to `nodes/` and justifies it via "ADR-0022 Three Strikes And You Refactor" — Phase 6 is strike one, Phase 7 is strike two, abstraction deferred to Phase 15. **Phase 6's `nodes/` directory does not exist as a final-design artifact** in the form the design implies. Phase 6's final-design has a single `graph/` package containing `vuln_loop.py`, `state.py`, `edges.py`, `nodes/` (Component 1, Component 2). The design's claim that Phase 6 already established `nodes/` as the *vuln-specific* nodes directory is technically true, but the design's mirror — `nodes_distroless/` — is a renaming of the namespace, not an extension. A `node_<role>.py` file pattern across both directories is more brittle than a single namespace; the "refactor at strike three" deferral is sensible *if and only if* there's a meaningful per-task-class namespace, but the design lists ten node names ("`ingest_target`, `select_recipe`, `rag_lookup`, `replan_with_phase4`, `apply_recipe`, `validate_in_sandbox`, `record_attempt`, `await_human`, `emit_artifact`, `escalate`") that map *one-to-one* with Phase 6's vuln nodes. Two directories, ten files each, the same shapes — that's not Three Strikes; that's copy-paste with worse ergonomics.
   **Why it matters:** Phase 8's hierarchical planner (the supervisor) has to import nodes from both directories. The shared `await_human`, `emit_artifact`, `escalate` are listed as "same shapes" — meaning they're either re-implemented (drift risk) or imported from Phase 6's `nodes/` (which the design says is forbidden because that would be using Phase 6 code from Phase 7's directory — except it isn't forbidden, that's reuse, which is the whole point).
   **Where:** Component 5 (`build_distroless_loop()` → "Two sibling subgraph trees (`nodes/` for vuln, `nodes_distroless/` for distroless)").

4. **Problem:** The `DockerfileRecipeEngine` ships an OpenRewrite primary plus a hand-rolled fallback (Component 3). Phase 3 shipped `OpenRewriteEngineStub` as exactly that — a stub, opt-in via `--engine=openrewrite`, registered-but-unavailable if `java` is missing, with one smoke-tested recipe. The best-practices design upgrades this from "anchor the contract" to "primary engine for two recipes including a multi-stage refactor", and acknowledges in Risk #3 that "`rewrite-docker` covers base-image swaps well and multistage refactors poorly." So the *intended* primary engine cannot do the harder of the two recipes the phase ships. The fallback path is the real path, and the OpenRewrite primary is decorative.
   **Why it matters:** The phase's headline deliverable per the roadmap is "Dockerfile base-image swap, multi-stage build refactor" (two recipes). The design's own risk admits one of them won't work on the primary engine. So the multi-stage refactor recipe always falls through to the hand-rolled fallback. That's fine for shipping a phase — but it means the OpenRewrite seat is structurally never exercised in Phase 7 for the harder case, and Phase 15 (agentic recipe authoring, OpenRewrite-shaped) inherits a never-tested integration.
   **Where:** Component 3 ("Primary path: `rewrite-docker` … Fallback path: hand-rolled `dockerfile-parse` mutator"); Risk #3.

5. **Problem:** The additive-diff CI gate (`tools/phase7_additive_gate.sh`, Component 10) is "one-shot — after Phase 7 merges, the gate is retired." Acknowledged blind spot 6 admits this. **But the gate is the only mechanical enforcement of extension-by-addition the design ships.** Phase 8 (Hierarchical Planner), Phase 15 (Agentic recipe authoring), and Phase 10 (Discovery + Assessment) all add code that could plausibly mutate Phase 0–6 surfaces; the design knows this ("Future task classes (Phase 15's recipe-authoring) get no automated 'did we accidentally edit Phase 0–6?' check") and explicitly defers the discipline to humans ("teams introducing future task classes are expected to re-instate it on their own PR labels"). For a lens whose entire premise is *idiomatic, maintainable, conventional, well-tested* — abandoning the only automated test of the load-bearing invariant after the first run is the opposite of best-practices.
   **Why it matters:** The whole point of Phase 7 per ADR-0028 is to *prove* the contracts extend by addition. If the only proof apparatus is one-shot, Phase 8's first PR can silently violate the same invariant, and the only review surface is human attention — which the design's own Risk #1 names as "security-review-fatigued."
   **Where:** Component 10 (the additive-diff CI gate); Acknowledged blind spot 6.

### Hidden assumptions

1. **Assumption:** Phase 2's skill schema has `additionalProperties: true` at the `applies_to` level (Component 7 → Skills, Open Question #4).
   **What breaks if it's wrong:** The design needs `applies_to.image_patterns` to be a free addition; if Phase 2's schema rejects unknown keys at that level, Phase 7 must *modify* the schema (Phase 2 source edit). The design admits not knowing.

2. **Assumption:** A Dockerfile recipe can be applied byte-deterministically with a `dockerfile-canonicalize` helper that is "whitespace-only" (Component 3 internal design).
   **What breaks if it's wrong:** Dockerfile semantics include subtle line-continuation rules, heredoc syntax, and Windows-line-ending tolerance the parser may normalize differently across `dockerfile-parse` versions. "Whitespace-only" canonicalization will silently produce different diffs on different machines for the same input.

3. **Assumption:** `dockerfile-parse` is "mature but not perfectly compatible with every BuildKit-specific syntax (heredocs in 1.5+)" and the probe degrades to `confidence: medium` (Component 1 tradeoffs).
   **What breaks if it's wrong:** "Confidence: medium" on a parser warning lets the recipe selector match anyway (the selector reads structural facts, not confidence). A BuildKit-heredoc Dockerfile with a `parser_skipped_lines > 0` warning gets a recipe applied to its partial parse — the security design specifically forbids this exact failure mode ("partial parses are never accepted").

### Things this design missed

- **The security design's adversarial-Dockerfile corpus (≥30 fixtures)** is absent here. Best-practices has "Dockerfile with embedded null bytes / 100 MB Dockerfile / Dockerfile whose `FROM` is `localhost:5000/<long-redirect-chain>` / Recipe YAML with reflective fields" — four adversarial tests vs the security design's 30. The lens claims "adversarial coverage 100% for Dockerfile parsing" but the test list doesn't approach that.

- **The performance design's `dive` cache + `shell_presence` projection** trick lets one tool invocation serve two signals; best-practices ships two independent signal collectors (Component 7 lists `shell_required_at_runtime` as a separate strict-AND signal driven by `ShellInvocationTraceProbe`), and the data flow doesn't show whether dive is re-invoked. The phase's wall-clock budget can't sustain two dive runs per gate.

- **The performance design's regression-suite wall-clock target** (p50 ≤ 4 min, p95 ≤ 7 min) and its xdist-with-LFS-pack plan is missing entirely from best-practices. Best-practices declares "Phase 6 regression suite: 100% pass before merge" (Goal §"Goals" bullet 7) but never tells you how long it takes. Hard gates without time budgets become gates that slip silently.

### Which disagreement matters most for *this* phase?

**The closed-`Literal` row.** Phase 3 shipped `RecipeSelection.reason` as a closed `Literal`. Phase 6 shipped `VulnLedger.last_engine` as a closed `Literal["recipe","rag","phase4_llm"]` and `VulnLedger.schema_version` as `Literal["v0.6.0"]`. Phase 4's `FallbackTier.run` does *not* take a `task_type` kwarg. **These are the actual points where extension-by-addition meets a wall.** The best-practices design is the only one that surfaces this honestly (Open Question #6) but resolves it badly (writes new values into closed Literals and calls it additive). The performance design avoids the wall by *forking* (new `MigrationLedger` with its own Literals, new `cli/sherpa.py` with its own subcommand surface) — which preserves Phase 6's frozen files but creates two of everything that Phase 8 has to merge. The security design ignores the wall entirely and writes APIs (`TrustGate.register_signal`, `applies_to_lifecycle` field on the ABC) that don't exist in the frozen phases.

The synthesizer must decide:

- **Does ADR-0028 mean "no Phase 0–6 source edits, ever"** (in which case the closed `Literal`s are themselves contract bugs and Phase 7 cannot land without amending ADR-0028 to permit additive `Literal` extension or open-enum migration)?

- **Or does it mean "no Phase 0–6 *behavioral* edits"** (in which case adding a value to a closed `Literal` is the kind of additive change ADR-0028 permits, but then the BLAKE3 file-freeze CI test from the security design is the wrong test)?

Phase 7's whole reason for existing per the roadmap entry is to *prove* extension-by-addition. The three designs together prove the opposite: the contracts as shipped in Phases 3–6 do not extend cleanly without either edits or forks. That finding is more important than any of the individual design picks.

### Where do all three quietly agree on something questionable?

1. **All three treat Phase 5's `gate_isolation_class: Literal["shared_kernel","microvm"]` as not their problem.** It's a closed `Literal` on a Phase 5 model that Phase 7's migration gate clearly wants to read and emit. None of the three designs say what `gate_isolation_class` is for a migration gate, whether Firecracker-on-Linux is "microvm" enough when it runs DinD inside, or whether the security design's rootfs-bumped microVM counts as the same isolation class. Phase 11's PR-promotion logic was promised to read this value — if all three designs leave it null, Phase 11 inherits an undefined invariant.

2. **All three assume Phase 4's `FallbackTier.run` either already routes by task class (false) or will be silently extended to (Phase 4 edit).** The performance design hand-waves it as "a distroless solved-example corpus shard". The security design says "Phase 4 LLM fallback (called only on retry-2)". The best-practices design flags it as Open Question #3. None of them propose how the fallback actually produces a distroless-shaped patch instead of a vuln-shaped one without Phase 4 learning about task classes.

3. **All three ship the OpenRewrite-shaped `rewrite-docker` integration despite none of them having operationally verified Chainguard image coverage.** The best-practices Risk #3 admits "rewrite-docker covers base-image swaps well and multistage refactors poorly" — but ships it anyway. The performance design names `recipes/catalog/distroless_node_swap.yaml` and `distroless_node_multistage.yaml` and assumes both can be byte-deterministic. The security design demands round-trip parse equivalence on every recipe output, which `rewrite-docker`'s output (formatted differently than a `dockerfile-parse` re-serialize) may not satisfy. None of the three demonstrates a single working migration on a real Chainguard image.

## Roadmap-level critiques

1. **Does this phase set up problems for later phases?** Yes, in three specific ways. (a) **Two ledgers in Phase 7** (perf, best-practices) means Phase 8's supervisor must read both schemas; one ledger with task-class discrimination (a Phase 6 edit) is the cleaner shape Phase 8 will eventually be forced to adopt — but Phase 7 picks the harder option and bequeaths the merge to Phase 8. (b) **Phase 11 reads `gate_isolation_class`** (per Phase 5 final-design) but none of the three Phase 7 designs say what value a migration gate emits — Phase 11 inherits an undefined enum. (c) **Phase 15 (agentic recipe authoring)** is supposed to grow the `cve_to_distroless.yaml` (perf) / `cve_image_recommendations.yaml` (best-practices) / `cve-image-map.yaml.sig` (security) catalog. Three different file names across three designs is one catalog name Phase 15 has to choose; whichever doesn't win has stale references in the design folder.

2. **Does it rely on something an earlier phase didn't establish?** Yes, repeatedly. (a) `FallbackTier.run(task_type=...)` — Phase 4 didn't add the kwarg. (b) `TrustGate.register_signal(name, schema_field, predicate)` — Phase 5 doesn't expose this method; the open registry is `@register_signal_kind`, which is a different shape. (c) `gates/catalog/stage6_validate_distroless.yaml` (perf) — Phase 5 named the YAML-gate-catalog pattern but didn't ship `stage6_validate_*` as a naming convention; it's a coined seam. (d) `applies_to_lifecycle` on Probe ABC — Phase 2's `consumes_peer_outputs` precedent is *one* such class attribute, ADR-gated; Phase 7 cannot replay the maneuver without amending ADR-0028.

3. **Does it violate any load-bearing commitment?**
   - **§2.5 Extension by addition.** Security and best-practices both admit edits-disguised-as-additions. Performance forks ledgers and CLIs to *avoid* edits, which preserves the letter of the commitment while breaking the spirit (Phase 8 inherits doubled surface area).
   - **§2.1 No LLM anywhere in the gather pipeline.** All three honor this; the performance design's `ShellInvocationTraceProbe` runs the *target binary* at gather time, which is a different commitment from the LLM rule but in the same threat-model neighborhood (Phase 2's "gather is deterministic and side-effect-free against the target").
   - **§2.2 Facts, not judgments.** Security design's `static_shell_binary_count == 0` strict-AND on the `dive` projection is a *threshold*, not a fact; the probe still emits the count, but the gate's binary collapse loses the fact-vs-judgment line. Performance treats `dive.efficiency_pct >= 0.85` as advisory at first, which is the right hedge.
   - **§2.3 Honest confidence.** Performance design's 10s strace budget with `confidence: medium` on budget exhaust is the most honest of the three; security design's hard 30s strict-AND is the least.
   - **`CLAUDE.md` "Single Python project, no services, no databases. Filesystem-backed everything."** Security design ships `codegenie-secretd` daemon + systemd unit + separate Unix user; directly contradicts this convention.
   - **`CLAUDE.md` "Humans always merge."** All three honor this via Phase 6's existing HITL contract; no violations.
